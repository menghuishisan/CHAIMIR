# 区块链教学·实验·竞赛三位一体平台 — 总体蓝图

> 本文档是平台的总纲,记录已确认的产品基线与模块划分,作为后续各模块详细设计的索引。
> 各模块的详细设计另行成文(`01-身份与租户.md`、`02-...` 等)。
>
> 最后更新:2026-05-29

---

## 一、产品定位

教学、实验、竞赛三位一体的闭环平台,三者地位均衡,互相复用、互相反哺。

- **教学**打基础,**实验**练能力,**竞赛**做检验与激励。
- 闭环价值:实验与竞赛站在同一套"沙箱 + 评测 + 题库"技术底座上,不重复造轮子;竞赛回放与题解又反哺教学。

## 二、已确认的产品基线

| 维度 | 决策 |
| --- | --- |
| 定位 | 教学 + 实验 + 竞赛三位一体,三者均衡 |
| 技术覆盖 | 全链栈,可扩展接入(**适配器/插件化**为核心架构原则,新链接入 = 加一个适配器) |
| 建设目标 | 一次性完整实现全部功能 |
| 部署形态 | 同时支持**私有化部署**(单校独立部署)与 **SaaS 多租户**(学校 ID 数据隔离) |
| 账号体系 | **禁止自助注册**;学校申请 → 平台管理员审核(仅 SaaS);师生由学校管理员导入 |
| 实验形态 | 云端容器沙箱(代码实验)+ 前端仿真实验(可视化)+ 教师可配置 + 评测插件化 |
| 竞赛形态 | 至少支持**解题赛**与**对抗赛**,理论答题作为轻量附加 *(方向已定,待细化)* |
| 登录标识 | **手机号必填**为主登录凭证 + 找回通道;学号/工号 + 学校ID 为学籍内部不可变标识;手机号全局唯一,一号多校登录后选学校 |
| 面向区域 | 国内高校为主,中文为主、多语言留扩展点 |
| 技术栈 | **各模块按特性单独选型,不强求单一语言**;当前统一口径以《总-技术选型》为准。 |
| 鉴权 | JWT **双 Token 无感刷新**(Access 短期 + Refresh 长期静默续期) |

### 角色层级

- **平台管理员**:仅 SaaS 模式存在,审核学校入驻申请。私有化部署无此角色。
- **学校管理员**:租户内最高权限,导入/管理教师与学生。私有化部署下即顶层管理员。
- **教师**:开课、布置实验、组织竞赛、批改。
- **学生**:学习、做实验、参赛。

## 三、核心架构原则

1. **业务薄、引擎厚**:实验与竞赛等业务模块尽量轻,共享能力沉淀到底层引擎。
2. **适配器/插件化**:区块链运行时、工具链、评测器、题型均为可插拔插件,平台不绑定任何具体链。
3. **多租户与私有化同构**:同一套代码,SaaS 下按学校 ID 隔离数据,私有化下单租户部署、去掉平台管理员层。

## 三之二、跨模块统一约定

> 11 模块交叉审查后沉淀的全局约定,所有模块共同遵守,避免接缝不一致。

### 1. 实时通道归属(M10 vs 各模块)
- **业务实时数据**(排行榜、对局状态、实例进度通知、个人通知红点)统一经 **M10**:模块调 `POST /notify/push(topic, payload)`,客户端订阅 M10 统一 `WS /ws` 的对应 topic。
- **引擎内部交互流**(沙箱终端 PTY、后端计算仿真 stream)保留各模块自有 WS,因其是与引擎实例的直连交互,不属于"广播型业务数据"。
- topic 命名:`tenant:{tenant_id}:<模块>:<资源id>:<频道>`,如 `tenant:10:contest:55:leaderboard`、`tenant:10:sandbox:9001:progress`。个人红点固定为 `tenant:{tenant_id}:notify:{account_id}`,租户告警固定为 `tenant:{tenant_id}:alert`,不得使用无租户前缀 topic。

### 2. source_ref 统一规范
- 格式:`<来源类型>:<年份>:<资源类型>:<资源id>`,如 `experiment:2026:instance:55`、`contest:2026:55`。
- 资源类型用全称不缩写(`instance` 不写 `inst`),保证 M2/M3/M4 按 source_ref 级联回收/审计匹配一致。

### 3. 对象存储(MinIO)路径规范
- bucket 按环境划分;路径前缀统一:`{tenant_id}/{模块}/{资源类型}/{资源id}/...`。
  - 各模块前缀:M2 `sandbox`、M3 `judge`、M4 `sim`、M5 `content`、M6 `course`、M7 `experiment`、M8 `contest`、M11 `transcript`。
- **学生代码主从约定**:学生在沙箱写的代码以 **M2 持久化为主**(`{tenant}/sandbox/...`);M3 判题、M6 作业、M8 竞赛提交存的是**指向该代码的引用 + 提交时刻的 code_hash 快照**,不另存整份副本(消灭冗余)。

### 4. 文件服务(横切能力)
- 统一文件上传/下载服务,提供:大小限制、类型白名单、病毒扫描、租户配额、防盗链直链鉴权。当前统一下沉到后端 `internal/platform/storage` 作为**唯一对外入口**;`internal/platform/upload` 只保留纯上传安全原语(大小/类型/归档/病毒扫描),不得再被模块直接拼装出第二套文件服务。
- M5 附件、M6/M7 报告、M11 成绩单 PDF 等所有文件存取走此统一服务,各模块不各写一套。

### 5. 审计日志
- 全平台共享 **M1 的 `audit_log` 单表**;各模块产生关键操作写入(以 `action` + `target_type` 区分来源)。M9 是对该表的查询中心,不跨多表聚合。
- `actor_role` 统一枚举:1平台管理员、2学校管理员、3教师、4学生、5已通过服务签名的内部系统任务。内部系统任务没有用户账号语义,`actor_id=0`,必须在 `detail.source_ref` 中记录已验证来源。

### 6. 私有化职责映射统一规则
- 私有化部署**无平台管理员**:所有 `/platform/*` 接口关闭;原平台管理员职责(系统公告全平台发布、全局看板、全局配置)由该校**学校管理员**承接本校范围。
- 各模块涉及"平台管理员"的功能,私有化下自动按上述规则映射,不再逐模块重复声明。

### 7. 跨租户受控场景
- 跨校共享(M5/M6 共享库)、跨校组队(M8):为**显式受控例外**。跨租户访问仅限被授权的具体资源范围;跨校队员的通知经 M10 按"竞赛 topic 授权订阅"投递,不开放其他租户数据。

### 8. 登录后落点
- **不设独立工作台/仪表盘落地页**。登录成功后按角色直达侧边栏第一个功能页:学生→课程页、教师→课程页、学校管理员→用户管理页、平台管理员→学校管理页。
- 与 M1 角色一致;私有化无平台管理员落点。

### 9. 统一导入导出中心(横切)
- 统一异步批量导入导出能力:任务化(进度/失败重试/下载中心),供 M1 师生导入、M6 成绩、M5 题库批量、M8 报名名单、M9 审计、M11 成绩单等复用。当前统一下沉到后端 `internal/platform/transfer`,只承载通用任务模型、重试语义、产物对象引用与下载中心能力,不能并入 M9 之类第 3 层聚合模块,避免下层模块反向依赖上层。

### 10. 帮助与新手引导(横切)
- 帮助中心 + 四类角色首登引导(onboarding),降低"建课→出题→组赛"等长链路上手成本。前端层实现,内容可配置。

> 说明:全局搜索、聚合待办工作台 经评估**不做**(登录直达功能页即可,无需聚合落地页)。

### 11. 多步骤流程状态持久化(强制)
- 任何**多步骤/长表单/向导类**流程的中间状态必须**持久化到服务端**,前端刷新、关闭重开、换设备都不丢失。前端 localStorage 仅作即时兜底,服务端为权威。
- 适用场景(已知):
  - M7 实验编排向导(6 步):每步配置自动存草稿,刷新回到原步骤。
  - M6 作业作答草稿(已设计:localStorage + 服务端双存,60s 自动保存)。
  - M8 竞赛出题向导、对抗赛参战物提交、报名组队流程。
  - M5 题目/实验模板创作(草稿态)。
  - M1 师生导入(预览→提交两步,预览结果暂存)。
  - 学校入驻申请填写。
- 实现:草稿数据落库(各模块自有草稿表/草稿字段或通用 `draft` 机制),带 owner + 过期清理;提交成功后草稿失效。
- 配置/环境变量同理不在前端持有敏感中间态。

## 四、模块蓝图

按依赖层次划分,下层是上层地基,建设顺序由此而定。

### 第 0 层 · 地基
- **M1 身份与租户**:多租户隔离、学校申请审核、角色体系、导入式账号管理、登录鉴权。一切模块的前置。

### 第 1 层 · 共享能力引擎
- **M2 区块链适配与沙箱引擎**:插件化运行时(EVM/Fabric/自研链…)、容器编排隔离。
- **M3 评测引擎**:可插拔判题器(测试用例 / 链上状态校验 / flag 检查 / 仿真检查点)。
- **M4 仿真可视化引擎**:交互式演示共识、哈希、默克尔树、P2P 传播等,服务教学与仿真实验。
- **M5 题库与模板中心**:实验模板、竞赛题、理论题的统一存放与复用枢纽。

### 第 2 层 · 三大业务模块
- **M6 教学**:课程/章节/内容、班级、学习进度、作业、**单课程成绩**(权重/汇总/调分/导出)。
- **M7 实验**:代码实验(Web IDE + 沙箱)+ 仿真实验,教师可配置。依赖 M2/M3/M4/M5。
- **M8 竞赛**:解题赛/对抗赛/理论赛、报名赛制、排行榜、防作弊、回放。依赖 M2/M3/M5。

### 第 3 层 · 运营支撑(横切)
- **M9 管理后台**:平台/学校运营看板、审计查询、运维配置与业务入口视图;学校审核、师生导入、租户变更等业务命令仍由所属模块自有接口执行。
- **M10 通知与实时推送**:WebSocket 底座,服务排行榜、对局状态、消息。
- **M11 成绩中心**:跨课程成绩聚合、GPA、学业预警、成绩单。单课程成绩由 M6 产出,M11 只做跨课程聚合(只读各模块成绩,不跨模块写)。

## 五、模块依赖与建设顺序

```
M1 身份与租户
  └─► M2 沙箱引擎 ─┐
      M3 评测引擎 ─┤
      M4 仿真引擎 ─┼─► M6 教学
      M5 题库中心 ─┘     M7 实验
                          M8 竞赛
                            └─► M9 管理后台 / M10 实时推送(按需穿插)
```

**建设顺序**:M1 → (M2、M3、M4、M5 共享引擎) → (M6、M7、M8 业务) → (M9、M10、M11 运营与成绩)。
