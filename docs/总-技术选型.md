## 1. 选型总览

| 层 | 选型 | 说明 |
| --- | --- | --- |
| 后端语言 | **Go** | 高并发、K8s 生态原生(client-go)、容器调度主场 |
| 后端架构 | **模块化单体(Modular Monolith)** | 11 模块强边界 package,接口调用,保留拆微服务能力 |
| 前端 | **React + TypeScript** | 生态最大,区块链前端库(wagmi/ethers)、Monaco、可视化(Canvas/SVG/D3)支持最好 |
| 数据库 | **PostgreSQL** | JSONB、行级安全 RLS 天然适配多租户隔离 |
| 数据访问 | **sqlc**(SQL→Go 代码生成)+ **golang-migrate/goose** 迁移 | SQL-first,RLS 透明可控;迁移版本化 |
| 配置 | **环境变量 + env(viper/envconfig)** | 统一注入,禁硬编码;密钥走 Secret/KMS |
| 缓存/会话 | **Redis** | 未读计数、在线状态、限频、队列 |
| 容器编排 | **Kubernetes(SaaS)/ k3s(私有化)** | 沙箱隔离、资源调度、弹性 |
| 事件总线 | **NATS** | 通知/事件解耦,发送方不阻塞 |
| 对象存储 | **MinIO**(兼容 S3) | 代码/附件/报告/PDF 持久化 |
| 镜像仓库 | **Harbor** | 私有仓库 + Trivy 扫描 + Cosign 签名 |
| 监控 | **Prometheus + Grafana**(外接) | 基础设施监控,M9 嵌入展示 |
| 实时通信 | **WebSocket** | M10 统一 Hub |
| CI/CD | **GitHub Actions / GitLab CI** | 路径触发、构建、扫描、签名、部署 |

## 2. 后端:Go 模块化单体

### 2.1 为什么模块化单体而非微服务
- 目标"一次性完整实现",团队非超大型;微服务带来服务治理、分布式事务、跨服务调试的高成本,拖慢交付。
- 模块化单体:11 模块在代码里是**强边界独立 package**,模块间只能经定义好的接口(即各模块文档的跨模块接口)调用,**不能直接读对方表/内部函数**。
- 既享受单体的部署简单、开发快、调试易,又因边界已划清而保留未来拆微服务的能力。

### 2.2 例外:独立伸缩的执行单元
- M2 沙箱的"学生实验沙箱、judge 判题沙箱、对局沙箱"本就是 K8s 里的独立 Pod(跑不可信代码、需隔离与弹性),由控制面调度。这是 M2 设计本身决定的,不违背单体——单体指的是**控制面/业务逻辑**单体,数据面容器独立。

### 2.3 Web 框架与库
- HTTP:Gin / Echo(任一,统一即可)。
- K8s 交互:client-go / controller-runtime(M2 沙箱编排)。
- **数据访问层:sqlc**(写 SQL → 生成类型安全 Go 代码)。SQL-first,RLS 多租户行为完全透明可控,避免重 ORM 隐藏越权风险。底层驱动 pgx。
  - 不用 Prisma(Node 生态,Go 无官方支持);不用重 ORM(GORM)以保证 RLS 显式。
- **数据库迁移:golang-migrate 或 goose**(版本化 SQL 迁移)。管理 83 张表 + RLS 策略 + 索引 + 种子数据(seed 镜像元信息/初始管理员)。迁移作为独立步骤(`chaimir/migrate` 镜像 + `scripts/`),CI/部署时执行。
- 鉴权:JWT(双 Token),见 M1。

### 2.4 配置管理(强制规范)
- **所有配置统一通过环境变量注入**,应用启动时读取(如 viper / envconfig),**禁止任何硬编码**(端口、连接串、密钥、第三方端点、阈值等一律不写死在代码里)。
- 提供 `.env.example` 模板列出所有配置项;本地用 `.env`,生产用 K8s ConfigMap/Secret 注入。
- 密钥类(JWT 签名、加密密钥、DB/中间件凭据)走 K8s Secret/KMS,绝不入代码、不入镜像、不入仓库。
- 配置分层:全局默认 → 环境覆盖 → 运行时(部分租户级配置见 M9 system_config)。

## 3. 前端:React + TS 四端

- **四端**(对应 M1 四角色,无独立工作台,登录直达角色第一个功能页):
  - 学生端 → 课程页 / 教师端 → 课程页 / 学校管理员端 → 用户管理页 / 平台管理员端 → 学校管理页。
- **关键库**:
  - 代码编辑:Monaco(Web IDE 类工具)。
  - 终端:xterm.js(接 M2 K8s exec)。
  - 仿真可视化:Canvas/SVG/WebGL(M4 SDK),数学公式 KaTeX。
  - 区块链交互:ethers.js / viem(按需)。
- 中文为主;国际化为扩展点(当前不实现,内容表预留语言字段)。

## 4. 数据与存储

- **PostgreSQL**:主库;RLS 强制多租户隔离(`app.tenant_id` 会话变量),雪花 ID 主键,软删 `deleted_at`,枚举用 SMALLINT。详见《数据库表总览》。
- **Redis**:缓存、未读计数、在线状态、限频、判题/撮合队列。
- **MinIO**:统一对象存储,路径 `{tenant_id}/{模块}/{资源类型}/...`(见总纲跨模块约定)。

---

## 5. 容器与编排

- **K8s(SaaS)**:沙箱/判题/对局/后端计算仿真以 Pod 运行;NetworkPolicy 默认 deny-all;ResourceQuota/LimitRange;Pod Security Restricted;按需 gVisor/Kata 加固高风险沙箱。
- **k3s(私有化)**:轻量单集群,逻辑与 SaaS 一致。
- **Harbor**:镜像仓库,DaemonSet 全节点预拉取(秒进优化);Trivy 扫描 + Cosign 签名门禁。

---

## 6. 异步与实时

- **NATS**:事件总线(通知、跨模块事件),失败重试 + 死信队列。
- **WebSocket**:M10 统一 Hub,业务实时数据(排行榜/进度/通知);引擎内部流(终端 PTY/仿真 stream)走各模块自有 WS。
- **判题/撮合队列**:Redis/NATS 实现优先级队列,worker 横向扩展。

---

## 7. 监控与运维

- **Prometheus + Grafana**:基础设施监控(CPU/内存/服务健康),M9 嵌入展示,不自建监控引擎。
- **业务级告警**:M9 自有(待审积压/配额超限等)。
- **日志**:结构化日志 + 集中采集(如 Loki,可选)。

---

## 8. 各模块语言补充说明

虽以 Go 为主,少数场景按特性补充(均为插件/执行器形态,不破坏单体):
- **M3 判题执行器**:可用任意语言容器(跑学生代码的判题逻辑,如 Solidity 测试用 Node/Foundry 镜像)。
- **M4 仿真**:前端 TypeScript 运行时;重计算仿真后端 Go 同协议。
- **M2 运行时镜像**:各链官方工具链(geth/Fabric SDK 等),语言随链。

> 这些是"插件/镜像内部"的语言,平台控制面统一 Go。
