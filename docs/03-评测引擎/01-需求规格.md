# M3 评测引擎 — 需求规格

> 模块定位:判题器。接收一次提交,调用 M2 沙箱执行判定,产出单次判定结果。被 M7 实验、M8 竞赛、M6 作业复用。
> 核心理念:只判单次、不算成绩。判题与成绩解耦(成绩聚合归 M6)。
> 最后更新:2026-05-29

---

## 1. 模块目标

- **可插拔判题**:6 类判题器,新判法 = 加一个插件,平台代码不改。
- **安全判题**:judge 专用沙箱,与学生交互沙箱完全隔离,判题套件与答案对学生黑盒。
- **确定性**:同一提交多次判定结果一致,可复现、可重判。
- **可解释**:不只给分,给出"为什么"(哪个用例挂、哪条断言不符)。
- **高并发**:异步队列,支撑竞赛瞬时大量提交。

## 2. 职责边界

| M3 负责 | M3 不负责(归属) |
| --- | --- |
| 判题器插件、判题调度 | 提供运行环境(→ M2) |
| 单次判定结果(分/通过/详情) | 最终成绩、GPA、成绩单(→ M6) |
| 提交级防作弊(限频/查重/快照) | 行为监控、违规处理(→ M8) |
| 相似度计算能力 | 作弊判定的业务决策(→ M8) |
| 判题输入快照、可复现 | 题目内容管理(→ M5 题库) |

**核心交互**:调用方传 `提交内容 + 判题配置(判题器类型+套件+期望) + 来源标识` → M3 异步判定 → 回写 `结果(分数/通过/可解释详情)`。

## 3. 判题器类型(可插拔)

| 编号 | 判题器 | 说明 | 依赖 |
| --- | --- | --- | --- |
| J1 | 测试用例判题 | 跑教师测试套件(如 Hardhat test),按通过数给分 | M2 沙箱 |
| J2 | 链上状态断言 | 部署学生合约后查链上状态(余额/存储/事件) | M2 链上能力 |
| J3 | Flag 判题 | 静态 / 动态 HMAC / 链上验证 flag | 部分依赖 M2 |
| J4 | 代码静态检查 | gas 消耗、SWC 安全漏洞、规范 | 静态,可不起链 |
| J5 | 仿真检查点 | 学生操作是否达成目标 | M4 仿真 |
| J6 | 人工评分 | 主观题/项目,教师录入分数与评语,不自动判 | 无 |

## 4. 功能需求

### A. 判题器管理
- A1 判题器注册:声明判题器类型、执行器、所需沙箱运行时、超时/资源。
- A2 判题器自检:接入时跑样例提交验证可用。

### B. 提交与判题
- B1 接收提交:调用方提交内容 + 判题配置 + 来源标识,入队。
- B2 异步判题:worker 消费,申请 judge 沙箱,注入代码+套件,执行判定。
- B3 结果回写:分数 / 通过与否 / 可解释详情,通知调用方。
- B4 判题超时与重试:超时终止,失败按策略重试。
- B5 优先级队列:竞赛提交可设高优先级。

### C. 结果与可复现
- C1 输入快照:记录学生代码 hash + 判题套件版本 + runtime_image_version + genesis_ref + 判题器版本。
- C2 可解释结果:逐用例/逐断言的期望值 vs 实际值。
- C3 重判:按快照原样重跑(申诉、判题器修复后)。

### D. 提交级防作弊
- D1 提交限频:同一用户/题目限频,防轰炸。
- D2 代码指纹:计算提交哈希/相似度,供查重。
- D3 提交留痕:每次提交快照存档。

## 5. 非功能需求

- **确定性**:禁判题依赖随机数/真实时间/外部网络;固定镜像与创世状态。
- **隔离**:judge 沙箱独立、用后即毁,学生不可见判题套件。
- **可扩展**:judge worker 横向扩展;判题器插件化。
- **多租户**:判题任务带 tenant_id,资源按租户隔离。

## 6. 明确不做(边界)

- 不提供运行环境(用 M2)、不算最终成绩(归 M6)。
- 不做行为级防作弊与违规处理(归 M8)。
- 不管题目内容与正确答案的存储(归 M5 题库,M3 判题时取用)。

## 7. 技术取向

Go 实现判题调度与 worker;复用 M2 沙箱执行;异步队列(Redis/NATS);判题器以容器化执行器形式插件化。最终技术细节由《技术选型》统一收口。
