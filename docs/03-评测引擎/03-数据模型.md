# M3 评测引擎 — 数据模型

> PostgreSQL。判题任务/结果为租户级(tenant_id + RLS);判题器定义为平台级。
> 建模遵循:消灭数据冗余、表合并有度。
> 最后更新:2026-06-08

---

## 1. 表清单总览

| 表名 | 用途 | 范围 |
| --- | --- | --- |
| `judger` | 判题器定义 | 全局 |
| `judge_task` | 判题任务(一次提交一条) | 租户 |
| `judge_result` | 判题结果历史(含可解释详情,多版本保留) | 租户 |
| `judge_event_outbox` | 判题终态事件 outbox,与结果同事务写入后可靠发布 | 租户 |
| `submission_fingerprint` | 提交指纹(查重) | 租户 |

> 消灭冗余/合并说明:
> - **判题结果与判题任务一对多版本**:普通查询只取最新版本,重判/人工改分追加新版本,不覆盖历史结果。
> - **判题配置(判题器类型+套件引用+期望)不在 M3 存正本**:正本在 M5 题库,judge_task 仅存"配置引用 + 输入快照",不冗余拷贝题目内容。
> - 提交内容本身(代码)存对象存储,表中只存引用与 hash。

## 2. 平台级表

### 2.1 judger — 判题器定义

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | BIGINT PK | |
| code | VARCHAR(64) UNIQUE | 判题器标识,如 `testcase`/`onchain-assert`/`flag`/`static-scan`/`sim-checkpoint`/`manual` |
| name | VARCHAR(128) | 展示名 |
| type | SMALLINT | 1测试用例/2链上断言/3Flag/4静态检查/5仿真检查点/6人工 |
| executor_ref | VARCHAR(128) | 执行器标识;J1/J4 指向容器化执行器,J2/J3/J5/J6 指向 M3 后端内置策略 |
| runtime_required | BOOLEAN | 是否需要起链沙箱 |
| default_timeout_sec | INT | 默认判题超时 |
| resource_spec | JSONB | judge 沙箱资源需求;运行型判题器必须包含 `runtime_code`、`runtime_image_version`、`genesis_ref`、`command`、`timeout_sec`、`max_retries` 等可复现执行配置 |
| selftest_status | SMALLINT | 1待测/2通过/3失败 |
| status | SMALLINT | 1可用/2停用 |
| created_at / updated_at | TIMESTAMPTZ | |

## 3. 租户级表

### 3.1 judge_task — 判题任务

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | BIGINT PK | |
| tenant_id | BIGINT | |
| judger_id | BIGINT | 使用的判题器 |
| source_ref | VARCHAR(128) | 来源标识(实验/竞赛/作业,M3 不解析语义) |
| source_owner_id | BIGINT | 来源归属账号快照,用于教师侧查询/重判/人工评分防横向访问 |
| source_course_id | BIGINT | 来源课程快照;教学来源必须传,其他来源可为 0 |
| source_scope | VARCHAR(32) | 来源类型快照,如 teaching/exp/contest |
| submitter_id | BIGINT | 提交者账号 |
| problem_ref | VARCHAR(128) | 题目引用 = M5 `item_code:item_version`(不拷贝内容) |
| code_storage_key | VARCHAR(255) | 提交代码在对象存储的路径 |
| code_hash | VARCHAR(64) | 提交内容哈希(查重/复现) |
| input_snapshot | JSONB | 输入快照:套件版本/`runtime_image_version`/`genesis_ref`/判题器版本;重判时追加 `rejudge=true` 标记 |
| sandbox_mode | SMALLINT | 1=fresh新起干净沙箱 / 2=reuse复用现场沙箱 |
| target_sandbox_ref | VARCHAR(64) NULL | reuse 模式指向调用方现有 M2 沙箱 |
| priority | SMALLINT | 队列优先级 |
| status | SMALLINT | 见状态机:queued/judging/done/timeout/error/failed/cancelled |
| retry_count | INT | 系统性失败/超时已重试次数 |
| max_retries | INT | 本任务最大重试次数,来自 M3 配置或判题器策略 |
| created_at / updated_at | TIMESTAMPTZ | |

> `source_ref` 是 M3 对调用方提交的幂等键,`judge_task` 必须保证 `(tenant_id, source_ref)` 唯一。同一来源因 M6/M8 outbox 重试或调用方网络重试再次提交时,M3 返回既有任务,不得创建第二条判题任务、第二份指纹或重复执行学生代码。需要重新判题时必须走 `rejudge` 流程,而不是复用提交接口绕过幂等键。

> `source_owner_id`、`source_course_id`、`source_scope` 是访问控制快照,数据库层强制非空:`source_owner_id > 0`,`source_course_id >= 0`,`source_scope IN ('teaching','exp','contest')`。教学来源必须携带课程 ID;不得用默认值、`unknown` 或 `extra_input` 旁路这些字段。

### 3.2 judge_result — 判题结果

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | BIGINT PK | 结果版本记录 ID |
| task_id | BIGINT | 对应 judge_task |
| tenant_id | BIGINT | |
| version | INT | 同一任务内结果版本号,从 1 递增 |
| passed | BOOLEAN | 是否通过 |
| score | INT | 得分 |
| max_score | INT | 满分 |
| details | JSONB | 可解释详情(逐用例/逐断言:期望摘要 vs 实际 + hint),不得保存完整答案、flag、私钥、套件源码或可直接复原答案的结构 |
| judge_sandbox_ref | VARCHAR(128) | 本次判题沙箱引用(审计) |
| judged_at | TIMESTAMPTZ | 判题完成时间 |
| is_rejudge | BOOLEAN | 是否重判产生 |

> 普通任务详情和列表只返回最新 `version` 的结果。每次自动判题完成、人工评分或重判完成均插入新的 `judge_result` 行,保留历史用于申诉、审计和复现。

### 3.3 judge_event_outbox — 终态事件 outbox

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | BIGINT PK | |
| tenant_id | BIGINT | |
| task_id | BIGINT | 判题任务 |
| subject | VARCHAR(128) | `judge.completed` / `judge.failed` |
| payload | JSONB | 事件载荷 |
| status | SMALLINT | 1=pending / 2=published / 3=failed |
| retry_count | INT | 发布失败重试次数 |
| next_attempt_at | TIMESTAMPTZ | 下次允许发布时间;失败后指数退避,避免 poison/stuck 记录阻塞后续事件 |
| last_error | VARCHAR(255) NULL | 脱敏后的最后失败原因 |
| created_at / updated_at | TIMESTAMPTZ | |

> 终态事件必须与 `judge_result`/`judge_task.status` 在同一租户事务内写入 outbox。worker 事务提交后发布 NATS,发布成功再标记 `published`;发布失败保留 pending/failed 状态供下一轮重试,不得把已经完成的判题结果回滚成 error 或重新执行学生代码。

### 3.4 submission_fingerprint — 提交指纹

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | BIGINT PK | |
| tenant_id | BIGINT | |
| source_ref | VARCHAR(128) | 来源(同题范围内查重) |
| problem_ref | VARCHAR(128) | 题目引用 |
| submitter_id | BIGINT | 提交者 |
| code_hash | VARCHAR(64) | 精确哈希(完全相同检测) |
| sim_vector | JSONB | M3 从提交代码对象提取的归一化 token 特征向量(供相似度计算) |
| created_at | TIMESTAMPTZ | |

> M3 仅提供相似度计算与指纹,"判定作弊"由 M8 决策。

## 4. 多租户隔离

- 租户级表带 `tenant_id` + RLS(同 M1)。
- `judger` 平台级,仅平台管理员可写。
- 私有化:`tenant_id` 恒定,逻辑一致。

## 5. 关键索引

- `judger(code)` UNIQUE
- `judge_task(tenant_id, status, priority)`(队列扫描)
- `judge_task(tenant_id, source_ref)` UNIQUE(提交幂等)、`judge_task(submitter_id)`
- `submission_fingerprint(tenant_id, problem_ref, code_hash)`(精确查重)
