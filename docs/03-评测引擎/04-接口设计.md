- Base:`/api/v1/judge`
- 统一响应体同 M1。错误码段:`3xxxx`(31xxx 判题器/32xxx 任务/33xxx 查重)。

## 2. 判题器管理 `[平台管理员]`

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/judgers` | 判题器列表 |
| POST | `/judgers` | 注册判题器 |
| PATCH | `/judgers/{id}` | 更新配置 |
| POST | `/judgers/{id}/selftest` | 样例提交自检 |

运行型判题器的 `resource_spec` 必须固定判题环境和执行策略:
```json
{
  "runtime_code": "evm-hardhat",
  "runtime_image_version": "2026.06.01",
  "genesis_ref": "runtime-image:2026.06.01:baked-genesis",
  "tool_codes": ["terminal"],
  "init_script_ref": "minio://chaimir-code/judgers/init.sh",
  "command": ["npm", "test", "--", "--reporter", "json"],
  "timeout_sec": 60,
  "max_retries": 2
}
```
`runtime_image_version` 由 M3 传给 M2 创建 fresh judge 沙箱,M2 必须按该版本选择已预拉取且固化创世态的运行时镜像;不得用当前默认镜像替代已提交任务的快照版本。

## 3. 提交判题 `[内部]`

### 3.1 提交
`POST /tasks` `[内部]`
请求:
```json
{
  "judger_code": "",              // 可空;为空时使用 M5 锁定题目版本的 judge_config.judger_code
  "item_code": "prob-2031",        // M5 题目 code(统一格式,见 §3.5)
  "item_version": "1.0.0",         // M5 题目锁定版本
  "code_storage_key": "minio://.../submission", // 仅代码类判题器必填
  "code_hash": "64位sha256hex",                 // 仅代码类判题器必填
  "submitter_id": 1001,
  "source_owner_id": 1001,         // 来源归属账号;教师侧查询/重判/人工评分必须据此防越权
  "source_course_id": 0,           // 教学来源必填课程 ID,其他来源可为 0
  "source_scope": "experiment",    // teaching/experiment/contest 等来源类型
  "sandbox_mode": "fresh",         // fresh=新起干净judge沙箱 / reuse=复用现场沙箱
  "target_sandbox_ref": null,      // sandbox_mode=reuse 时必填,指向调用方现有 M2 沙箱
  "extra_input": {},               // 可选:flag 提交值/仿真检查点快照等,只用于本次输入快照
  "priority": 2
}
```
响应为跨模块契约 `contracts.JudgeTaskInfo`:`{ "task_id": 8801, "status": 1 }`,状态码含义见 §6;用户向查询与进度接口返回字符串状态。

内部 HTTP 调用的 `tenant_id` 与 `source_ref` 来自服务签名头(`X-Chaimir-Tenant-Id` / `X-Chaimir-Source-Ref`),请求体不得再携带第二份来源标识;Go 模块内同步调用仍通过 `contracts.JudgeSubmitRequest.SourceRef` 传递同一契约字段。

`source_owner_id`、`source_course_id`、`source_scope` 是 M3 访问控制的一等契约字段,不得放入 `extra_input` 隐式传递。教学来源必须传课程教师账号与课程 ID;实验来源传实例归属账号与课程 ID(无课程时为 0);竞赛来源传提交者或对局归属账号,课程 ID 为 0。

`judger_code` 为空时,M3 必须使用 M5 锁定题目版本的 `judge_config.judger_code`;调用方不得为了解题赛、作业或实验硬编码模块内判题器别名。若调用方显式传入 `judger_code`,M3 必须与 M5 判题配置逐字匹配,不一致即拒绝提交。

`code_storage_key`/`code_hash` 仅代码类判题器使用。J1 测试用例、J4 静态检查、fresh 模式 J2 链上断言必须携带二者;J3 Flag、J5 仿真检查点、J6 人工评分、reuse 模式 J2 链上断言必须留空,不得上传空归档或伪代码对象作为旁路。Flag 提交值必须按 `flag_input_key` 直接放入 `extra_input`,仿真检查点必须放入 `extra_input.checkpoint`。

### 3.5 两种判题沙箱模式

| sandbox_mode | 说明 | 适用 |
| --- | --- | --- |
| `fresh`(默认) | M3 向 M2 申请全新干净 judge 沙箱,注入代码+套件判定,用后即毁 | 测试用例判题、独立判定;学生不可见判题环境 |
| `reuse` | 复用调用方传入的 `target_sandbox_ref` 现场沙箱,对其链上状态执行断言 | 链上状态断言:需读学生当前实验/竞赛沙箱的真实链状态 |

> reuse 模式下,M3 只读现场沙箱的链上状态做断言(经 M2 链上能力接口),不注入测试套件、不暴露答案;判定逻辑仍隔离执行。
> item_code/item_version 统一对应 M5 的 `(code, version)`,M3 据此调 M5 `GET /items/{code}/{version}/full` 取判题配置与答案。
> fresh 模式下,只有代码类判题器必须读取 `code_storage_key` 指向的对象,校验对象内容 SHA-256 与 `code_hash` 一致,按 `JUDGE_INPUT_ARCHIVE_MAX_FILES` / `JUDGE_INPUT_ARCHIVE_MAX_UNPACKED_BYTES` 校验归档成员并重打包后注入 judge 沙箱;不得直接把原对象引用作为沙箱 `InitCodeRef`。非代码判题器即使使用 fresh 模式,也必须通过结构化 `extra_input` 或人工评分接口传入判定输入。

### 3.6 判题器 `expectation` 约定

M5 通过 `ContentJudgeSpec.Expectation` 给 M3 提供黑盒判题配置。M3 不保存题目正本,只把本次锁定版本写入 `input_snapshot`。

| 判题器 | expectation 关键字段 | 说明 |
| --- | --- | --- |
| J1 测试用例 | `command_result=json` | 判题命令 stdout 必须输出 `{passed,score,max_score,details}`;`details` 为非空数组,每项至少含 `passed` 和 `case/source/target` 之一 |
| J2 链上断言 | `chain_steps[]`、`assertions[]` | `chain_steps` 支持 `deploy/tx/query/reset`;fresh 模式先注入学生提交代码,reuse 模式只读现场沙箱状态;`assertions` 支持 `eq/ne/contains/exists`;无断言为配置非法;失败详情只含 `expected_label`/`actual`,不得返回完整期望结构 |
| J3 Flag | `flag_hash`、`flag_input_key` | M5 只提供已固化的 64 位 HMAC-SHA256 `flag_hash`;M3 不接收明文 flag 或 HMAC secret;worker 运行时只接受 `flag_hash`;提交值入库前统一固化为 `submitted_flag_hash` |
| J4 静态检查 | `command_result=json` | 与 J1 一样解析结构化 stdout,但不要求链运行时 |
| J5 仿真检查点 | `checkpoint` | 与 `extra_input.checkpoint` 做结构化相等比较;结果详情只含 `expected_label`/`actual`,不得返回完整标准检查点 |
| J6 人工评分 | 无 | 不进 worker,教师接口写入结果;`details` 结构化保存 `comment/score/max_score/passed` |

判题器命令输出必须是结构化 JSON,不得把完整测试套件或答案回传给学生端。J2/J5 的 `expected_label` 只能是教师配置的断言标签、目标字段、操作符或脱敏短文本,不能直接放标准答案结构。J3 只允许 M5 提供已固化 `flag_hash`,M3 在构建 `input_snapshot` 时只保存学生提交值的 `submitted_flag_hash`;重判使用快照中的 hash,不再读取或保存答案正本。

### 3.2 进度(WebSocket)
`WS /tasks/{id}/progress`
推送:`{ "status": "judging", "message": "正在 judge 沙箱中执行测试" }`

### 3.3 查询结果
`GET /tasks/{id}`
响应:
```json
{ "task_id": 8801, "status": "done",
  "result": {
    "passed": false, "score": 60, "max_score": 100,
    "details": [
      { "case": "转账后余额正确", "passed": true },
      { "case": "超额转账应 revert", "passed": false,
        "expected_label": "应拒绝超额转账", "actual": "转账成功", "hint": "缺少余额校验" }
    ],
    "snapshot_ref": "judge:2026:sub:8801"
  }}
```

### 3.4 取消
`DELETE /tasks/{id}` `[内部]`:取消排队中的任务。

## 5. 人工评分 `[教师]`

J6 人工评分判题器对应接口:

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/tasks?source_ref=&pending_manual=true` | 待人工评分列表 |
| POST | `/tasks/{id}/manual-score` | 录入分数 + 评语,生成 judge_result |

教师侧查询、重判和人工评分必须校验 `source_owner_id`/提交者归属;缺少来源归属快照的历史任务不得按租户全量返回给普通教师。学校管理员如需跨教师巡检应走后续明确的聚合只读接口,不得复用普通教师列表绕过来源校验。

## 6. 查重能力 `[内部]`

> M3 提供能力,M8 做作弊判定。
> `submission_fingerprint.sim_vector` 由 M3 在提交入队前读取 `code_storage_key` 指向的代码对象生成,不接受前端上传的相似度向量,避免伪造查重特征。
> `threshold` 为空时使用 `JUDGE_SIMILARITY_DEFAULT_THRESHOLD`;阈值必须由部署配置进入 `platform/config`,不得在 M3 service 内写死。

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/fingerprints/exact?problem_ref=&code_hash=` | 精确哈希查重(完全相同提交) |
| POST | `/fingerprints/similarity` | 传入待比较提交对象引用,由 M3 读取对象生成特征向量并返回相似对 |

## 7. 典型时序(竞赛提交判题)

```
M8 → POST /tasks(problem_ref+source_ref+code或answer, priority=高)
M3 → task_id, queued
前端 → WS /progress
M3 worker → 申请 judge 沙箱 → 注入 → 判定 → 销毁
M3 → status=done, 回写 result
M3 → 发布判题完成事件,M8 据此更新排行榜
```
