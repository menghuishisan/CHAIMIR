# 工程目录设计

> Monorepo 单仓 + Go 模块化单体。本文规范每个目录的职责("该放什么/不该放什么"),用模块边界强约束,避免开发时职责偏离。
> 最后更新:2026-05-29

---

## 1. 顶层结构(Monorepo)

```
chaimir/
├── backend/            后端(Go 模块化单体)
├── frontend/           前端(React + TS,四端)
├── images/             全量镜像治理(平台构建镜像 + 上游固定镜像 + 私有化离线清单)
├── deploy/             部署配置(K8s/k3s manifests、Kustomize、CI/CD)
├── docs/               设计文档(本目录)
├── scripts/            运维/初始化脚本(建库、seed、镜像预拉取)
└── README.md
```

> 单仓统一版本;跨模块改动一次提交;契约同步。

### 2.0 装配方式与配置

- **每模块一个装配文件**:`cmd/server/<module>.go` 各负责一个模块的装配(构造模块 store/repo、构造服务、注入它依赖的 contracts 实现、注册 HTTP 路由与事件订阅)。`main.go` 只做总入口:加载配置 → 初始化基础设施(db/redis/nats/k8s)→ **按依赖层次顺序**调用各模块装配函数 → 启动 HTTP/WS。装配顺序遵循分层(地基→引擎→业务→聚合),保证依赖就绪。
- **服务构造不接收数据库**:`service.go` 与 `service_<domain>.go` 不得 import `platform/db`,不得在 `NewService` 内部调用 `newRepo(database)` 或构造 sqlc 查询对象。service 构造函数只接收模块内 store/repo 接口、contracts 依赖与必要基础设施能力(audit/eventbus/storage/k8s/ws/config 等)。数据库连接只在 repo/store 构造边界出现,由 `cmd/server/<module>.go` 装配后注入 service。禁止同时保留 `NewService(*db.DB)` 与 `NewServiceWithStore(...)` 这类兼容双轨。
- **部署期命令独立**:`cmd/migrate` 只负责 migration、应用角色授权和 seed 编排。首个学校管理员、首个平台管理员等业务初始化必须调用对应模块服务,不允许在 shell/SQL 中复制业务建账号逻辑。
- **配置只读环境变量**:`platform/config` 统一从环境变量/`.env` 读取(viper/envconfig),**全局禁止硬编码**任何端口/连接串/密钥/端点/阈值。`deploy/config/chaimir.env` 是全平台环境变量键的唯一来源,可登记后端、中间件、镜像/沙箱内服务的非密配置键,以及 `# secret-env:` 密钥键名;`backend/.env.example` 只列后端 `platform/config` 实际读取的运行配置,不得混入仅供镜像/沙箱内服务使用的键。生产经 K8s ConfigMap/Secret 注入;密钥走 Secret/KMS 不入仓。
- **后台任务统一运行**:`platform/background` 只提供通用 ticker、panic recovery、结构化日志与 context 停止语义;业务 worker/scheduler 必须留在所属模块 service 内,并暴露 `RunXOnce(ctx)` 供测试和重试。`cmd/server/<module>.go` 只负责把模块后台任务注册/启动,不得承载业务状态机。轮询间隔、批量大小、超时阈值全部来自 `platform/config`。
- **跨租户维护任务受控例外**:后台清理/回收这类系统维护任务如需扫描 RLS 租户表,只能在所属模块 repo 内通过 platform/db 的特权事务读取/锁定**本模块自有表**,随后按行内 `tenant_id` 回到普通租户事务完成业务状态变更。不得借此读取/写入其他模块表,不得把特权事务作为普通业务路径。
- **通用能力优先复用成熟库**:Excel/CSV、JWT、加密、K8s、对象存储、解析器等通用能力不得随手自研。若标准库或成熟第三方库能满足文档需求,必须复用;只有现有库无法支撑功能、安全、性能或架构边界时才允许自研,且需先在设计文档说明原因。

### 2.1 单模块内部结构(以 teaching 为例)

```
internal/modules/teaching/
├── api.go                 HTTP 路由注册、请求绑定、鉴权中间件、响应封装
├── api_<domain>.go        API 层按入口拆分;WebSocket/反向代理等 HTTP 边界也放这里
├── service.go             服务构造、依赖注入字段、模块主业务编排
├── service_<domain>.go    服务层按领域拆分;文件名前缀必须是 service_
├── repo.go                数据访问入口和通用事务封装,只操作本模块表
├── repo_<domain>.go       repo 按表组/领域拆分,仍只能做数据访问和事务
├── model.go               本模块领域模型、状态常量、内部快照类型
├── dto.go                 请求/响应结构,不放业务函数
├── convert.go             DTO/领域模型/契约对象之间的纯转换
├── row_convert.go         sqlc 行到领域模型的纯转换
├── rules.go               业务规则、状态机校验、输入校验
├── validation.go          复杂校验可独立拆分;不得访问 repo/db
├── events.go              事件订阅、解码、调用 service
├── audit.go               模块审计动作封装,只包装 platform/audit
└── internal/              模块私有,绝不被其他模块 import;sqlcgen 只能由本模块 repo 使用
```

#### 2.1.1 模块内文件职责细则

- **api / api_<domain>**:只做路由注册、HTTP/WS 协议接入、请求参数绑定、鉴权中间件组合、调用 service、统一响应。不得直接 import `internal/sqlcgen`、`platform/db`、`platform/eventbus`,不得直接调用 repo/sqlc 写库。WebSocket 连接升级、HTTP 反向代理、multipart 读取属于 API 边界;读取后的业务处理必须委托 service。
- **service / service_<domain>**:负责业务编排、权限校验、状态流转、调用 contracts、调用 repo/store 接口、触发审计/通知/事件。服务层可以使用外部 HTTP 客户端调用受控第三方源,但不得持有 Gin/response/httpx 写响应逻辑,不得出现 `http.ResponseWriter` 或 `gin.Context`。服务层不得 import `platform/db`、不得接收 `*db.DB`、不得构造 repo/sqlc;其持有的持久化依赖必须是本模块定义的接口。后台 worker/scheduler 留在所属模块 service,并暴露 `RunXOnce(ctx)` 供测试。
- **repo / repo_<domain>**:只封装本模块自有表的数据访问、sqlc 调用和事务边界。repo 返回模块领域模型、DTO 或真实内部快照,不向 service 泄漏 sqlc row 作为业务模型。repo 不调用 contracts/eventbus/storage/audit/ws,不做权限编排或跨模块查询。模块如需对外暴露构造入口,统一使用 `NewStore(database *db.DB)` 返回模块内 store 接口或未导出 repo 实现,不得让 service 参与 DB 装配。
- **model**:放模块领域模型、内部快照、状态常量和必要的轻量方法;不依赖 Gin、repo、sqlc 或跨模块实现。若 DTO 已经是模块统一输出模型,且 service 不需要额外领域字段或行为,不得再通过 `type X YDTO`、`type X = YDTO` 或同字段结构制造伪快照;只有真实内部投影才放在 model。
- **dto / enum**:只放请求/响应结构与枚举常量;不得放顶层业务函数。需要解析/转换时放到 convert/rules/service。
- **convert / row_convert**:只做纯映射,不得访问 repo/db/tenant/eventbus/storage/audit,不得发事件或写库。若转换只是为了兼容两套长期模型,应统一模型而不是保留双轨映射。
- **rules / validation**:只放业务规则、状态机校验和输入校验;不得访问 repo/db 或调用 contracts。需要读取数据才能判断的规则由 service 编排 repo 查询后再调用纯规则函数。
- **events**:只做订阅注册、事件解码、幂等校验入口和调用 service;不得直接访问 db/sqlc/repo 事务,不得在事件文件内写业务表。
- **audit**:只封装审计 action/target/detail 和 `platform/audit` 调用,不得自建审计表或绕过 identity 的 `audit_log`。
- **internal/sqlcgen**:只由 sqlc 生成,业务代码不得手改;除本模块 `repo*.go` 与纯映射的 `row_convert.go` 外,其他文件不得直接依赖 sqlcgen。

#### 2.1.2 模块内命名规则

- 服务层拆分文件统一命名为 `service_<domain>.go`,不得使用 `<domain>_service.go`。凡包含 `func (s *Service)` 的生产文件,除 `audit.go`、`events.go` 这类明确边界文件外,必须命名为 `service.go` 或 `service_<domain>.go`。
- repo 拆分文件统一命名为 `repo_<domain>.go`;只要文件中直接调用 sqlc 查询或承载事务,就应归入 repo 命名空间。
- API 拆分文件统一命名为 `api_<domain>.go`;只要文件承接 Gin、`http.ResponseWriter`、`*http.Request`、WebSocket 升级、反向代理或 multipart 协议细节,就应归入 API 命名空间。
- contracts 适配实现属于 service 层,统一命名为 `service_contract.go`;不得使用 `contracts.go` 放含 `Service` 方法的实现,避免与 `internal/contracts` 契约定义职责混淆。
- 文件名应表达职责而不是历史来源。重命名后必须同步测试守护,避免职责扫描因命名漂移漏检。

## 3. 模块边界强约束(核心,防职责偏离)

> 这是本文最重要的部分。违反即视为架构破坏,需 code review 拦截。

### 3.1 铁律
1. **模块只能通过 `internal/contracts` 的接口互相调用**,禁止 `import` 其他模块的 package。
   - ✅ `experiment` 调 `contracts.SandboxService`(由 `sandbox` 实现)。
   - ❌ `experiment` 直接 `import internal/modules/sandbox`。
2. **模块只能读写自己的表**。跨模块数据经接口取,不直接查对方表。
   - ❌ `admin` 直接 SQL 查 `course` 表;✅ 调 `contracts.TeachingStats`。
3. **聚合层(admin/grade)只读不跨写**:只能调其他模块的只读/内部接口,无直接写路径。若需驱动下层维护本模块自有投影,只能发布事件,不得直接调用下层写接口或跨写下层表。
4. **审计**:任何模块写审计都经 `platform/audit` 写入 identity 的共享 `audit_log`,不自建表。
5. **通知**:任何模块发通知都经 `contracts.NotifyService`(notify),不自己写通知表、不自建 WS 广播。

### 3.1.1 分层依赖规则(避免循环依赖)★

模块按依赖层次单向引用,**下层绝不反向依赖上层**,从根上杜绝循环依赖:

```
第0层 地基:      identity                        (不依赖任何业务模块)
第1层 共享引擎:  sandbox / judge / sim / content  (仅可依赖第0层)
                  └ judge 可依赖 sandbox、content;sim/sandbox/content 互不依赖
第2层 业务:      teaching / experiment / contest  (可依赖第0、1层,业务间不互相依赖)
第3层 聚合/横切:  admin / grade / notify           (可依赖下层只读接口;不被下层依赖)
```

规则细则:
- **单向**:高层依赖低层;低层接口里**不出现**高层类型。
- **同层不互相依赖**:如 teaching 不依赖 contest,sim 不依赖 sandbox(各自独立)。需要协作时由更高层(business 编排)或事件解耦。
- **反向通信用事件**:低层要通知高层(如 sandbox 回收完成通知 experiment),**不直接调高层接口**,而是发 `eventbus` 事件,高层订阅。这是打破"A调B、B需回调A"循环的标准手段。
  - 例:judge 判完不直接调 teaching,而发 `judge.completed` 事件,teaching/experiment/contest 各自订阅处理。
  - 例:sandbox 回收发 `sandbox.recycled` 事件,experiment 订阅后更新实例状态(替代直接回调)。
- **跨层投影同步也走事件**:若聚合层状态变化需要下层同步维护本模块自有投影(如 M11 审核通过后,M6 同步维护 `course_grade.is_locked` 写保护投影),也只能发布事件并由下层写回**自己的表**;聚合层不得直接写下层表或调用下层写接口。
- **回调用事件而非接口依赖**:之前文档里写的"M2 回收回调来源""M3 失败回调调用方",实现上**一律走事件总线**,不让 sandbox/judge 反向 import 业务模块。
- **CI 守护**:用依赖检查工具(如 `go-arch-lint` / import 规则)在 CI 阻断违规的跨层/反向/同层依赖。

### 3.2 contracts 目录职责
- 定义模块对外提供的 **interface** + 跨模块传递的 **DTO** + **事件类型定义**。
- 每个模块在 `main.go` 装配时注入依赖的 contracts 实现(依赖倒置)。
- 接口按层组织,低层 contracts 不引用高层 DTO,保证单向。
- 改跨模块接口 = 改 contracts + 双方适配,一次提交,保持文档契约同步。

### 3.3 各模块目录职责一句话

| 目录(模块) | 层 | 职责 | 不该放 |
| --- | --- | --- | --- |

| platform/* 其他 | 跨模块基础设施(DB/auth/k8s/...) | 任何业务逻辑 |

## 4. frontend/ — 四端

```
frontend/
├── apps/
│   ├── student/        学生端(默认落点:课程)
│   ├── teacher/        教师端(默认落点:课程)
│   ├── school-admin/   学校管理员端(默认落点:用户管理)
│   └── platform-admin/ 平台管理员端(默认落点:学校管理)
├── packages/           四端共享
│   ├── ui/             组件库
│   ├── api-client/     对接后端 API(按模块组织)
│   ├── ws-client/      WebSocket 客户端(订阅 M10 topic)
│   ├── sim-sdk/        M4 仿真渲染 SDK + 通用交互渲染器 + 前端确定性运行时
│   ├── ide/            Monaco + xterm 终端封装
│   └── auth/           登录/Token 刷新/角色路由
└── package.json
```

> 四端无独立工作台落地页;登录直达角色第一个功能页(见总纲约定)。

> **可视化(M4)的归属**:M4 仿真可视化引擎是"前后端协同"的——**渲染、SDK、通用交互、前端确定性状态机运行时全部归前端**(`frontend/packages/sim-sdk`),这是 M4 的主体(绝大多数仿真前端运行);后端 `modules/sim` 只负责仿真包的管理/审核/版本,以及少数 `compute=backend` 重计算仿真的执行。因此**可视化不单独成顶层目录**:前端有 sim-sdk 包,后端有 sim 模块,各归其位。

## 5. images/ — 全量镜像治理

详见《镜像与容器设计》。结构:
```
images/
├── service/       平台服务镜像(backend/frontend/migrate)
├── runtime/       链运行时镜像(evm-hardhat/fabric/...)
├── infra/         区块链配套设施镜像(oracle/ipfs/thegraph/...)
├── tool/          工具镜像(code-server/remix/blockscout/...)
├── judger/        判题执行器镜像(testcase-evm/static-scan/...)
├── sim/           后端计算仿真镜像(少数 compute=backend 仿真)
├── sidecar/       Sidecar 镜像(collector/log-shipper/...)
├── init/          Init 镜像(sandbox-seed/...)
├── base/          基础镜像
├── middleware/    平台中间件上游固定或薄封装镜像(postgres/redis/minio/nats/couchdb/harbor)
├── observability/ 可观测上游固定或薄封装镜像(prometheus/grafana)
└── ingress/       入口网关上游固定或薄封装镜像(ingress-nginx)
```
镜像层按镜像职责划分,不按后端模块划分。目录名、镜像名前缀、Harbor Project、`manifest.yaml` 的 `category` 必须完全一致,统一取值为:`service`、`runtime`、`infra`、`tool`、`judger`、`sim`、`sidecar`、`init`、`base`、`middleware`、`observability`、`ingress`;禁止再做分类映射层或兼容别名。

每个镜像目录至少包含 `manifest.yaml` + `README.md`。`source.type=platform-built`、`thin-wrapper`、`build-base` 必须包含 Dockerfile,生产 digest 统一来自 CI 推送后生成的 `image-digests.lock`;`source.type=upstream-pinned` 不重打包,不得包含 Dockerfile,但必须在 manifest 中声明上游 registry/image/version/license、上游不可变 digest、扫描/签名/准入、离线导入、`deploy/config/chaimir.env` 环境变量键、端口、卷、备份、网络和学生权限。`runtime`、`infra`、`tool`、`judger`、`sim` 通过 manifest seed 入对应业务表;其他分类的 manifest 用于构建或上游固定、签名、扫描、离线分发、准入策略和部署审计。

端口、安全和生产门禁以《镜像与容器设计》为准:容器内部端口优先沿用官方默认端口,生产禁止固定宿主机端口/`hostPort`/`hostNetwork`,本地开发宿主机访问只能用可配置且默认绑定 `127.0.0.1` 的映射端口;学生可进入容器必须与判题、答案、密钥、控制面和其他用户数据隔离。

## 6. deploy/ — 部署

详见《部署架构设计》。结构:
```
deploy/
├── base/          Kustomize base(通用 manifests)
├── overlays/      环境覆盖:local-dev / staging / prod-saas / prod-school(私有化)
├── components/    Kustomize components(本地/私有化中间件等可复用片段)
├── config/        ConfigMap/Secret 模板与环境变量单一来源
└── ci/            CI/CD pipeline 定义
```

> **Helm/charts 约定**:Chaimir 平台自身的 Deployment/Service/RBAC/NetworkPolicy/Ingress/CronJob
> 统一维护在 Kustomize `base/` + `overlays/`,不再另建平台 Helm chart,避免 Kustomize 与 Helm
> 双轨维护同一批清单。若确需安装第三方 K8s 组件(如 ingress-nginx、Harbor、External Secrets、
> Sigstore policy-controller、Prometheus Adapter),优先引用官方 Helm chart;仓库内只在需要固定
> values/安装说明时创建 `deploy/charts/` 存放第三方 values,不得放平台自身模板。未引入第三方
> values 前,`deploy/charts/` 可以不存在。
