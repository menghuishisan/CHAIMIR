# 镜像与容器设计

> 平台是容器密集型(K8s)。本文定义全部镜像清单、构建规范、仓库与分发。镜像是 M2 沙箱、M3 判题、M8 对局落地的基础。
> 最后更新:2026-05-29

---

## 一、镜像分类总览

| 类别 | 用途 | 数据驱动 | 大致数量 |
| --- | --- | --- | --- |
| 平台服务镜像 | 后端单体、前端、迁移工具 | 否 | ~3 |
| 链运行时镜像 | M2 各区块链环境(EVM/联盟链/其他公链/自研) | 是(runtime 表) | 首批 ~8,可扩展 |
| 区块链配套设施镜像 | 预言机/去中心化存储/索引/跨链/L2/账户抽象等链下生态 | 是(tool/服务组件) | 首批 ~10,可扩展 |
| 工具镜像 | M2 可接入工具(编辑/浏览/安全分析/通用) | 是(tool 表) | 首批 ~10,可扩展 |
| 判题执行器镜像 | M3 各类判题 | 是(judger 表) | ~5 |
| 仿真算法镜像 | M4 后端计算仿真(少数) | 是(sim_package) | 少量 |
| 中间件镜像 | PG/Redis/MinIO/NATS/Harbor/监控(官方) | 否 | ~7 |
| Sidecar/Init 镜像 | 数据采集/初始化/日志 | 否 | ~3 |
| 基础镜像 | 上述镜像的构建基座 | 否 | ~5 |

> 链运行时/工具/判题器/仿真 均**数据驱动 + 插件化**:新增 = 加镜像 + manifest 入库,平台代码不改(呼应插件化架构)。中间件用官方镜像不重打包。
> **避免冗余**:同生态多变体按"教学目的"区分(开发体验/真实节点/轻量),不重复打包相同能力;链工具公共层(`base/chain-tools`)被多运行时复用。

### 三之二、多容器编排(关键:一个沙箱 ≠ 一个镜像)

> **重要澄清**:一个沙箱实例往往是**一组协同容器(Pod 组)**,不是单镜像。M2 架构定义"每沙箱 = Namespace + Pod(s)"。运行时 manifest 声明该沙箱由哪些容器组成,控制面据此编排出多容器 Pod 组。

典型多容器编排:
| 实验场景 | 沙箱内的容器组 |
| --- | --- |
| Fabric 联盟链 | peer + orderer + CA + (可选)CouchDB 状态库 + (可选)fabric-explorer |
| EVM 攻防实验 | 链节点(geth/anvil) + 工具容器(Remix/blockscout) + (可选)预言机 mock |
| 多人联盟链组网 | 每个角色一个 peer 容器,组成一条链(多人共享,见 M7) |
| 混合实验 | 链节点容器 + collector sidecar(供 M4 取数) |

- **状态库**:Fabric 若用 CouchDB 富查询,CouchDB 作为该 Pod 的一个容器(或复用官方 couchdb 镜像),由 runtime manifest 声明;用内嵌 LevelDB 则无需。
- **工具容器**:web-embed 类工具(code-server/remix/blockscout)作为沙箱 Pod 内的 sidecar 容器,与链节点同 Pod 便于互通。
- 镜像层仍是"每种容器一个镜像",**编排(几个容器、怎么连)由 runtime manifest + M2 控制面负责**,镜像本身不耦合编排。

### 三之三、forked 运行时与特殊能力

- **forked 漏洞题**:由具 fork 能力的 EVM 运行时承担(`runtime/evm-foundry` 的 anvil fork 或 `runtime/evm-hardhat` fork 模式),无需独立镜像。出题阶段从真实链历史区块派生(经受控出网代理白名单,见 M2 安全),拉取所需状态后**固化进题目**;答题期锁定区块号 + 缓存状态,零外部依赖。
- **预言机/外部依赖合约**(如 DeFi 闪电贷题):作为题目的**固化资产**(合约源码+bytecode+初始化交易,导入即固化)在 isolated 沙箱内部署,**无需独立镜像**;仅当需要实时喂价 mock 服务时再评估增配容器。

## 八、构建规范

1. **薄 Dockerfile + manifest.yaml**:官方工具镜像薄封装;自研多阶段构建。
2. **non-root**:所有镜像以非 root 用户运行;只读根文件系统(可写层限挂载卷)。
3. **多阶段精简**:构建与运行分离,最小化体积。
4. **可复现构建**:锁定基础镜像 digest 与依赖版本。
5. **manifest 驱动**:`manifest.yaml` 描述镜像元信息,`seed-images` 脚本写入对应业务表(runtime/tool/judger),实现数据驱动接入。

## 九、镜像仓库与安全

- **Harbor 私有仓库**,按类别分 Project(service/runtime/tool/judger/sim/base)。
- **安全门禁**:Trivy 漏洞扫描(高危阻断) + Cosign 签名(未签名不得部署)。
- **预拉取**:官方/审核通过镜像经 M2 创建的 DaemonSet 在目标沙箱节点全节点预拉取(M2 秒进优化第一关);只有 DaemonSet 全目标节点 Ready 且无拉取失败事件时,才允许标记 `runtime_image.prepulled=true`。
- **自定义镜像审核**:教师/第三方上传镜像须经平台管理员审核 + 扫描后入预拉取池。

## 十、私有化分发

- 私有化部署无法访问公网 Harbor:用**离线镜像包**(`docker save/load` 或 Harbor Replication)导入校内 Harbor。
- 镜像包按需裁剪(只含该校启用的链/工具),减小体积。
- 链运行时镜像随平台版本一起打包交付。

## 十一、与文档/数据的关系

- 镜像元信息(地址/版本/端口/资源)存对应业务表(M2 runtime_image/tool、M3 judger、M4 sim_package),**运行时数据驱动加载**,不硬编码。
- 本文是镜像清单与构建规范 SSOT;具体某镜像的运行行为(怎么起节点/判题)见对应模块的适配器/判题器设计。
