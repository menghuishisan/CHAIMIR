# 镜像与容器设计

> 平台是容器密集型(K8s)。本文定义全量镜像治理清单、构建规范、上游固定、仓库与分发。镜像是 M2 沙箱、M3 判题、M8 对局落地的基础。
> 最后更新:2026-06-20

---

## 一、镜像分类总览

| 类别 | 用途 | 数据驱动 | 大致数量 |
| --- | --- | --- | --- |
| `service` 平台服务镜像 | 后端单体、前端、迁移工具、受控定时任务 | 否 | ~4 |
| `runtime` 链运行时镜像 | M2 各区块链环境(EVM/联盟链/其他公链/自研) | 是(runtime 表) | 首批 ~8,可扩展 |
| `infra` 区块链配套设施镜像 | 预言机/去中心化存储/索引/跨链/L2/账户抽象等链下生态 | 是(tool/服务组件) | 首批 ~10,可扩展 |
| `tool` 工具镜像 | M2 可接入工具(编辑/浏览/安全分析/通用) | 是(tool 表) | 首批 ~10,可扩展 |
| `judger` 判题执行器镜像 | M3 需容器化运行的判题套件/扫描器 | 是(judger 表) | ~5 |
| `sim` 仿真算法镜像 | M4 后端计算仿真(少数) | 是(sim_package) | 少量 |
| `sidecar` Sidecar 镜像 | 数据采集/日志 | 否 | ~2 |
| `init` Init 镜像 | 沙箱初始化/种子注入 | 否 | ~1 |
| `base` 基础镜像 | 上述镜像的构建基座 | 否 | ~5 |
| `middleware` 平台中间件镜像 | PostgreSQL、Redis、MinIO、NATS、ClamAV、CouchDB、Harbor | 否 | ~7 |
| `observability` 可观测镜像 | Prometheus、Grafana | 否 | ~2 |
| `ingress` 入口网关镜像 | ingress-nginx | 否 | ~1 |

> 镜像层按镜像职责划分,不按后端模块划分;M2/M3/M4/M8 是消费者,不是目录边界。`images/` 本地目录名、镜像名前缀、Harbor Project、`manifest.yaml` 的 `category` 必须完全一致,统一取值为 `service`、`runtime`、`infra`、`tool`、`judger`、`sim`、`sidecar`、`init`、`base`、`middleware`、`observability`、`ingress`;禁止分类映射层、兼容别名和多套命名。
> `images/` 是**全量镜像治理目录**,不是只放平台自建 Dockerfile 的目录。平台自建镜像、官方薄封装镜像、纯上游固定镜像都必须在此登记 manifest、README、来源、版本、不可变 digest 锁定、许可、安全、端口、卷、备份、离线导入和学生权限。纯上游固定镜像的上游 digest 写入 manifest;平台自建、官方薄封装和构建基座镜像的生产 digest 统一来自 CI 推送后生成的 `image-digests.lock`,不得在 manifest 中维护第二套构建产物 digest 来源。配置和密钥变量名必须引用 `deploy/config/chaimir.env`;真实密钥值只由 Secret/KMS 在运行期注入同名环境变量,不得写入仓库、manifest、Dockerfile 或镜像层。
> 链运行时/配套设施/工具/判题器/仿真 均**数据驱动 + 插件化**:新增 = 加镜像 + manifest 入库,平台代码不改(呼应插件化架构)。M3 中由后端策略直接、安全承接的轻量判题能力不再重复做镜像,避免同一判题逻辑在后端和容器内双轨实现。
> **避免冗余**:同生态多变体按"教学目的"区分(开发体验/真实节点/轻量),不重复打包相同能力;通用 CLI 由 `base/chain-tools` 复用,从固定源码构建的 Fabric 专用二进制由 `base/fabric-tools` 复用,二者不得混成同时膨胀全部消费者的通用大镜像。
> **沙箱身份契约**:`runtime`、`tool`、`judger` 以及由 M2 编排进沙箱 Pod 的 `infra` 镜像必须固定以 `1000:1000` 运行。薄封装镜像应复用上游已有 UID/GID 1000,不存在时在构建期创建;只允许对 `/workspace`、`/runtime-state`、工具声明的临时缓存目录、判题工作目录等必要路径做最小 `chown 1000:1000`,禁止 `chmod 777`、禁止 `chown -R /`、禁止保留 root group。M2 编排层保持 `runAsUser=1000`、`runAsGroup=1000`、`fsGroup=1000`,不得按镜像做 UID 兼容分支。

### 一之二、镜像单元与运行期工作负载

镜像层和运行期编排必须分成两层,不得混用:

- **镜像单元(Image Unit)**:`images/<category>/<name>/manifest.yaml` 只描述一个可拉取镜像自身的来源、digest、许可证、端口、安全、资源建议、自检和离线分发。`upstream.components` 只表示构建来源或上游组件清单,例如判题镜像同时复用 Foundry 与 Node 工具链;它不是 Kubernetes 多容器编排定义。
- **运行期工作负载(WorkloadSpec)**:描述一次沙箱、判题或后端计算运行需要启动哪些组件、服务、路由、卷域和 NetworkPolicy。`runtime.adapter_spec`、`tool.resource_spec`、`judger.resource_spec` 复用同一套字段语义,但由各自模块按边界消费;M4 标准 `stdio-json` 能力的工作负载只由部署侧 `SIM_BACKEND_STDIO_ADAPTERS_JSON` 目录生成,不接受仿真包提交工作负载字段。
- **单组件也是 WorkloadSpec**:code-server、Remix、Jupyter、CLI 判题器等单进程场景也按单组件工作负载表达,不得保留一套单镜像字段和一套多组件字段并行。
- **组合不进镜像层**:Blockscout、Fabric Explorer、The Graph、Chainlink、IPFS Cluster、noVNC 等需要多进程或配套服务时,由运行期工作负载声明组件和路由;不得把 frontend/backend/proxy/db 强行塞进一个大镜像,也不得为某个工具写后端特判。
- **资源与网络按组件汇总**:配额校验、ResourceQuota、NetworkPolicy、Service、DNS、路由前缀和学生入口必须按运行期组件汇总生成。组合超出命名空间或租户配额时必须拒绝创建,不得静默降配或改端口。
- **边界不变**:M2 负责沙箱类 WorkloadSpec 编排;M3 通过 `internal/contracts.SandboxService` 申请 judge 沙箱或复用已有沙箱,不直接编排 K8s;M4 `compute=backend` 通过自有 `BackendAdapter` 或受控仿真运行器执行,不得直接调用 M2。

### 三之二、多容器编排(关键:一个沙箱 ≠ 一个镜像)

> **重要澄清**:一个沙箱实例往往是**一组协同容器(Pod 组)**,不是单镜像。M2 架构定义"每沙箱 = Namespace + Pod(s)"。运行时 manifest 声明该沙箱由哪些容器组成,控制面据此编排出多容器 Pod 组。

典型多容器编排:
| 实验场景 | 沙箱内的容器组 |
| --- | --- |
| Fabric 联盟链 | peer + orderer + CA + (可选)CouchDB 状态库 + (可选)fabric-explorer |
| EVM 攻防实验 | 链节点(geth/anvil) + 工具容器(Remix/blockscout) + 题目固化资产/教师脚本部署的依赖合约 |
| 多人联盟链组网 | 每个角色一个 peer 容器,组成一条链(多人共享,见 M7) |
| 混合实验 | 链节点容器 + collector sidecar(供 M4 取数) |

- **状态库**:Fabric 若用 CouchDB 富查询,CouchDB 作为该 Pod 的一个容器(或复用官方 couchdb 镜像),由 runtime manifest 声明;用内嵌 LevelDB 则无需。
- **工具容器**:web-embed 类工具(code-server/remix/blockscout)作为沙箱 Pod 内的 sidecar 容器,与链节点同 Pod 便于互通。
- **运行时配置工具**:Fabric Explorer 等依赖链网络 profile、证书或配套数据库的工具,manifest 必须显式声明 `tool.runtime_config_required=true`;在运行时/实验 WorkloadSpec 没有真实生成这些配置前,seed 只能写入停用状态,不得因镜像已构建就标记可调度。
- 镜像层仍是"每种容器一个镜像",**编排(几个容器、怎么连)由 runtime manifest + M2 控制面负责**,镜像本身不耦合编排。
- **预拉取口径**:预拉取使用运行时/工具 WorkloadSpec 中的 `prepull_command` 与 `prepull_hold`,命令必须来自镜像 `manifest.yaml` 的 `selftest.commands` 或同等受控声明。运行期 `command` 只表达容器真实启动方式,不得兼作镜像自检;多镜像运行时必须用一个保持容器加若干 initContainer 覆盖所有 runtime/infra/tool 镜像。
- **禁止固定组合矩阵**:`images/` 层不得维护 `composition.yaml`、固定 bundle、"某镜像只能搭配某镜像"等 N×M 关系。单镜像 `manifest.yaml` 只声明本镜像能力、生态标签、端口、安全域、资源和自检;具体容器组由 `runtime.adapter_spec`、实验/题目配置和 M2 编排器按数据动态生成。
- **组合校验位置**:M2 在 seed 与创建沙箱前按 runtime manifest 校验工具/运行时生态标签、同组 Pod 端口唯一、Service/DNS/路由名唯一、NetworkPolicy 放行范围、共享卷安全域和学生入口权限;校验失败必须拒绝创建,不得在镜像层用固定白名单替代。

### 三之二之一、端口与多人组网约束

镜像层只定义容器内部进程监听端口、协议、健康检查和用途,不定义宿主机端口。端口暴露、Service、Ingress、临时访问 URL、多人组网拓扑由 M2 沙箱控制面和部署层统一编排。

端口分层口径:
- **容器内部端口**:优先沿用官方默认端口,例如 PostgreSQL `5432`、Redis `6379`、geth/anvil RPC `8545`、Fabric peer `7051`。这样可以复用官方启动参数、SDK、健康检查和教学资料,不要为避免宿主机冲突而修改镜像内端口。
- **集群内部访问**:Pod/Namespace/Service 之间可以重复使用同一个容器端口,不会互相冲突;服务发现靠 Service/DNS 名称,不是靠宿主机端口区分。
- **本地宿主机访问**:只有开发者需要从宿主机直连数据库、Redis、MinIO、链 RPC 等服务时,才允许在本地 compose/K8s 覆盖层做可配置端口映射,默认绑定 `127.0.0.1`,例如 `127.0.0.1:${CHAIMIR_DEV_POSTGRES_PORT:-15432}:5432`。
- **生产外部入口**:生产不暴露数据库、中间件和沙箱内部服务的宿主机端口;用户访问统一走平台网关、Ingress 或受控代理,并带鉴权、审计和租户/环境路由。

强制规则:
1. **禁止固定宿主机端口**:镜像、Dockerfile、manifest、启动脚本均不得写 `hostPort`、固定 `NodePort`、`hostNetwork: true` 或依赖宿主机端口范围;本地开发也只能通过 compose/K8s 覆盖层临时映射,不得进入镜像规范。
2. **容器端口只作契约**:`manifest.yaml` 只能声明 `containerPort`、协议、端口用途、是否需要平台代理暴露和健康检查路径;真实对外入口由控制面生成 Service/Ingress/Gateway 路由。
3. **同组 Pod 端口唯一**:同一 Pod 网络命名空间内,协同容器不得监听相同的 IP/端口组合;多人联盟链、工具 sidecar、collector 等组合由 runtime manifest 自检验证端口无冲突后才允许 seed。
4. **多人组网不共享宿主机端口**:多人实验按 `tenant_id`、`session_id`、`env_id` 生成 Namespace、Service、DNS 和 NetworkPolicy;角色节点通过集群内 DNS 互联,外部访问经平台网关带鉴权路由进入,不为每个用户分配宿主机端口。
5. **端口来自数据和配置**:业务服务端口、内部 RPC 端口、工具 Web 端口、健康检查端口均来自 `manifest.yaml` 或环境变量,禁止硬编码在后端代码、镜像入口脚本和部署模板中。
6. **冲突前置校验**:`seed-images` 与 M2 编排前必须校验 manifest 端口、Service 名、DNS 名、路由前缀和资源名唯一性;冲突直接拒绝接入,不得运行时静默改端口或做兼容映射。

因此,多人组网场景的端口冲突风险不通过"约定大家别占用同一宿主机端口"解决,而是通过不使用宿主机端口、按环境隔离命名空间和由控制面生成网络对象来消除。

### 三之三、forked 运行时与特殊能力

- **forked 漏洞题**:由具 fork 能力的 EVM 运行时承担(`runtime/evm-foundry` 的 anvil fork 或 `runtime/evm-hardhat` fork 模式),无需独立镜像。出题阶段从真实链历史区块派生(经受控出网代理白名单,见 M2 安全),拉取所需状态后**固化进题目**;答题期锁定区块号 + 缓存状态,零外部依赖。
- **预言机/外部依赖合约**(如 DeFi 闪电贷题):作为题目的**固化资产**(合约源码+bytecode+初始化交易,导入即固化),或由教师上传初始化脚本在 isolated 沙箱内确定性部署,**无需独立 mock 服务镜像**。如果课程需要真实链下预言机能力,使用 `infra/chainlink-node`;不得把自研 HTTP mock 作为平台必需镜像。

## 六、仿真算法镜像(M4,少数)

- 绝大多数仿真前端运行,无镜像。
- 仅 `compute=backend` 的重计算仿真才有镜像:`sim/<package-code>`,经审核 + 沙箱化验证后入 `sim_package`。
- `sim/<package-code>` 是单个后端计算包的执行镜像,不是"覆盖所有仿真"的总计算包。仿真覆盖能力来自 M4 自描述仿真包协议、7 类平台可视化模式、审核与运行时;每新增一个后端重计算仿真,都必须新增 `sim_package` 版本、算法镜像与服务端受控能力目录项。
- M4 后端计算适配器按**执行协议**复用,不得按算法复制 Kubernetes 生命周期代码。使用相同 `stdio-json` 协议的算法统一由共享 K8s 适配器执行;新增同协议算法只增加镜像、manifest、digest 证明和 `SIM_BACKEND_STDIO_ADAPTERS_JSON` 目录项,平台 Go 代码不改。只有出现长期驻留 gRPC、GPU 作业等不同执行协议时,才允许新增另一种适配器实现。
- 当前 `sim/graph-layout` 以能力编号 `graph-layout-stdio` 登记在受控目录中,由共享 `stdio-json` K8s 适配器消费:每个 WS 会话独立命名空间/Pod,每次事件通过受控 Kubernetes exec 把 JSON 写入镜像标准输入并读取标准输出。镜像只负责算法和算法输入校验,M4 负责鉴权、交互白名单、事件持久化、资源隔离、通用 I/O 上限、状态封装和回收。

## 八、构建规范

1. **来源类型决定 Dockerfile**:`source.type=platform-built`、`thin-wrapper`、`build-base` 必须有 Dockerfile,生产拉取 digest 只来自 CI 产出的 `image-digests.lock`;`source.type=upstream-pinned` 不重打包,不得有 Dockerfile,但必须有 manifest、README、上游不可变 digest、扫描、签名/准入和离线导入记录。
2. **non-root**:所有镜像以非 root 用户运行;M2 沙箱消费的 runtime/tool/judger/infra 镜像统一 `USER 1000:1000`,只读根文件系统(可写层限挂载卷)。
3. **多阶段精简**:构建与运行分离,最小化体积。
4. **可复现构建**:锁定基础镜像 digest 与依赖版本。Dockerfile 的镜像构建参数必须使用显式失败默认值,由 `images/build-images.ps1` 或共享 CI action 注入 Harbor digest;内部构建基座按依赖拓扑先构建,不得依赖目录字母顺序。
5. **统一分类**:`manifest.yaml.category` 必须等于本地目录名、镜像名前缀和 Harbor Project,只能取 `service`、`runtime`、`infra`、`tool`、`judger`、`sim`、`sidecar`、`init`、`base`、`middleware`、`observability`、`ingress`。
6. **manifest 驱动**:`manifest.yaml` 描述镜像元信息;`runtime`、`infra`、`tool`、`judger`、`sim` 由 `seed-images` 脚本写入对应业务表,实现数据驱动接入;`service`、`sidecar`、`init`、`base`、`middleware`、`observability`、`ingress` 用于构建/上游固定、签名、扫描、离线分发、准入策略和部署审计。
7. **统一供应链路径**:本地验收、预发布和生产都必须走同一条链路:构建镜像直接推送到 Harbor 分类 Project → 从 Harbor 解析不可变 digest 生成候选锁 → 按 digest 拉回 → Trivy/Cosign 门禁 → 通过后才生成正式 `image-digests.lock` 或准入证明。唯一职责入口为 `images/build-images.ps1`(构建与候选锁)、`images/pull-images.ps1`(按 digest 拉取)、`deploy/scripts/image-attestations-generate.ps1`/`deploy/ci/build-scan-sign-push`(扫描、SBOM、签名与验证)和 `images/sync-image-metadata.ps1`(晋升);不得再增加并行的本地刷新或发布编排。禁止用本地 Docker tag、`local` 镜像名、未推送镜像或手工 digest 作为验收依据。
8. **候选锁不等于准入**:`images/build-images.ps1 -Push -DigestLockOut <candidate.lock>` 只能生成候选 digest 锁,用于回拉和安全校验。候选锁中任一镜像存在 HIGH/CRITICAL、SBOM 生成失败、签名或证明验证失败时,不得复制为正式 `images/image-digests.lock`,不得写入 `SANDBOX_IMAGE_ATTESTATIONS_JSON`,不得 seed 为可调度。
9. **代理不进入镜像来源**:公网慢或受限时,只能在 Docker Desktop、BuildKit、CI runner 或离线同步任务中配置代理/缓存/registry mirror;`Dockerfile`、`manifest.yaml`、`image-digests.lock` 和离线包清单必须仍记录官方 registry、官方上游 digest 或 Harbor 产物 digest。禁止把第三方镜像站、临时加速域名或个人代理地址写成镜像来源,避免本地和生产供应链分叉。
10. **产物元数据自动晋升**:后端流水线只产出 `service/backend`、`service/migrate`、`service/cron` digest,前端流水线只产出 `service/frontend` digest,通用镜像流水线不得重复构建这四项。每项 digest 只能在 Trivy 阻断扫描、SBOM 生成、Harbor 推送、Cosign 签名/证明和验证全部成功后写入统一片段。`image-metadata-promotion` 工作流串行合并片段,调用 `images/sync-image-metadata.ps1` 更新 `images/image-digests.lock`、local-dev Kustomize digest、受控仿真引用和本地/私有化静态准入列表;同步器必须清除 catalog 已删除、已阻断或不再位于正式锁中的旧项,并能把首次晋升的旧 tag 条目原位转换为 digest,不得要求人工预填 digest。`backend/.env.example` 的准入列表保持空值,生产证明由 Secret/KMS 注入,不得把本地证明冒充生产配置。随后由机器人 PR 自动合并,禁止流水线直接写 `main`。机器人必须使用能触发 PR 检查的专用 `IMAGE_METADATA_BOT_TOKEN`,不得用默认 `GITHUB_TOKEN` 绕过或卡住检查链。
11. **禁止元数据回环构建**:`images/image-digests.lock` 与自动同步产生的部署引用只记录已完成的构建结果,同一逻辑镜像自己的 digest 晋升不得触发自身重建。共享基座 digest 变化是消费者的真实构建输入变化:backend 仅监听 `base/go-builder`、`base/runtime-min`、`middleware/postgres`,frontend 仅监听 `base/node-builder`,并各触发一次服务重建;随后服务 digest 晋升不得再次触发。镜像源码、Dockerfile、manifest、共享构建 action、应用源码或上述明确基座变化才触发对应产物流水线。晋升工作流缺少片段、签名/证明验证失败、引用无法映射或 PR 无法自动合并时必须显式失败,不得继续使用旧 digest。
12. **部署只认不可变引用**:local-dev 由自动晋升后的 Kustomize digest 驱动;staging 只在权威锁合入 `main` 后部署;生产发布从 tag 所指提交读取同一权威锁。两者都必须一次解析 `service/backend`、`service/frontend`、`service/migrate`、`service/cron` digest,任一缺失就失败。环境 overlay 中的 tag 仅为可渲染模板默认值,实际部署必须经共享脚本生成临时 Kustomize 覆盖并替换为 digest,不得将模板 tag 当成已发布产物。

## 八之二、运行安全与生产门禁

镜像层按生产交付标准设计,不得出现只为演示可用的弱隔离、弱校验或临时降级路径。所有镜像在进入 Harbor 和 seed 前必须通过以下门禁:

1. **最小权限运行**:默认 `runAsNonRoot`、`allowPrivilegeEscalation=false`、只读根文件系统、显式 `drop` Linux capabilities;禁止 `privileged`、宿主机 PID/IPC/Network、宿主机路径挂载。确需例外的链节点能力必须写入 manifest 的安全例外字段,经安全评审后才能入库。
2. **网络默认拒绝**:沙箱 Namespace 默认 deny-all;只放行同一环境内被 manifest 声明的容器互通、平台控制面必要回调、对象存储/日志/指标等受控出口。fork 取链状态、下载依赖等出网行为只能走白名单代理,答题/判题期不得依赖公网。
3. **密钥不进镜像**:私钥、JWT、数据库密码、对象存储密钥、RPC 凭据、教师答案、flag、判题配置不得写入 Dockerfile、镜像层、构建参数和仓库;运行期只允许通过 K8s Secret、短期凭证或题目固化资产注入,并按权限最小化挂载。
4. **资源硬限制**:manifest 必须声明 CPU/内存/临时存储/PID/超时/并发限制和健康检查;判题与不可信代码镜像必须禁用默认出网并设置用后即毁策略。
5. **供应链可追溯**:平台构建镜像锁基础镜像 digest 并锁定依赖版本;纯上游固定镜像锁上游 digest 和许可证来源。所有可部署镜像都必须生成 CycloneDX SBOM,通过统一 Trivy 配置扫描并以 HIGH/CRITICAL 阻断,再由 Cosign 对镜像和 SBOM 证明分别签名、验证后才能部署。当前使用项目私钥且未部署私有 Rekor,本地、私有化和 CI 均关闭公共透明日志上传,不得向第三方端点发送私有仓镜像元数据;部署侧只按 digest 拉取。
   - 若上游最新正式版本仍触发高危门禁,镜像 manifest 必须在 `supply_chain.deployable=false` 写明 `block_reason`;供应链脚本不得把该镜像写入 `SANDBOX_IMAGE_ATTESTATIONS_JSON`,seed/调度也不得把它标记为可用。恢复只能来自官方安全版本、可验证源码重建或已完成契约适配的替代组件,不得跳过扫描、伪造证明或用本地例外绕过。
6. **启动显式失败**:入口脚本必须 `set -e` 等价处理错误,关键依赖、端口、权限、健康检查失败必须退出并暴露结构化日志;禁止吞错、自动降级到弱安全模式或写临时兼容分支。
7. **日志与数据脱敏**:日志不得输出密钥、token、私钥、flag、答案、学生代码全文和连接串;调试端口、管理后台、指标端点默认只在环境内可见,外部访问必须经平台鉴权代理。
8. **自检可复现**:每个镜像目录必须提供 selftest 或等价健康验证,覆盖启动、端口、权限、网络策略、资源限制和关键业务命令;未通过自检不得 seed 入库或标记可调度。

## 八之三、学生可进入容器权限边界

学生能通过 Web 终端、IDE、调试工具或浏览器间接进入的容器,必须按"不可信用户环境"处理。学生容器只能看到完成实验所需的工作区、公开题面、公开链状态和经平台授权的工具,不能看到平台控制面、判题答案、其他用户或宿主机信息。

强制规则:
1. **学生入口与敏感容器分离**:学生终端默认进入 `tool` 或受控 `runtime` 容器;不得进入 `service`、`judger`、`init`、控制面 sidecar、数据库、中间件或含答案/flag/判题配置的容器。确需链节点交互时,只暴露受控 CLI/RPC,不开放高权限 shell。
2. **最小文件可见性**:学生容器只挂载本人/本队工作区和公开只读素材;不得挂载宿主机路径、Kubeconfig、ServiceAccount token、K8s Secret、平台配置目录、教师答案、判题脚本、隐藏测试用例、其他学生目录或控制面日志。
3. **关闭默认 ServiceAccount token**:学生可进入的 Pod 必须设置 `automountServiceAccountToken=false`;如确需访问平台 API,只能使用平台签发的短期、最小权限、按租户/环境绑定的令牌,并由网关注入/回收。
4. **共享卷分级**:同一环境内的共享卷分为 `workspace`、`public-assets`、`runtime-state`、`judge-private` 等安全域;学生容器只能读写被授权域。`workspace` 才是学生文件接口和代码持久化根目录;`runtime-state` 默认不对学生文件接口可见,只在显式快照策略下由 M2 受控保存;`judge-private`、flag、隐藏断言、评分脚本、教师私钥和答案永不挂载到学生容器。
5. **进程与系统信息收敛**:学生容器禁止 `privileged`、宿主机 PID/IPC/Network、Docker socket、container runtime socket、主机 `/proc`/`/sys` 敏感挂载和内核能力扩展;容器内只能看到本容器或本环境允许的进程与网络目标。
6. **网络只到必要目标**:学生容器默认不能访问数据库、Redis、MinIO 管理端、NATS、Harbor、K8s API、元数据服务、控制面内部接口和其他环境 Namespace;只放行题目声明的链节点、工具、公开代理和平台授权 API。
7. **调试端点受控**:IDE、区块链浏览器、RPC、metrics、pprof、admin API 等端点不得裸露给学生;需要开放时必须经平台代理鉴权、按 `tenant_id/session_id/env_id` 路由,并记录审计。
8. **环境销毁清理**:学生环境结束后必须销毁 Pod、临时卷、短期凭证和路由;持久化产物只能通过平台文件/作业/报告接口按租户权限保存,不得遗留在节点本地目录。

## 九、镜像仓库与安全

- **Harbor 私有仓库**,按统一分类分 Project:`service`、`runtime`、`infra`、`tool`、`judger`、`sim`、`sidecar`、`init`、`base`、`middleware`、`observability`、`ingress`。
- **安全门禁**:Trivy 漏洞扫描(高危阻断) + Cosign 签名(未签名不得部署)。
- **无本地分叉**:所有镜像引用必须使用 Harbor 分类命名空间和不可变 digest。本机 Docker 只允许作为构建缓存和调试执行环境;本地 tag 不能作为后端完整测试、K8s 预拉取、seed 或准入证明的来源。
- **预拉取**:官方/审核通过镜像经 M2 创建的 DaemonSet 在目标沙箱节点全节点预拉取(M2 秒进优化第一关)。预拉取粒度是运行时默认工作负载镜像集合,包含 runtime 主镜像、声明式 infra 镜像和按生态匹配的工具镜像;只有 DaemonSet 全目标节点 Ready 且无拉取失败事件时,才允许标记 `runtime_image.prepulled=true`。
- **自定义镜像审核**:教师/第三方上传镜像须经平台管理员审核 + 扫描后入预拉取池。

### 九之一、项目级 Docker 清理边界

- 可自动判定为 Chaimir 的 Docker 对象仅包括:`com.docker.compose.project=chaimir-supply-chain` 标记的供应链容器、`images/` catalog/正式锁/当前候选锁能映射的镜像引用,以及脚本明确创建的 Chaimir 临时标签。不得按创建时间、镜像体积或笼统名称猜测归属,不得执行全局 `docker system/image/volume prune`。
- 常规清理只删除已停止的供应链临时容器、已失去任何 Chaimir catalog/正式锁/当前候选锁引用且未被容器使用的临时标签和悬空层。运行中的容器、仍被任一不可变引用使用的层不得删除。Docker 删除最后一个 tag 时可能同时移除本地 RepoDigest;清理前必须逐项证明正式锁的全部 digest 在删除后仍可由 `docker image inspect image@digest` 访问,不能仅凭“同一 image ID”推断引用会保留。
- Harbor Kubernetes 工作负载及 PVC、`chaimir-supply-chain-trivy-cache`、Docker registry 认证、Cosign 密钥和尚未完成晋升的证据均是可复用状态,不属于“冗余”。只有用户明确要求重置供应链数据时才能单独删除,且仍不得影响其他项目。

## 十、私有化分发

- 私有化部署无法访问公网 Harbor:用**离线镜像包**(`docker save/load` 或 Harbor Replication)导入校内 Harbor。
- 镜像包按需裁剪(只含该校启用的链/工具/判题器/仿真包和必需中间件/可观测/入口组件),减小体积。
- 链运行时镜像、判题镜像、平台服务镜像和必需基础设施镜像随平台版本一起打包交付。
- 纯上游固定镜像作为 `upstream-pinned` 纳入离线包,按 `middleware`、`observability`、`ingress` 等统一分类导入对应 Harbor Project;不得因采用官方镜像而绕过 digest 锁、漏洞扫描、签名/准入、备份和网络策略。
- 离线包制作和校内 Harbor 预热统一使用 `images/pull-images.ps1`。脚本按 manifest 顺序逐个拉取 `image@digest`,默认读取已经由机器人 PR 晋升的 `images/image-digests.lock` 获取平台自建、薄封装和构建基座镜像 digest;缺少 digest 直接拒绝。单个镜像拉取失败后必须清理失败引用并重试,全部重试失败返回非 0,不得静默跳过或回退到可变 tag。本地完整功能测试也必须使用 Harbor + digest 锁:先用 `images/build-images.ps1 -Push` 将构建产物直接推入同一 Harbor 分类命名空间并生成候选锁,再用 `images/pull-images.ps1 -DigestLock <candidate.lock>` 按 digest 拉回并扫描。拉取脚本只消费不可变 digest,不得发布本地可变 tag,不得替代 Trivy/Cosign 门禁和正式 `image-digests.lock`。
