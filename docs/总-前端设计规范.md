# 总-前端设计规范

> Chaimir 四端前端的**唯一设计真相源**:从设计令牌、组件库、信息架构,到沉浸式工作台、数据可视化、响应式自适应、无障碍基线与工程落地的完整规范。
> 本文与 `CLAUDE.md`/`AGENTS.md` §10、`docs/总-工程目录设计.md`(`frontend/` 结构)互为一体:**§10 是铁律摘要,本文是展开;工程目录定义代码落点。**
> 改动设计先改本文,再写代码 —— 保持一致(CLAUDE.md 总则)。
>
> 注意:**高保真原型已清理,后续重建。** 重建时必须以本文、已实现功能清单、后端角色/接口语义和真实业务流程为输入,不能照抄旧原型或当前不合理前端外观。**一切以本规范 + 实际功能语义为准**;本文未尽处以工程实践与无障碍/响应式标准补足。

## 0. 文档定位与铁律

### 0.1 这份文档解决什么问题

此前前端设计只散落在 `CLAUDE.md §10` 几行和早期示意中,没有正式规范。真做 React 四端时,四套工作台、上百个页面若各做一套,必然风格漂移、无障碍失守、组件重复。本文把设计沉淀为**可检索、可校验、可落地**的单一标准,让 `frontend/packages/ui` 直接照此实现。后续高保真原型必须基于已实现功能与页面语义重新设计,不继承旧示意的简化和错误。

### 0.2 前端铁律(违反即不可合并)

| # | 铁律 | 依据 |
| --- | --- | --- |
| FE-1 | **全令牌化**:组件零裸 hex,一律引用语义令牌(`var(--color-*)`)。原始色阶→语义令牌单一真相源。 | CLAUDE.md §10 |
| FE-2 | **无障碍达标**:关键文字/控件对比度 ≥ WCAG AA(正文 4.5:1、大字/图形 3:1);焦点可见、键盘可达、尊重 `prefers-reduced-motion`、信息不靠颜色单一传达。 | §3 |
| FE-3 | **禁用 emoji**:界面/文案/代码一律不用 emoji,图标统一 **Lucide**(单一图标族、描边一致)。 | CLAUDE.md §10 |
| FE-4 | **文案面向用户**:可见文字用用户能懂的自然语言,不暴露开发术语(`null`/`timeout`/字段名/`error_code`)。每条文案斟酌。 | CLAUDE.md §8/§10 |
| FE-5 | **无独立工作台落地页**:四端登录后直达该角色侧栏的第一个功能页。 | CLAUDE.md §10、蓝图 §8 |
| FE-6 | **双模态**:日常用侧栏导航(深色框架+白底内容);做实验/仿真/答题进沉浸式全屏。 | §6/§7 |
| FE-7 | **多步骤状态服务端持久化**:向导/长表单中间态以服务端为权威,前端 localStorage 仅即时兜底。 | CLAUDE.md §9、蓝图 §11 |
| FE-8 | **错误分层暴露**:前端只展示友好文案 + `trace_id`,绝不显示堆栈/SQL/内部字段。按 `code` 决定交互,按 `message` 兜底。 | CLAUDE.md §8 |
| FE-9 | **窗口自适应(响应式)必做**:所有页面在常见视口(手机/平板/笔记本/大屏)与窗口任意缩放下均**正常可用、无横向滚动、无遮挡/错位**;布局随窗口流式重排,而非固定像素。沉浸式工作台同样自适应。 | §6.4 |
| FE-10 | **质感开发时打磨,不以旧稿为标准**:组件精度、交互手感、微动效必须在**真实开发中实测、优化、重做**。守住本文**原则**,以真实产品手感为达标判据。 | §4 |

### 0.3 设计气质一句话

**开发者工具的精确感 × 教学产品的亲和力。** 不是消费级炫酷,也不是后台管理系统的呆板;是「专业、克制、可信、有学科代入感」。设计智能(ui-ux-pro-max)对区块链开发者工具的判定印证此方向:Inter 字体、暗色骨架、状态色体系(绿/琥珀/红)、最小辉光、可见焦点;**明确避免** AI 紫粉渐变、装饰性持续动画、慢动画(>500ms)。

## 1. 设计理念与品牌

### 1.1 青电 × 琥珀(冷暖双色)

品牌色经多轮迭代定为**青电 × 琥珀双色**,取代单一色板,解决「不单一又不杂」:

- **主色 = 青电(teal-cyan)**:链上感、数字感,替代千篇一律的科技蓝。用于**交互/CTA/导航激活/焦点/链接/进度**。
- **副色 = 琥珀(amber)**:能量、对抗、高亮。用于**统计强调/通知角标/告警/竞赛对抗/能量态**。
- **骨架 = 墨黑/石板(slate)**:深色框架(顶栏/侧栏底/沉浸面板)与中性层级。
- **冷暖对撞**:青电(冷)与琥珀(暖)形成张力,这是「突破常规」的落点 —— 突破在**配色**,不在字号/布局(布局保持克制)。

> 换主色 = 改一个令牌。所有品牌色点位都引用语义令牌,主色调整只动 `--color-primary*` 与 `--color-accent`。

### 1.2 双模态:框架深、内容白

| 模态 | 场景 | 配色 | 依据 |
| --- | --- | --- | --- |
| **导航态** | 日常:看课、管理、看板、列表 | 深色顶栏/侧栏白底 + **白底内容区** | 长时间读题/读文档,白底正文更护眼;设计智能虽推纯 OLED 暗色,但教学场景**经论证偏离**——正文不全黑。 |
| **沉浸态** | 做实验/仿真/答题/对抗回放 | **深色全屏工作台**(IDE/终端/仿真画布) | 代码与可视化在深色下对比更高、更聚焦,符合开发者工具直觉。 |

这套「深框架 + 白内容 + 沉浸深色」是**有意的折中**,不是没做暗色模式。纯暗色模式作为后续增量(见 §11)。

### 1.3 Logo 与标识

- Logo mark:青电渐变方块(`linear-gradient(135deg, cyan-400, cyan-600)`)+ Lucide 图标,深色顶栏上对比 ≥ 9:1。
- 不臆造第三方品牌 Logo;引用任何外部标识遵守其使用规范。

## 2. 设计令牌(Design Tokens)

> 令牌是**原始色阶(Primitive)→ 语义令牌(Semantic)** 两层。组件只引用语义层。真相源:`frontend/packages/ui/tokens/`。

### 2.1 颜色 — 原始色阶

完整 50–950 色阶定义以 `frontend/packages/ui/tokens/` 为准。关键锚点:

| 色族 | 角色 | 关键档位 | 用途锚点 |
| --- | --- | --- | --- |
| **青电 cyan** | 品牌主色 | `400 #22D3EE`(深底点睛,对 ink ≈9:1)、`500 #06B6D4`(纯品牌:指示条/焦点环)、`700 #0E7490`(浅区文字/CTA 底,对白字 ≈4.8:1) | 交互/导航/焦点 |
| **琥珀 amber** | 品牌副色 | `400/500`(深底点睛/装饰)、`700 #B45309`(浅区文字/底,对白字 4.5:1 达 AA) | 能量/对抗/告警 |
| **石板 slate** | 中性骨架 | `50→950` 全阶 | 背景/文字/边框/深色面板 |
| **状态** | 反馈 | green/red/blue/purple/teal 的 `50/100`(浅底)+ `600/700`(文字) | 成功/危险/信息/特殊 |

**无障碍硬约束(写进令牌注释)**:琥珀做文字/按钮底一律用 `amber-700`(对白字 4.5:1);亮琥珀 `400/500` 只用于深色面板上的点睛与**非文字装饰**(指示条/焦点环/图标),禁止用亮琥珀承载浅底上的正文。

### 2.2 颜色 — 语义令牌

分三组:**浅色内容区 / 深色面板 / 语义反馈**。命名规约 `--color-<role>[-<variant>]`。

```
/* 浅色内容区 */
--color-bg / surface / surface-sunken / surface-hover
--color-text / text-strong / text-sub / text-faint
--color-border / border-strong / border-subtle

/* 主色(青电) */
--color-primary / primary-hover / primary-fg / primary-soft / primary-text / accent / focus-ring
/* 副色(琥珀) */
--color-secondary / secondary-hover / secondary-soft / secondary-text / secondary-accent

/* 语义反馈(文字用 *-700,底用 *-50/100) */
--color-success/-bg/-border  --color-warning/-bg/-border
--color-danger/-bg/-border   --color-info/-bg/-border  --color-purple/-bg/-border
--color-danger-solid/-hover  /* 实心危险按钮 */

/* 深色面板(顶栏/Hero/沉浸工作台) */
--color-dark-bg / dark-surface / dark-elevated / dark-text / dark-text-sub / dark-border
--color-on-dark-accent  --color-terminal-bg  --color-editor-bg
```

> `text-faint` 仅用于占位/禁用,**不承载信息**(对比度不足,信息态禁用)。深底次要文字用 `slate-300`(≈7:1),不用 `slate-400`(偏低)。

### 2.3 字体

- **拉丁/数字:Inter**(400/500/600/700/800);**中文:系统中文**(PingFang SC / Microsoft YaHei)渐进增强。
- **等宽:JetBrains Mono**(代码、哈希、地址、trace_id、数据列)。
- 生产环境**自托管字体**(`font-display: swap`),本地预览允许离线回退系统字体。
- **字号阶梯**(令牌):`xs 12 / sm 13 / base 14 / md 15 / lg 16 / xl 18 / 2xl 22 / 3xl 28 / 4xl 34`。正文基准 14(数据密集后台),移动端正文 ≥16 防 iOS 自动缩放。
- **行高**:`tight 1.3 / normal 1.5 / relaxed 1.65`;正文 1.5,长文 1.65。
- **数字列必用等宽数字**(`font-variant-numeric: tabular-nums`):统计数值、成绩、计时、表格数字列,防跳动。

### 2.4 间距 · 圆角 · 阴影 · 动效 · 层级

| 维度 | 令牌 | 值 |
| --- | --- | --- |
| **间距**(4/8 节奏) | `--space-1..16` | 4/8/12/16/20/24/32/40/48/64 |
| **圆角** | `--radius-xs/sm/(base)/lg/full` | 6/8/12/16/999 |
| **阴影**(柔和分层) | `--shadow-xs/(base)/md/lg` | 见 §4.1,低透明度多层叠加,杜绝硬黑影 |
| **过渡** | `--t-fast/(t)/-slow` | .12s / .18s / .3s |
| **签名缓动** | `--ease / --ease-out` | `cubic-bezier(.22,.61,.36,1)` / `cubic-bezier(.16,1,.3,1)` |
| **层级 z-index**(统一阶,禁止裸数字) | `--z-below/ground/base/sticky/dropdown/drawer/immersive/modal/toast` | -1/0/1/50/60/80/100/120/200 |
| **布局尺寸** | `--topnav-h/sidebar-w/-collapsed/content-max` | 56/248/64/1200px |

> z-index **只能引用令牌阶**,严禁组件里写 `z-index: 9999`。新层级需求先在令牌表新增一阶。

## 3. 无障碍基线(WCAG AA,强制)

无障碍是铁律 FE-2,不是可选项。设计智能将其列为**最高优先级(CRITICAL)**。

### 3.1 对比度

| 元素 | 标准 | 本规范落点 |
| --- | --- | --- |
| 正文文字 | ≥ 4.5:1 | `text` 对白底 ≈12:1;`text-sub` ≈4.6:1 |
| 大字(≥18px/粗 14px)、图形/图标 | ≥ 3:1 | 状态点、指示条、图标达标 |
| CTA 按钮文字 | ≥ 4.5:1 | 青电 CTA 用 `cyan-700` 对白字 ≈4.8:1;琥珀底用 `amber-700` |
| 深底文字 | 主 ≥4.5:1 / 次 ≥3:1 | `dark-text` ≈15:1;`dark-text-sub`(slate-300)≈7:1 |
| 焦点环 / 装饰 | ≥ 3:1 | `accent`(cyan-500)在深浅底均达标 |

历史修复方向:①琥珀 CTA 使用 `amber-700`;②激活态、深底次要文字对比度上调。**新增颜色必须查表验证后入令牌。**

### 3.2 焦点 · 键盘 · 动效 · 颜色

- **焦点可见**:全局 `:focus-visible` 焦点环(2px `focus-ring` + 2px offset),**绝不 `outline:none` 不补替代**。鼠标点击不显示焦点环(`:focus:not(:focus-visible)`)。
- **键盘可达**:Tab 顺序 = 视觉顺序;所有交互元素可聚焦可操作;模态/抽屉/向导提供 Esc/取消等逃生路径;拖拽操作提供键盘替代。
- **跳过链接**:`.skip-link` 跳到主内容(读屏/键盘用户);`.sr-only` 供读屏专用文本。
- **reduced-motion**:`@media (prefers-reduced-motion: reduce)` 全局把动画/过渡降到 ~0;流式图表冻结动画、保留当前值大字 KPI。
- **颜色非唯一信息**:状态除颜色外**必带图标或文字**(成功√/危险!/状态点配文字标签);图表系列用线型/纹理区分,不靠颜色单一。
- **图标按钮**:纯图标按钮必带 `aria-label`;触达区 ≥ 44×44(不足则扩 hitArea)。
- **表单无障碍**:`label[for]` 显式关联;错误 `role="alert"`/`aria-live`;提交后自动聚焦首个错误字段;Toast 用 `aria-live="polite"` 不抢焦点。
- **标题层级**:`h1→h6` 顺序,不跳级。

### 3.3 无障碍验收清单(每页过)

- [ ] 所有文字/控件对比度达标(查 §3.1)
- [ ] 仅用键盘可完成核心流程,焦点全程可见
- [ ] 关闭动画(reduced-motion)后无信息丢失、无布局错乱
- [ ] 状态/图表不靠颜色单一传达
- [ ] 图标按钮有 `aria-label`,图片有 alt
- [ ] 表单错误就近显示、读屏可感知、自动聚焦首错

## 4. 质感三件套(从「好」到「精致」)

> 这一章是设计上限的所在 —— 同样的功能,质感决定它看起来是「能用的后台」还是「专业的产品」。三件套:**间距·布局·层次 / 交互微动效 / 组件精度**。
>
> 注意:**本章数值是参考基线(起点),不是标准(终点)。** 组件精度、交互手感这类东西**无法靠抄一遍数值达成**,必须在**真实开发中反复实测、打磨、优化**——在真实数据、真实内容长度、真实交互节奏下逐帧调。下文给出的具体数值(时长/位移/字距等)是**经验起点供参考**,开发时应大胆调整,只要守住本章的**原则**(因果动效、统一缓动、海拔一致、可中断、无障碍)与设计气质。**判定质感是否达标,看真实产品手感,不看是否等于某份静态稿。**

### 4.1 间距 · 布局 · 层次

**目标:让眼睛一眼看到主次,而不是平铺一片。**

- **竖向节奏分层**:同级内容块统一间距(卡片与卡片、网格与卡片之间 18px);区块级 16/24/32/48 分层,层级越高间距越大。禁止相邻区块用相近但不等的间距(视觉抖动)。
- **内容区呼吸**:内容区内边距 `30px 36px`,最大宽 1200px 居中(`content-max`);数据密集页可放宽,但保持左右对称 gutter。响应式断点增大 gutter。
- **标题层次(kicker 范式)**:小标签(kicker:11px 大写字距 +.09em 青电)+ 大标题(25px 紧字距 -.02em)+ 副述。三段式建立强层次,替代「光秃秃一行标题」。
- **卡片内边距**:`card-pad` 20px;卡头 `15px 20px`。信息密度高的表格卡可贴边。
- **留白即分组**:用留白把相关项聚成组、把无关项分开(白空间是结构,不是浪费)。
- **阴影分层**:四档阴影对应四种海拔 —— `xs`(卡片静置)/`base`(浮起)/`md`(下拉/悬浮卡)/`lg`(模态/抽屉/Toast)。**禁止随机阴影值**,海拔与阴影一一对应。

```
--shadow-xs: 0 1px 2px rgba(15,23,42,.05);
--shadow:    0 1px 2px rgba(15,23,42,.04), 0 2px 8px rgba(15,23,42,.05);
--shadow-md: 0 4px 14px rgba(15,23,42,.08), 0 2px 6px rgba(15,23,42,.04);
--shadow-lg: 0 12px 32px rgba(15,23,42,.12), 0 4px 12px rgba(15,23,42,.06);
```

### 4.2 交互微动效

**目标:每个动效都表达「因果」,不为动而动。** 设计智能红线:动效 150–300ms、只动 `transform/opacity`、入场 ease-out 出场 ease-in 且更快、单视图最多 1–2 个关键元素动、装饰性持续动画一律禁止。下表数值为**参考起点**,实际手感开发时实测微调(不同元素、不同内容长度的体感不同)。

| 场景 | 动效 | 时长/缓动 |
| --- | --- | --- |
| 页面进入 | 内容区位移淡入(translateY 10px → 0) | .3s ease-out |
| 卡片悬浮 | 抬升 translateY(-3px)+ 阴影升 `lg`+ 边框转主色 | .22s ease |
| 按钮悬浮/按压 | 悬浮微升 + 辉光阴影;按压 `scale(.985)` | .16s / .12s |
| 侧栏项悬浮 | 右移 2px + 背景渐变 | .15s ease |
| Tab 切换 | 下划线**滑入**(scaleX 0→1,居中扩展) | .22s ease |
| 行悬浮 | 背景渐变(青电极浅底) | .14s ease |
| 弹层/下拉 | 从触发源缩放淡入(transform-origin 对齐) | .18s ease-out |
| 模态入场 | translateY(12px)+ scale(.97) 淡入 | .22s ease-out |
| 加载/处理中 | 图标旋转(`loader` 自动 spin),>300ms 操作必给反馈 | 1s linear 无限(仅加载) |
| 进度/步骤 | 宽度/状态平滑过渡 | .3–.4s ease |
| 列表入场(可选) | 逐项错峰 30–50ms | ease-out |

**统一签名缓动**:全局共用 `--ease`/`--ease-out`,所有动效同一节奏感。**可中断**:用户操作立即取消进行中的动画,绝不阻塞输入。

### 4.3 组件精度

**目标:细节经得起放大看。** 以下是每个组件**至少要打磨的维度**(参考起点,非定值):开发时在真实数据下逐项实测,该加的细节(过渡、层次、对齐、状态反馈)做透,做到比下列基线更精致。**数值仅示意方向,以实际观感为准。**

- **按钮**:字重 600 + 字距 +.01em;主按钮带内高光(`inset 0 1px 0 rgba(255,255,255,.12)`)+ 主色投影;悬浮升投影,按压回落微缩。
- **表格**:表头大写 + 字距 +.05em + 11px 弱色;数字列等宽数字;行悬浮过渡;边框用 `border-subtle` 弱化横线。
- **表单**:输入框 hover 边框加深、focus 主色边框 + 3px 青电浅环;label 下距 7px;错误态红边 + 就近红字 + 图标。
- **向导步骤**:激活态圆点带**发光环**(`0 0 0 4px primary-soft`);完成态绿;连接线状态平滑。
- **统计卡**:数字大号紧字距 + 等宽;图标容器内描边(`inset 0 0 0 1px`)。
- **徽章**:`3px 10px` + 字距;语义色对(底 100 / 字 700/800)。
- **Tab 下划线**:滑入式(见 4.2),非整条 border 切换。
- **滚动条**:细化(10px,thumb 圆角 + padding-box 内缩),精致不抢视觉。
- **侧栏激活**:主色浅底 + 左侧 3px 青电指示条 + `inset` 描边。

## 5. 组件库规范

> 四端共享组件落在 `frontend/packages/ui`。每个组件 = **样式令牌契约 + React 组件契约(props/状态/无障碍)**。组件的精度、状态完整度、交互手感以本规范原则为准,开发时打磨到位(见 §4)。

### 5.1 组件清单(基础层)

| 组件 | 关键变体 | 必备状态 | 无障碍要点 |
| --- | --- | --- | --- |
| **Button** | primary / outline / ghost / danger / on-dark;sm/lg/icon/block | hover/active/disabled/loading | loading 时 disabled + spinner;icon-only 带 aria-label |
| **Card** | 静置 / hover(可点) / 带卡头 | hover(可点卡) | 可点卡整卡可聚焦 |
| **Badge / Dot** | amber/green/red/blue/purple/gray/teal | — | 状态点必配文字(色非唯一) |
| **Table** | 可排序 / 行操作 / 空态 / 骨架 | hover/sort/loading/empty | `aria-sort`;表头 `th[scope]` |
| **Form**(input/select/textarea/checkbox/radio/switch) | 图标输入 / 错误态 | focus/hover/disabled/invalid/readonly | `label[for]`;`aria-invalid`;错误 `role=alert` |
| **Tabs** | 滑入下划线 | active/hover | `role=tablist`,键盘左右切换 |
| **Steps** | 向导步骤 | active/done | 当前步 `aria-current` |
| **Progress** | 主色 / 绿 | — | 配文字百分比 |
| **Stat** | 统计卡 | 升/降 delta | 等宽数字;delta 配箭头图标 |
| **Modal / Drawer** | sm/lg/xl / 右抽屉 | 入场/关闭 | scrim 40–60%;Esc 关闭;焦点陷阱;关闭按钮 aria-label |
| **Popover / Menu** | 下拉菜单 / 危险项 | hover | 从触发源缩放;键盘可达 |
| **Toast** | success/error/info | 自动消失 3–5s | `aria-live=polite` 不抢焦点;带 trace_id |
| **Callout** | info/warn/danger/success | — | 左 3px 语义边 + 图标 |
| **Empty** | 空状态 | — | 图标 + 标题 + 引导 + 行动按钮 |
| **Skeleton** | line/title/block | shimmer | 占位预留空间防 CLS |
| **Pagination** | 页码 | active/disabled | 按钮 aria-label(上一页/下一页) |
| **Breadcrumb** | 面包屑 | — | 3+ 层级页必带;`nav[aria-label]` |
| **Autosave** | saving/saved | — | 配 §9 服务端草稿 |
| **PageScaffold** | PageHeader / PageBody / PageSection / 可选 rail | 响应式重排 | 只提供日常页面结构原语,四端页面自行组合,不得用共享层强套同一业务布局 |
| **WorkbenchShell** | 左说明 / 中舞台 / 右状态 / 控制区 | `<lg` 舞台优先堆叠 | 只提供沉浸式工具壳和槽位,IDE/仿真/答题按自身语义组织内容 |

### 5.2 React 组件契约(强制)

- **零裸 hex**:组件只用令牌类名 / CSS 变量;新颜色先入令牌再用。
- **状态齐全**:每个交互组件必须实现 hover/focus-visible/active/disabled,异步组件加 loading,列表类加 empty/skeleton/error 四态。**缺态即不完整**(对应 CLAUDE.md「禁止占位/假实现」)。
- **受控优先**:表单组件受控,校验在 blur(非每次击键);提交 loading→成功/失败反馈。
- **props 命名**:语义化(`variant`/`size`/`loading`/`disabled`/`invalid`),不用样式名当 prop。
- **图标**:统一 Lucide,经图标组件注入,尺寸走令牌(`icon-sm/md/lg`),描边一致;禁 emoji、禁裸 SVG 散落。
- **可组合**:复合组件(Modal=Head/Body/Foot,Table=列定义驱动)用组合而非巨型 props。
- **文案外置**:可见文案集中管理(i18n 友好),不硬编码在组件深处;面向用户(FE-4)。

### 5.3 组件化原则(该组件化的就组件化)

判定一段 UI 是否该提为组件:

1. **复用 ≥ 2 处** 且 结构稳定 → 提 `packages/ui` 基础组件。
2. **业务语义强、跨端复用**(如「成绩单元格」「沙箱状态灯」「检查点判分卡」)→ 提**业务组件**层(`packages/ui/biz` 或各端 `features/*/components`),引用基础组件。
3. **仅单页用、结构易变** → 留在页面内,不过早抽象(对应工程总则「三行相似优于过早抽象」)。

> 反面:不把「按钮 + 一段文案」这种一次性组合硬抽成组件;也不把出现 5 处的「沙箱状态灯」复制 5 份。**复用即组件,一次即内联。**

## 6. 信息架构与导航

### 6.1 四端结构

四个角色端**同一个浏览器入口、同一套外壳、同一套组件、不同角色路径和导航配置**。私有化部署无 platform-admin 端。

| 端 | 登录落点(侧栏首项) | 导航分组(摘要) |
| --- | --- | --- |
| **学生 student** | `student/courses` 我的课程 | 学习(课程/实验/竞赛/仿真实验室)· 成绩(成绩/学业预警)· 账户 |
| **教师 teacher** | `teacher/courses` 课程管理 | 教学(课程/批改)· 实践(实验/竞赛/实时监控)· 资源(题库/组卷/仿真场景/共享库)· 成绩报送 · 组织 · 账户 |
| **学校管理员 school-admin** | `school-admin/accounts` 账号管理 | 用户与组织 · 概览 · 成绩(审核/申诉/预警/配置)· 系统 |
| **平台管理员 platform-admin** | `platform-admin/tenants` 租户 | 租户 · 运营 · 系统(SaaS 专属) |

### 6.2 导航规范

- **登录直达功能页**(FE-5):无独立工作台落地页;登录后跳该角色侧栏第一项。
- **侧栏 = 二级导航**:分组标题(11px 大写弱色)+ 菜单项(图标 + 文字 + 可选角标计数)。**激活态**:主色浅底 + 左 3px 青电指示条 + 字重升。角标计数用于待办(批改 5、审核 4、申诉 2、预警 1)。
- **图标 + 文字并存**:导航项绝不纯图标(可发现性)。当前位置必高亮。
- **面包屑**:3+ 层级深页必带(`nav[aria-label=breadcrumb]`),辅助定位。
- **导航一致性**:四端导航位置/行为一致,不随页面类型变化;危险操作(登出等)与普通导航项视觉/空间分离。
- **顶栏(深色)**:Logo + 全局搜索 + **通知铃铛** + 帮助 + 角色 pill + 头像菜单。

### 6.3 通知:顶栏铃铛(已从侧栏移出)

通知**统一在顶栏铃铛下拉**,不再占侧栏菜单项:

- 铃铛带未读角标(琥珀 `amber-500`,深色顶栏上点睛);
- 下拉含:最近通知列表 + 系统公告 + 「查看全部」入口;
- 四端统一;通知数据经 M10(通知模块),前端不自建通知存储(对应架构铁律「通知统一经 notify」)。

### 6.4 窗口自适应(响应式,FE-9 必做)

> **强制项**:所有页面与组件随窗口流式自适应,而非固定像素布局。侧栏折叠/抽屉、沉浸式工作台窄屏堆叠、表格重排等必须按真实功能补齐,不得以静态示意代替生产体验。

**断点系统(统一,禁止随手写魔法值)**

| 令牌 | 宽度 | 典型设备 | 主要行为 |
| --- | --- | --- | --- |
| `--bp-sm` | 640 | 大手机 | 单列;侧栏抽屉化;表格转卡片/横滚 |
| `--bp-md` | 768 | 平板竖 | 双列;侧栏可收起为图标条 |
| `--bp-lg` | 1024 | 平板横/小笔记本 | 侧栏常驻(可折叠 248↔64);多列恢复 |
| `--bp-xl` | 1280 | 桌面 | 内容区 1200 居中,完整布局 |
| `--bp-2xl` | 1536 | 大屏 | 内容不拉满长文行宽,留白增大 |

**自适应行为约定**

- **流式优先**:布局用 `flex`/`grid`(`minmax`/`auto-fit`/`fr`)、`clamp()` 弹性间距与字号、`min()/max()`,而非写死宽度;容器用 `max-width` + 居中,内部弹性。
- **侧栏**:`≥lg` 常驻且可折叠(248↔64,令牌 `--sidebar-w`/`--sidebar-w-collapsed`);`<lg` 收为**抽屉**(汉堡触发,遮罩 + Esc 关闭,焦点陷阱)。
- **顶栏**:窄屏收起全局搜索为图标、次要项进「更多」溢出菜单;Logo + 铃铛 + 头像始终可达。
- **内容网格**:统计/卡片网格按断点 4→2→1;`auto-fit minmax()` 优先,让卡片按可用宽自动换行。
- **表格(数据密集)**:`<md` 不靠纯横向滚动硬塞——优先**列优先级**(隐藏次要列)或**转卡片式**(每行变一张键值卡);确需横滚时横滚区独立、不带动整页。
- **沉浸式工作台**:三栏在 `<lg` **堆叠或转 Tab/抽屉**(左说明、右状态收为可展开面板,中舞台优先);画布(仿真节点图/拓扑)用 `viewBox` 等比缩放或容器查询自适应,**不溢出、不裁切关键信息**;移动端可降级为「只读回放 + 提示用大屏操作」,但不得报错或空白。
- **弹层**:Modal `width:100%` + `max-width` + `max-height:88vh`;Drawer `max-width:92vw`;窄屏底部弹出(bottom-sheet)更顺手。
- **媒体/图表**:声明宽高或 `aspect-ratio` 防布局抖动(CLS<0.1);图表用容器宽度自适应,小屏简化(减刻度/横向条)。
- **触控**:触屏视口下交互目标 ≥44×44,间距 ≥8px;不靠 hover 传达关键信息(hover 仅增强)。

**红线**

- 禁止:任何视口出现**非预期横向滚动**(表格/代码块的独立横滚区除外)。
- 禁止:写死容器像素宽导致窗口缩放后**遮挡、错位、截断**。
- 禁止:禁用缩放(`user-scalable=no`/`maximum-scale=1`)。
- 禁止:固定定位元素(顶栏/工作台条/底部 CTA)未给内容预留安全内距,压住内容。

**验收**:每页在 **375 / 768 / 1024 / 1440** 及**任意拖拽缩放**下走查;横竖屏均可用;`reduced-motion` 与大字模式下不破版。

### 6.5 资源页统一渲染范式

> 四端大量页面由后端资源契约统一渲染。资源页不能退化为「标题、统计、按钮、表格」纵向堆叠,也不能照搬静态图;必须在真实 React 页面中提供完整的信息层级、动作区、反馈状态和响应式重排。

**页面结构(从上到下)**

1. **页面头**:面包屑(kicker) + `h1` + 用户向描述 + Lucide 页面图标。页面头只做信息定位,不做卡片嵌套;底部分隔线用当前角色/页面强调色建立方向感。
2. **指标带**:统计卡按 `auto-fit` 流式布局,数字等宽,状态用文字说明补足颜色语义。
3. **主工作区**:数据列表为主,操作表单为辅。桌面端采用「主数据区 + 右侧动作区」;移动端单列重排,不出现整页横向滚动。
4. **动作区**:页面级操作集中在右侧动作栏,每个操作是可聚焦卡片,带序号、标题、说明、显式表单标签和提交 loading。提交成功/失败就近反馈,失败只显示用户向文案 + `trace_id`。
5. **数据区**:列表容器含分组标题、当前记录数、刷新动作、空态与表格。表格在窄屏按列优先级转卡片或独立滚动,不得撑破页面。
6. **加载/空/错三态**:加载用骨架屏预留空间,不是空白页或单独转圈;空态给下一步引导;错误态可恢复并保留报障编号。

**角色与分组表达**

- 视觉强调色由角色/功能分组映射到语义令牌,只用青电与琥珀体系;不得为了“区分页面”引入裸 hex 或新色系。
- 角色端页面保持同一外壳和交互模型,差异体现在信息密度、页面图标、指标语义、动作表单和数据列,不是四端各做一套风格。
- 资源页动效只用于进入、反馈、卡片焦点/按压和进度表达,时长遵守 §4.2;禁止装饰性持续动画。

### 6.6 多步骤流程持久化(FE-7)

适用:实验编排向导、作业草稿、竞赛出题/参战/报名、内容创作草稿、导入预览、入驻申请等。

- **服务端为权威**:每步/定时(参考 M6 作业草稿 60s 自动保存)存服务端;前端 localStorage 仅即时兜底。
- **可恢复**:刷新/关闭重开/换设备不丢;进入向导先拉服务端草稿。
- **自动保存指示**:`autosave` 组件显示 saving(琥珀)→ saved(绿),给用户确定感。

## 7. 沉浸式工作台范式

> 做实验/仿真/答题/对抗回放时进**深色全屏**沉浸态(`z-immersive`),退出回导航态。四个工作台是平台的「创新点」与教学引擎门面 —— 必须是**真教学工具,不是空动画**。

### 7.1 通用范式

- **三栏骨架**:左(说明/上下文)· 中(主舞台:编辑器/画布/题面)· 右(状态/结果/操作)。
- **深色面板令牌**:`dark-bg/surface/elevated`、`terminal-bg`、`editor-bg`;点睛用 `on-dark-accent`(青电 400)。
- **顶部工作台条**:退出、标题、进度、主操作(运行/提交/下一步);**单一主 CTA**。
- **WebSocket 实时进度**:长操作(判题/挖矿/对局)经 WS 推进度,前端给骨架/进度而非冻结。
- **共享与业务边界**:`packages/ui` 的 `WorkbenchShell` 只负责深色壳、槽位、响应式和无障碍;`packages/ide` 的 `IdeWorkbench` 只负责文件/编辑器/终端/工具状态布局;`packages/sim-sdk` 的 `SimulationWorkbench` 负责仿真叙事、舞台、控制和渲染器。四端业务页可以使用这些原语,但不得照搬共享层生成的统一页面布局。
- **仿真舞台对齐**:节点图、链、树、矩阵、泳道、趋势图都必须由仿真包的真实 `view` 数据驱动;复杂拓扑不得硬塞所有边标签导致重叠,趋势图必须提供数据表替代,选择节点/消息/单元格要有键盘可达和可读标签。

### 7.2 四个工作台

#### A. 代码实验 IDE(学生·实验)
- 编辑器 + 终端 + 检查点判分 + 链上操作 + WS 进度。
- **创新点①「区块孵化器」**:把「沙箱在干活」做成 VM 活体状态机,替代转圈 spinner —— 见 §9 整改项(状态机做全 + 中文 + 失败可点开)。

#### B. PBFT 共识仿真(学生·仿真,按 M4 文档)
教学引擎形态,**能逐阶段看清效果**:
- **左栏**:讲「这一步做了什么 + 为什么重要 + 法定人数 2f+1 判定」;
- **中间**:节点图按阶段放不同**消息飞线**(如 Prepare 全网状);
- **右侧**:时序泳道进度 + **投票矩阵** + 法定人数结果;
- **控制**:单步 / 播放;支持注入拜占庭节点 + 设问检查点。
- 可扩展更多仿真(PoW 挖矿 / 默克尔树 / 分叉),套同一范式。

#### C. 对抗赛回放(学生·竞赛复盘)
- 攻防拓扑(守方绿 / 攻方红)+ 链上日志流(随时间轴构建)+ **时间轴回溯条**。
- **创新点②「时空回溯器」**:倒带 VM 执行树 —— 见 §9,回溯须绑真实区块高度快照/执行 trace,不可是假动画。

#### D. 解题赛答题(学生·竞赛)
- 题列表 + 题面 + 起沙箱/作答 + 提交判定 + 防作弊。
- **答案黑盒**(安全铁律):题面与全量(答案/判题配置/flag)接口分离,学生端不可见。

### 7.3 工作台无障碍

- 深色面板文字对比照 §3.1(主 ≥4.5:1 / 次 ≥3:1)。
- 仿真/回放动画**必须**支持 reduced-motion(冻结动画 + 保留当前态文字)。
- 单步/暂停控制 = 无障碍刚需(让动画可被人为掌控,不强制自动播放)。

## 8. 数据可视化规范

> 看板、统计、监控、仿真、对抗回放都用图表。图表选型经设计智能(ui-ux-pro-max chart 域)校验,落表如下。**底线:色非唯一、有图例/提示、空/错/载有态、无障碍降级。**

### 8.1 图表选型表

| 场景 | 图表 | 关键约束 |
| --- | --- | --- |
| 成绩/活跃**趋势** | 折线图(Line) | <4 点用统计卡;>6 系列太杂;系列用**线型**(实/虚/点)区分非颜色;Recharts/Chart.js |
| 维度**对比** | 柱状图(Bar) | >5 类别不用饼图,改柱 |
| 占比 | 饼/环(≤5 类) | >5 类改柱状 |
| 实时监控(挖矿/对局**流式**) | 流式面积/Ticker | ≥1Hz 才用;**必带暂停/继续**;当前值大字 KPI;Canvas;reduced-motion 冻结 |
| 异常/告警 | 带高亮折线 | 异常点用**形状标记**(非颜色)+ 文字标注 |
| **投票矩阵 / 活动密度**(PBFT 共识、教务监控) | 热力/网格矩阵(Heatmap/Grid) | 冷→暖色阶 + **数值标签** + 图例刻度;色盲纹理补丁;hover 出精确值 |
| 攻防/节点关系 | 网络拓扑图 | 节点/边状态配图标+文字;守绿攻红但不靠色单一 |

### 8.2 图表无障碍与降级(强制)

- **色盲友好**:不用红/绿单一对;补线型/纹理/形状;divergent 色阶配数值图例。
- **数据表替代**:图表配可切换数据表(读屏不友好的纯图必给表),`aria` 摘要描述关键洞察。
- **交互**:hover/tap 出精确值;图例可点切换系列;键盘可达(非仅 hover)。
- **态齐全**:空数据(「暂无数据」+ 引导,非空白轴)、加载(骨架非空轴框)、错误(报错 + 重试)。
- **数字格式**:locale 感知(数字/日期/货币);轴标签不挤、小屏自动跳刻度;时间序列标明粒度(日/周/月)可切换。
- **reduced-motion**:入场动画可关,数据立即可读。

## 10. React 工程落地

> 落点遵循 `docs/总-工程目录设计.md` 的 `frontend/` monorepo 结构。本节定与设计相关的约定。

### 10.1 分层

```
  frontend/
    apps/
      web/            # 唯一浏览器入口:登录页 + 四角色路径
        src/route-kit/ # 四端页面动作、表格列、资源结果装配;属于 web 入口侧,不放 packages/shared
        src/styles/   # 四端体验层样式:分端视觉 + page-type 深页/流程/治理/沉浸页补强,不写进 shared
      immersive/       # 或各端内 features/immersive:四个工作台
    packages/
      ui/              # 设计系统:tokens + 基础组件 + 业务组件(四端共享唯一来源)
        tokens/        # 原始→语义令牌(CSS 变量 + TS 常量双出)
        components/    # Button/Card/Table/... (§5.1)
          PageScaffold/ # 日常页面结构原语:头部、主体、侧栏、区块,不绑定四端业务布局
          WorkbenchShell/ # 沉浸式工具结构原语:深色壳、三栏、控制区
        biz/           # 沙箱状态灯/投票矩阵/攻防拓扑... 业务组件
        charts/        # 仅保留兼容再导出,实现归属 packages/charts
      charts/          # 数据可视化封装:图形 + 数据表降级 + reduced-motion 契约(§8)
      shared/          # 跨端工具:配置、错误处理、hooks、类型、路由、通用应用壳;不得承载四端业务页面定义
      ide/             # IDE 工作台能力:Monaco/xterm 生命周期 + IdeWorkbench 布局壳
      sim-sdk/         # M4 仿真 SDK:Worker 运行时 + SimulationWorkbench + 封闭模式渲染器
      auth/            # 认证入口组件、登录/找回/激活/入驻/SSO、会话上下文、路由守卫、角色跳转
```

### 10.2 单入口路由约定

- `pnpm run dev` 只启动 `apps/web`,本地浏览器始终访问一个前端端口。
- 登录页位于同一入口内;手机号登录后不展示角色选择,由后端返回的账号角色决定跳转路径。
- 角色路径固定为 `/student/`、`/teacher/`、`/school-admin/`、`/platform-admin/`,进入后由共享应用壳渲染该角色的第一个功能页。
- 禁止恢复四套独立 Vite app 作为默认启动形态;如需临时调试角色页面,也必须通过 `apps/web` 的同源路径完成。

### 10.3 约定

- **组件零裸 hex**:ESLint 规则禁裸颜色,只许令牌;CI 卡。
- **命名**:组件 PascalCase;目录 kebab-case(`school-admin`);hooks `useXxx`;令牌 `--color-*` / `--space-*`。
- **状态管理**:服务端状态用查询库缓存;多步骤向导中间态走服务端(FE-7),前端只缓存即时态。
- **错误边界**:每端套错误边界,兜底友好页(含 trace_id),不白屏不泄技术细节。
- **图标**:Lucide 经统一图标组件;尺寸令牌化;tree-shake 按需引入。
- **可访问性测试**:关键页跑 axe 类自动检查 + 键盘走查纳入验收。
