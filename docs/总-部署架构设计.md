# 部署架构设计

> 双形态部署:SaaS 公网统一运营 + 学校私有化。K8s/k3s 编排,Kustomize 分层,CI/CD 自动化。
> 最后更新:2026-06-04

---

## 一、双形态部署

| 形态 | 编排 | 租户 | 平台管理员 | 镜像分发 |
| --- | --- | --- | --- | --- |
| **SaaS 公网** | Kubernetes | 多租户(school_id + RLS) | 有 | 公网 Harbor + 上游 digest 锁 |
| **私有化(单校)** | k3s(轻量) | 单租户(tenant_id 恒定) | 无(学校管理员顶层) | 全量离线镜像包 |

- 同一套代码、同一套镜像;差异仅配置(deploy_mode、是否启用 platform 层)。
- "多租户与私有化同构"原则落地:私有化即单租户特例。
- `images/` 是部署镜像的单一治理来源:平台自建镜像、官方薄封装镜像和纯上游固定镜像都必须按 manifest 锁定来源、版本、digest、许可证、离线导入、安全准入和 `deploy/config/chaimir.env` 环境变量键;敏感值只由 K8s Secret/KMS 注入同名环境变量。

## 二、K8s 命名空间规划

| 命名空间 | 内容 |
| --- | --- |
| `chaimir-system` | 后端单体、前端、中间件接入 |
| `chaimir-data` | PostgreSQL、Redis、MinIO、NATS、ClamAV(或外部托管) |
| `monitoring` | Prometheus、Grafana(外接) |
| `chaimir-prepull` | M2 镜像预拉取 DaemonSet |
| `sbx-*`(动态) | M2 学生沙箱(每实例一个,用后即毁) |
| `judge-*`(动态) | M3 判题沙箱(用后即毁) |
| `battle-*`(动态) | M8 对局沙箱(用后即毁) |

- **动态命名空间**由后端 ServiceAccount(最小 RBAC,仅能在 `sbx-*/judge-*/battle-*` 内操作)创建/销毁。
- **NetworkPolicy 默认 deny-all**:动态沙箱命名空间间、与系统命名空间严格隔离,按需精确放行。

## 三、组件部署拓扑

```
                    ┌─────────────┐
   用户 ──HTTPS──►  │  Ingress/网关 │ ──► 前端(Nginx静态)
                    └──────┬──────┘
                           │ /api
                    ┌──────▼──────────┐
                    │ backend 单体(多副本) │ ──► PG / Redis / MinIO / NATS / ClamAV
                    │ + 模块后台任务        │
                    └──────┬──────────┘
                           │ client-go(创建/销毁动态 Pod)
              ┌────────────▼────────────┐
              │  sbx-* / judge-* / battle-*  │ (动态沙箱 Pod,隔离)
              └─────────────────────────┘
```

- 后端单体多副本水平扩展(无状态,会话在 Redis、JWT 无状态)。
- 判题、撮合、回收、outbox 等后台任务当前由 server 内统一 background runner 承载;部署层不得创建没有真实入口命令的 worker Deployment。
- 动态沙箱 Pod 按需创建,受租户配额约束。

## 四、环境分层(Kustomize)

```
deploy/
├── base/                通用 manifests(Deployment/Service/RBAC/NetworkPolicy)
├── components/          可复用 Kustomize components(本地/私有化中间件等)
├── config/              环境变量与 Secret 模板(配置单一来源)
└── overlays/
    ├── local-dev/       本地开发(当前 kubeconfig 指向的 Kubernetes 集群)
    ├── staging/         预发布
    ├── prod-saas/       SaaS 生产(公网多租户)
    └── prod-school/     私有化(k3s,单校,关闭 platform 层)
```

- base 定义通用资源;overlays 覆盖环境差异(副本数、资源、域名、deploy_mode、是否启用平台管理员)。
- 私有化 overlay:去掉 `/platform/*` 入口、平台级定时任务,`chaimir/migrate migrate-and-seed` 创建单校租户与首个学校管理员。
- Chaimir 自身部署清单统一走 Kustomize,不维护平台 Helm chart,避免与 base/overlays 双轨重复。
- 第三方 K8s 组件可按官方方式安装;若官方推荐 Helm,仅在需要固定 values 或交付说明时创建
  `deploy/charts/` 保存第三方 chart values,不得放 Chaimir 平台自身模板。
- SaaS 环境密钥统一经 `deploy/config/external-secret/` 的 ExternalSecret 模板同步;overlay 只覆盖
  SecretStore 名称、刷新周期和远端键前缀。
- 数据库凭据分工固定:`PG_PRIV_USER/PG_PRIV_PASSWORD` 是 PostgreSQL 属主/特权连接,仅用于中间件初始化、migration 和 M1 受控登录前跨租户定位;`PG_USER/PG_PASSWORD` 固定为 `chaimir_app` 应用角色,常规后端请求必须走该角色以确保 RLS 生效。禁止再引入单独的 owner DSN 或第二份应用角色口令。
- 沙箱私有镜像拉取凭据只以 Kubernetes `dockerconfigjson`/`dockercfg` Secret 存放在控制命名空间;后端通过 `SANDBOX_IMAGE_PULL_SECRET_NAMES` 读取 Secret 名称,复制到 `chaimir-prepull` 和动态沙箱命名空间供 ServiceAccount/预拉取 DaemonSet 引用。Harbor robot token 不得写入 env、Kustomize manifest、镜像层或工具 manifest。

## 五、中间件

| 中间件 | SaaS | 私有化 |
| --- | --- | --- |
| PostgreSQL | 托管/自建主从 | 单实例(可加备) |
| Redis | 托管/集群 | 单实例 |
| MinIO | 集群 | 单实例 |
| NATS | 集群 | 单实例 |
| ClamAV | 集群/托管扫描服务 | 单实例 |
| CouchDB | Fabric 富查询实验按需编排 | Fabric 富查询实验按需编排 |
| Harbor | 公网私有仓 | 校内仓 + 离线导入 |

> 私有化按校规模可单机部署全套(k3s + 单实例中间件)。
> 中间件、Prometheus/Grafana 和 ingress-nginx 的镜像来源以 `images/middleware`、`images/observability`、`images/ingress` manifest 为准。纯上游固定镜像不重打包,但必须按 digest 拉取、扫描、签名/准入校验并进入离线包。

## 六、CI/CD

```
代码提交 → 路径触发(backend/frontend/images 各自流水线)
  ├─ lint + 单元测试
  ├─ 构建平台镜像(多阶段)或校验上游固定镜像 digest
  ├─ Trivy 漏洞扫描(高危阻断)
  ├─ Cosign 签名或验证准入策略认可的上游签名/镜像锁
  └─ 推 Harbor
部署:
  main 分支 backend 镜像流水线成功 → 自动部署 staging
  打 tag → 人工审批 → 部署 prod-saas
  私有化 → 生成离线镜像包 + 部署清单交付学校
```

- 文档、README、前端纯静态资源或独立镜像清单变更不得直接创建 staging Deployment；staging 自动部署只消费已完成扫描、签名并推送到 Harbor 的后端镜像 SHA，避免部署流水线早于镜像产物或用不存在的 tag 发布。

- 定时任务(CronJob):每日数据备份由 `service/cron` 镜像执行真实 `pg_dump` + MinIO 对象备份,并记录入 M9 `backup_record`。业务生命周期清理留在所属模块后台任务中运行,不得在部署层保留无命令实现的占位 CronJob。

## 七、资源回收

- M2/M4/M7/M8 各自的回收逻辑为主(空闲/最长生命周期/来源级联/事件通知)。
- 资源回收由模块 service 内的后台任务统一执行并暴露 `RunXOnce(ctx)`,通过 `cmd/server/<module>.go` 接入 `platform/background`;部署层不得保留未实现的强制清理 CronJob。若后续需要集群级孤儿资源治理,必须新增真实受控运维命令、权限边界和审计设计后再接入部署清单。
- M2 快照功能要求集群安装 CSI VolumeSnapshot CRD、snapshot-controller、支持快照的 CSI 存储驱动、可用的 StorageClass 与 VolumeSnapshotClass,并在环境变量 `SANDBOX_STORAGE_CLASS_NAME`、`SANDBOX_VOLUME_SNAPSHOT_CLASS_NAME` 中显式声明;未配置或不可读取时不得启用 `snapshot_enabled`。部署层提供 `make snapshot-up` 安装官方 CSI snapshotter 的 CRD 与 controller,但不安装具体存储驱动;生产必须接入 Longhorn、Ceph RBD、云盘 CSI 等真实支持快照的存储,不得把普通 Docker volume、`rancher.io/local-path` 或演示用 hostpath CSI 当作生产快照能力。
- M2 沙箱资源用量查询要求集群安装 metrics-server 并暴露 `metrics.k8s.io`;未安装或不可读取时,后端必须返回沙箱专用错误码,不得用 Pod requests/limits 伪装成实时用量。部署层提供 `make metrics-up` 安装官方 metrics-server;仅 Docker Desktop 等本地集群可使用 `make metrics-up-local` 追加 `--kubelet-insecure-tls`,生产集群不得使用该本地放宽参数。

## 八、可用性与备份

- 后端无状态多副本;中间件按形态做主从/集群(SaaS)或单实例+备份(私有化)。
- 当前 SaaS 生产按后端单体副本水平扩展,后台任务随 server 副本运行;若后续拆独立 worker,必须先新增真实入口命令、权限边界、队列指标和部署清单,再启用队列积压 HPA。
- 备份:数据库每日全量使用 PostgreSQL 官方 `pg_dump` 自定义格式;MinIO 代码/附件/报告桶对象服务端复制到备份桶;备份记录入 M9。当前只登记全量备份能力,任何新的备份类型都必须先补齐真实策略、恢复流程和配置后再同步本文档。
- 私有化数据不出校内网,符合数据合规。

## 九、安全部署基线

- 全站 HTTPS;Ingress WAF(可选)。
- 最小权限 RBAC;后端 ServiceAccount 仅能操作动态沙箱命名空间。
- ValidatingAdmissionPolicy 在准入层限制后端 ServiceAccount 只能管理带平台所有权标签的
  `sbx-*`/`judge-*`/`battle-*` 命名空间。
- 密钥(JWT 签名、加密密钥、中间件凭据)经 K8s Secret/KMS 注入,不入镜像不入仓。
- 镜像签名校验门禁由 Sigstore policy-controller 执行。平台构建镜像必须带 Cosign 签名;纯上游固定镜像必须命中 `images/` manifest 中的 digest 锁与准入策略,不得以可变 tag 绕过;
  Pod Security Restricted;高风险沙箱 gVisor/Kata 加固。
- M2 预拉取 DaemonSet 只能运行在 `chaimir-prepull` 命名空间,并使用与沙箱节点一致的 node selector/toleration。

## 十、私有化交付清单

1. 离线镜像包(平台服务 + 该校启用的链运行时/工具/判题器/仿真包 + 必需中间件/可观测/入口镜像),所有条目来自 `images/` manifest 并按 digest 导入校内 Harbor。
2. k3s 部署清单(prod-school overlay)。
3. 初始化脚本/`chaimir/migrate` 命令(建库 + RLS + 角色授权 + 首个学校管理员 + seed 镜像元信息)。
4. 运维手册(备份/升级/监控接入)。
