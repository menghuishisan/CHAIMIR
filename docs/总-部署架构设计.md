# 部署架构设计

> 双形态部署:SaaS 公网统一运营 + 学校私有化。K8s/k3s 编排,Kustomize 分层,CI/CD 自动化。
> 最后更新:2026-06-04

---

## 一、双形态部署

| 形态 | 编排 | 租户 | 平台管理员 | 镜像分发 |
| --- | --- | --- | --- | --- |
| **SaaS 公网** | Kubernetes | 多租户(school_id + RLS) | 有 | 公网 Harbor |
| **私有化(单校)** | k3s(轻量) | 单租户(tenant_id 恒定) | 无(学校管理员顶层) | 离线镜像包 |

- 同一套代码、同一套镜像;差异仅配置(deploy_mode、是否启用 platform 层)。
- "多租户与私有化同构"原则落地:私有化即单租户特例。

---

## 二、K8s 命名空间规划

| 命名空间 | 内容 |
| --- | --- |
| `chaimir-system` | 后端单体、前端、中间件接入、worker |
| `chaimir-data` | PostgreSQL、Redis、MinIO、NATS(或外部托管) |
| `monitoring` | Prometheus、Grafana(外接) |
| `chaimir-prepull` | M2 镜像预拉取 DaemonSet |
| `sbx-*`(动态) | M2 学生沙箱(每实例一个,用后即毁) |
| `judge-*`(动态) | M3 判题沙箱(用后即毁) |
| `battle-*`(动态) | M8 对局沙箱(用后即毁) |

- **动态命名空间**由后端 ServiceAccount(最小 RBAC,仅能在 `sbx-*/judge-*/battle-*` 内操作)创建/销毁。
- **NetworkPolicy 默认 deny-all**:动态沙箱命名空间间、与系统命名空间严格隔离,按需精确放行。

## 三、组件部署拓扑

```
                    ┌─────────────┐
   用户 ──HTTPS──►  │  Ingress/网关 │ ──► 前端(Nginx静态)
                    └──────┬──────┘
                           │ /api
                    ┌──────▼──────────┐
                    │ backend 单体(多副本) │ ──► PG / Redis / MinIO / NATS
                    │  + worker(判题/撮合) │
                    └──────┬──────────┘
                           │ client-go(创建/销毁动态 Pod)
              ┌────────────▼────────────┐
              │  sbx-* / judge-* / battle-*  │ (动态沙箱 Pod,隔离)
              └─────────────────────────┘
```

- 后端单体多副本水平扩展(无状态,会话在 Redis、JWT 无状态)。
- 判题/撮合 worker 按队列积压伸缩。
- 动态沙箱 Pod 按需创建,受租户配额约束。

## 四、环境分层(Kustomize)

```
deploy/
├── base/                通用 manifests(Deployment/Service/RBAC/NetworkPolicy)
├── components/          可复用 Kustomize components(本地/私有化中间件等)
├── config/              环境变量与 Secret 模板(配置单一来源)
└── overlays/
    ├── local-dev/       本地开发(docker-compose 或 kind)
    ├── staging/         预发布
    ├── prod-saas/       SaaS 生产(公网多租户)
    └── prod-school/     私有化(k3s,单校,关闭 platform 层)
```

- base 定义通用资源;overlays 覆盖环境差异(副本数、资源、域名、deploy_mode、是否启用平台管理员)。
- 私有化 overlay:去掉 `/platform/*` 入口、平台级定时任务,`chaimir/migrate migrate-and-seed` 创建单校租户与首个学校管理员。
- Chaimir 自身部署清单统一走 Kustomize,不维护平台 Helm chart,避免与 base/overlays 双轨重复。
- 第三方 K8s 组件可按官方方式安装;若官方推荐 Helm,仅在需要固定 values 或交付说明时创建
  `deploy/charts/` 保存第三方 chart values,不得放 Chaimir 平台自身模板。
- SaaS 环境密钥统一经 `deploy/config/external-secret/` 的 ExternalSecret 模板同步;overlay 只覆盖
  SecretStore 名称、刷新周期和远端键前缀。
- 数据库凭据分工固定:`PG_PRIV_USER/PG_PRIV_PASSWORD` 是 PostgreSQL 属主/特权连接,仅用于中间件初始化、migration 和 M1 受控登录前跨租户定位;`PG_USER/PG_PASSWORD` 固定为 `chaimir_app` 应用角色,常规后端请求必须走该角色以确保 RLS 生效。禁止再引入单独的 owner DSN 或第二份应用角色口令。

## 五、中间件

| 中间件 | SaaS | 私有化 |
| --- | --- | --- |
| PostgreSQL | 托管/自建主从 | 单实例(可加备) |
| Redis | 托管/集群 | 单实例 |
| MinIO | 集群 | 单实例 |
| NATS | 集群 | 单实例 |
| Harbor | 公网私有仓 | 校内仓 + 离线导入 |

> 私有化按校规模可单机部署全套(k3s + 单实例中间件)。

## 六、CI/CD

```
代码提交 → 路径触发(backend/frontend/images 各自流水线)
  ├─ lint + 单元测试
  ├─ 构建镜像(多阶段)
  ├─ Trivy 漏洞扫描(高危阻断)
  ├─ Cosign 签名
  └─ 推 Harbor
部署:
  main 分支 → 自动部署 staging
  打 tag → 人工审批 → 部署 prod-saas
  私有化 → 生成离线镜像包 + 部署清单交付学校
```

- 定时任务(CronJob):PV/VolumeSnapshot 清理、镜像 GC、每日数据备份(记录入 M9 backup_record)、僵尸沙箱兜底回收。

---

## 七、资源回收(部署层兜底)

- M2/M4/M7/M8 各自的回收逻辑为主(空闲/最长生命周期/来源级联/事件通知)。
- 部署层 CronJob 兜底:扫描超期未回收的动态命名空间强制清理,防资源泄漏。
- M2 快照功能要求集群安装 CSI VolumeSnapshot CRD 与支持快照的 StorageClass;未安装时不得启用 `snapshot_enabled`。

---

## 八、可用性与备份

- 后端无状态多副本;中间件按形态做主从/集群(SaaS)或单实例+备份(私有化)。
- SaaS 生产 worker 通过 Prometheus Adapter 暴露的队列积压 external metric 伸缩,不使用 CPU 兜底替代。
- 备份:数据库每日全量 + 增量;MinIO 对象存储备份;备份记录入 M9。
- 私有化数据不出校内网,符合数据合规。

---

## 九、安全部署基线

- 全站 HTTPS;Ingress WAF(可选)。
- 最小权限 RBAC;后端 ServiceAccount 仅能操作动态沙箱命名空间。
- ValidatingAdmissionPolicy 在准入层限制后端 ServiceAccount 只能管理带平台所有权标签的
  `sbx-*`/`judge-*`/`battle-*` 命名空间。
- 密钥(JWT 签名、加密密钥、中间件凭据)经 K8s Secret/KMS 注入,不入镜像不入仓。
- 镜像签名校验门禁由 Sigstore policy-controller 执行,平台自建镜像必须带 Cosign 签名;
  Pod Security Restricted;高风险沙箱 gVisor/Kata 加固。
- M2 预拉取 DaemonSet 只能运行在 `chaimir-prepull` 命名空间,并使用与沙箱节点一致的 node selector/toleration。

## 十、私有化交付清单

1. 离线镜像包(平台服务 + 该校启用的链运行时/工具/判题器)。
2. k3s 部署清单(prod-school overlay)。
3. 初始化脚本/`chaimir/migrate` 命令(建库 + RLS + 角色授权 + 首个学校管理员 + seed 镜像元信息)。
4. 运维手册(备份/升级/监控接入)。
