# M10 通知与实时推送 — 接口设计

> RESTful + WebSocket。鉴权沿用 M1 JWT。内部 Send 接口供各模块调用。
> 最后更新:2026-05-29

---

## 1. 通用约定

- Base:`/api/v1/notify`
- 统一响应体同 M1。错误码段:`A0xxx`(站内信/公告/实时)。
- `[内部]` 供各模块调用(服务间鉴权)。

### 1.1 M10 错误码

| 错误码 | 含义 | 用户向文案 |
| --- | --- | --- |
| A0001 | 通知请求不正确 | 通知内容不完整,请检查后重试 |
| A0002 | 通知模板不存在或不可用 | 通知模板不可用,请联系管理员处理 |
| A0003 | 通知发送失败 | 通知暂时无法发送,请稍后重试 |
| A0004 | 站内信不存在或不可访问 | 站内信不存在或已被移除 |
| A0005 | 接收偏好不允许修改 | 该类通知为必要通知,不能关闭 |
| A0006 | 公告请求不正确 | 公告内容不完整,请检查后重试 |
| A0007 | 公告不存在或不可访问 | 公告不存在或已过期 |
| A0008 | 实时主题不允许订阅 | 你不能订阅该实时消息 |
| A0009 | 实时推送失败 | 实时消息暂时无法发送,请稍后重试 |
| A0010 | 通知发送过于频繁 | 通知发送过于频繁,请稍后再试 |

## 2. 统一通知服务 `[内部]`

### 2.1 发送通知
`POST /send` `[内部]`
请求:
```json
{ "type": "assignment.due", "receivers": [1001, 1002],
  "params": { "course": "区块链基础", "due": "2026-06-01 23:59" },
  "tenant_id": 10 }
```
- M10 按 type 取模板渲染 → 写 notification → 推未读红点。
- 强制类(系统/成绩)忽略用户偏好。
- 模块化单体内对应 `contracts.NotifyService.Send(ctx, NotifySendRequest)`;HTTP `[内部]` 路径用于未来服务化或受控内网调用。

### 2.2 事件总线(NATS)
各模块亦可发事件到总线,M10 订阅消费转通知,等价于调 Send 但完全异步解耦。

## 3. 站内信 `[用户]`

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/inbox?is_read=&type=` | 收件箱(分页) |
| GET | `/inbox/unread-count` | 未读数(Redis 缓存;miss 时按 `notification` 重建) |
| POST | `/inbox/{id}/read` | 标记已读 |
| POST | `/inbox/read-all` | 全部已读 |
| DELETE | `/inbox/{id}` | 删除 |
| GET/PUT | `/preferences` | 接收偏好(强制类不可关) |

## 4. 系统公告

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/announcements` `[管理员]` | 发布公告(全平台/本校/指定角色) |
| GET | `/announcements` `[用户]` | 公告列表 + 本人已读状态 |
| POST | `/announcements/{id}/read` `[用户]` | 标记已读 |

## 5. 实时通道(WebSocket)

### 5.1 统一连接
`WS /ws?token=<jwt>`
- 鉴权后建立连接;心跳保活;断线重连。
- 单端登录:被踢时服务端关闭旧连接(联动 M1)。

### 5.2 订阅主题
连接后发送订阅消息:
```json
{ "action": "subscribe", "topics": ["contest:55:leaderboard", "exp:88:progress"] }
```
- 主题命名:`<模块>:<资源id>:<频道>`。
- 服务端按订阅推送对应主题消息。

### 5.3 推送消息(各模块产生,经 Hub 推送)
| 主题 | 内容来源 |
| --- | --- |
| `contest:{id}:leaderboard` | M8 排行榜 |
| `contest:{id}:match` | M8 对局状态 |
| `sandbox:{id}:progress` | M2 沙箱进度 |
| `sim:{id}:state` | M4 仿真状态 |
| `alert:{tenant}` | M9 业务告警 |
| `notify:{account}` | 个人通知红点 |

### 5.4 内部推送接口 `[内部]`
`POST /push` `[内部]`
请求:`{ "topic": "contest:55:leaderboard", "payload": {...} }`
> 各模块经此接口把实时数据推到对应主题。
> 模块化单体内对应 `contracts.NotifyService.Push(ctx, NotifyPushRequest)`;M10 只转发 payload,不解释业务字段。

## 6. 典型时序

### 站内信
```
M6 作业截止临近 → POST /notify/send(type, receivers)
M10 → 渲染模板 → 写 notification → Redis 未读+1 → WS 推红点
学生 → GET /inbox/unread-count
M10 → 优先读 Redis;缓存 miss 时按 notification 未读状态重建并回写缓存
学生 → GET /inbox 查看
```

### 实时排行榜
```
M8 判题完成更新排名 → POST /notify/push(topic=contest:55:leaderboard)
M10 Hub → 推送给订阅该主题的所有在线选手
```
