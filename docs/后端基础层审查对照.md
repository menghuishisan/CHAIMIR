# 后端基础层审查对照

> 本文用于审查 `backend/internal/contracts`、`backend/internal/platform`、`backend/pkg` 三层基础能力时对照使用。它记录当前基础层职责、复用入口、模块接入状态和禁止事项；后续修改仍必须以 `AGENTS.md`、`docs/总-工程目录设计.md`、`docs/总-技术选型.md`、`docs/00-平台总体蓝图.md` 为第一规范，并用现状扫描确认是否漂移。

## 1. 三层职责边界

| 层 | 应放内容 | 禁止内容 | 当前判断 |
| --- | --- | --- | --- |
| `internal/contracts` | 模块间 interface、跨模块 DTO、事件载荷、角色常量 | 模块实现、基础设施客户端、业务流程 | 当前文件均为契约/DTO/事件/角色，11 个模块均有生产接入；角色守卫统一使用 `contracts.IdentityService`，不再保留 `platform/auth` 第二套角色接口。 |
| `internal/platform` | 仅供后端内部模块复用的基础设施与横切能力，如 DB、auth、eventbus、ws、storage、transfer、upload、netx、jsonx、timex | 模块私有业务字段、具体业务导入导出流程、模块表写入逻辑 | 总体职责正确；所有 platform 子包均有生产引用；需继续跟踪 `storage.Service` 与 `transfer.Manager` 后续导入导出入口接入。 |
| `pkg` | 可外部复用、不得依赖 `internal` 的通用库，如错误、响应、日志、加密、雪花 ID、链断言 | Chaimir 模块私有实现、依赖 `internal/*` 的代码 | 当前 `pkg` 未依赖 `internal`，所有 pkg 子包均有生产引用；`chainassert` 是链上断言通用判定，供 judge/contest 复用，暂可留在 `pkg`。 |

## 2. contracts 包职责

| 文件 | 职责 | 使用要求 |
| --- | --- | --- |
| `roles.go` | 平台稳定角色编码与角色匹配。 | 模块鉴权只使用角色字符串/编号转换，不自造角色映射。 |
| `events.go` | 反向通信事件主题和载荷。 | 下层通知上层只发事件，不直接依赖上层模块。 |
| `foundation_identity.go` | M1 身份、租户、审计只读契约。 | admin/其他模块读取身份信息必须走这些接口。 |
| `engine_sandbox.go`、`engine_judge.go`、`engine_sim.go`、`engine_content.go` | 第 1 层引擎对业务层暴露的跨模块能力。 | teaching/experiment/contest 不得 import 引擎模块实现。 |
| `business_teaching.go`、`business_experiment.go`、`business_contest.go` | 第 2 层业务对聚合层的只读契约。 | admin/grade 只读聚合，不跨写下层表。 |
| `aggregation_notify.go` | M10 通知与实时推送契约。 | 模块发通知或业务实时消息走 notify 契约，不自建通知表/WS 广播。 |

## 3. platform 包职责与接入状态

| 包 | 职责 | 当前接入状态 | 审查结论 |
| --- | --- | --- | --- |
| `audit` | 审计 Entry、actor_role 解析、detail JSON 序列化。 | 多数模块已通过 `audit.BuildEntry`/`audit.DetailString` 或模块 audit 包封装。 | 模块不得绕过 identity `audit_log` 自建审计。 |
| `auth` | JWT、服务间 HMAC、Gin 鉴权中间件、角色守卫。 | API 与组合根使用；角色守卫入参统一为 `contracts.IdentityService`。 | 使用成熟 `golang-jwt/jwt/v5`，符合第三方库优先；不得再定义第二套角色接口。 |
| `background` | 通用 ticker、panic recovery、停止语义。 | 组合根后台任务使用。 | 业务状态机不得放入该包。 |
| `config` | 启动期环境变量加载和 fail-fast 校验。 | 组合根使用。 | 后端读取的新配置必须同步 `deploy/config/chaimir.env` 与 `backend/.env.example`;仅供镜像/沙箱内服务使用的键只登记在 `deploy/config/chaimir.env` 和必要的密钥模板中。 |
| `db` | pgx 连接池、租户事务、受控特权事务。 | repo/store 构造边界使用。 | service 不得接收 `*db.DB` 或构造 repo。 |
| `eventbus` | NATS 封装、事件 decode、重试与死信语义。 | 事件发布/订阅使用。 | events 文件只订阅/解码/调 service，不直接写表。 |
| `httpx` | HTTP handler 绑定、响应、路径/查询参数解析。 | 多数 API 使用。 | API 层优先使用，减少手写响应和参数错误。 |
| `ids` | 雪花 ID 字符串边界处理。 | judge/sim/httpx/pgtypex 使用。 | 外部字符串 ID 解析统一使用。 |
| `jsonx` | JSON 序列化、强/弱读取、拒绝未知字段解码、JSON 标量和路径读取。 | JSONB/外部 JSON 已统一接入；仍允许 DTO/model 声明 `json.RawMessage`。 | service/audit/events/rules/api/repo/row_convert 不得直接散落 `json.Marshal`/`json.Unmarshal`/`json.NewDecoder`/`json.Valid` 等语义。 |
| `k8s` | client-go 客户端与 metrics client 封装。 | sandbox 装配和编排使用。 | 不在模块内新建第二套 K8s client。 |
| `netx` | 租户/用户可配置外部端点的 SSRF 防护、受控 HTTP client、解析后拨号。 | contest 漏洞源、identity SMS/SSO 使用。 | 所有外部 HTTP/LDAP 配置必须走该边界。 |
| `pagex` | 分页默认值和上限。 | identity/judge API/service 使用。 | 新分页能力优先使用。 |
| `pgtypex` | pgtype 可空值转换。 | repo_convert/row_convert 使用；本轮已清理 content/contest/experiment/judge/sandbox/sim/teaching 中重复的 text/int/timestamptz helper。 | 模块不得再定义同类 `textParam`、`int64Value`、`pgText`、`timeFromPG` 等转换。 |
| `redis` | go-redis 客户端封装。 | identity/notify 装配使用。 | 模块不直接创建 Redis client。 |
| `secretmap` | JSON 配置 map 中敏感字段加密、脱敏、还原。 | admin/contest 使用。 | 系统配置和外部源密钥字段必须复用。 |

| `tenant` | 租户身份上下文。 | 鉴权、service、审计使用。 | 不接受客户端传角色/租户覆盖服务端上下文。 |

| `ws` | WebSocket Hub、Origin 校验、订阅广播。 | notify/sandbox/judge/sim 使用。 | 不在模块内自建 Hub/Origin 策略。 |

## 4. pkg 包职责与第三方库使用

| 包 | 职责 | 第三方/标准库复用判断 |
| --- | --- | --- |
| `apperr` | 统一错误码和用户向错误文案。 | 自研合理，因为错误码分段和用户文案是项目契约。 |
| `response` | HTTP 响应信封、trace_id、错误分层暴露。 | 自研合理，因为响应结构是平台 API 契约。 |
| `logging` | slog 配置、上下文属性、错误脱敏。 | 基于标准库 `log/slog`，符合优先复用。 |

## 5. 已确认风险与后续修复原则

1. `platform/transfer` 已落地通用任务表和下载授权 API。修复其他导入导出入口时不能把 M1/M6/M9/M11 的具体业务解析搬入 `platform/transfer`;只应复用其任务状态、重试、产物对象引用和下载授权语义。M1 `import_preview/import_batch`、M6 `course_grade`、M11 `transcript_record`、M9 `backup_record` 是业务记录,不是 `transfer_task` 的冗余替代或重复实现。
2. `platform/storage.Service` 当前接入范围偏窄。修复时只把上传规划、病毒扫描、短时下载授权、防直链这类横切能力接入 `Service`；内部引擎在已完成业务鉴权后读取对象，可继续使用底层 `Storage`。
3. 出站网络必须用 `platform/netx`。允许 SSO 的学校内网 LDAPS 使用 `PrivateResolvedURL`，公网漏洞源/SMS/CAS 必须用公网校验和受限 HTTP client。
4. JSONB/外部 JSON 语义优先使用 `platform/jsonx`。DTO/model 中为了声明字段类型使用 `encoding/json.RawMessage` 可以保留；service/audit/events/rules/api/repo/row_convert 中不得手写第二套 JSON 语义。
5. 上传/归档/病毒扫描必须用 `platform/upload` 和成熟库；CSV 用标准库，XLSX 用 `excelize`，K8s/MinIO/NATS/Redis/JWT/LDAP 均使用成熟第三方库。
6. 基础层不能为测试让步。新增接口或抽象必须有生产接入理由，不能只是为了 fake/mock。
7. 清理未用代码前必须区分“完全无引用符号”和“文档要求但尚未接入的基础能力”。前者可删除，后者应补完整或同步调整文档。
8. 当前已加模块边界测试守护：模块不得直接 import 其他模块；service/repo 文件职责命名受控；service/audit/events/api/repo/row_convert 走 `jsonx`；rules 层不得 import `encoding/json` 或直接调用 JSON helper；模块和 `cmd/server` 当前时间走 `timex`；模块出站 HTTP 走 `netx`；service 层 HMAC 走 `pkg/crypto`。
9. 本轮已新增并通过的基础层复用守护：模块不得重复定义 pgtype 可空值转换；service 层不得直接使用 `crypto/rand` 自建随机凭证；service 层不得手拼 `minio://` 对象引用；`platform/auth` 不得定义 `RoleChecker` 第二套角色接口；ClamAV 扫描器不得静默吞关闭错误。
10. `teaching` 原课程邀请码在 CSPRNG 失败时退化为时间码，已改为 `pkg/crypto.RandomToken` 并显式返回错误；生产路径不得出现随机失败后的可预测降级。
11. 符号/包引用清理结论：当前 `internal/platform` 和 `pkg` 各子包都有生产引用；contracts 各跨模块契约均有生产接入,仅 `RolePlatformAdmin` 属于 `RoleCode`/`RoleNumber` 内部常量实现细节。
