## 1. 成绩审核工作流

```mermaid
stateDiagram-v2
    [*] --> 待审: 教师提交
    待审 --> 已通过: 管理员通过(锁定)
    待审 --> 已驳回: 管理员驳回
    已驳回 --> 待审: 教师修改重提
    已通过 --> 待审: 管理员解锁回退
    已通过 --> [*]
```

## 2. GPA 聚合(只读 M6)

```mermaid
flowchart TD
    A[课程成绩审核通过] --> B[M11 读 M6 课程成绩+学分]
    B --> C[按 grade_level_config 映射绩点]
    C --> D[Σ绩点×学分 / Σ学分]
    D --> E[更新 student_semester_grade]
    E --> F[累计 GPA 重算]
    F --> G[触发学业预警扫描]
    Note over B: 只读 M6,不写 M6
```

---

## 3. 申诉改分(跨模块,经解锁→M6 改分事件→重审→重锁完整回路)

```mermaid
sequenceDiagram
    participant S as 学生
    participant M11 as 成绩中心
    participant M6 as 教学
    S->>M11: 提交申诉(时效内)
    M11->>M11: 记 grade_appeal(待处理)
    Note over M11: 教师/管理员受理
    M11->>M11: 解锁该课程审核状态(is_locked=false)
    M11-->>M6: 通过通知/审计提示教师处理改分
    M6->>M6: 教师执行单课程改分(M6 写 course_grade)
    M6-->>M11: 发布 teaching.grade.updated 事件
    M11->>M11: 重算 GPA + 复核预警
    M11->>M11: 重新审核 + 重锁(is_locked=true)
```

> **A6/C4 修复**:已锁定成绩不被旁路修改。申诉受理走完整回路:解锁 M11 审核状态 → M6 教师成绩入口改分 → M6 发布成绩变更事件 → M11 重算 GPA → 重审重锁。M11 仅保存自有聚合与审核状态,单课程成绩修改归属 M6。

## 4. 学业预警

```mermaid
flowchart LR
    A[周期扫描/GPA变更] --> B{满足预警规则?}
    B -->|挂科数超限| C[生成挂科预警]
```

## 5. 成绩单生成

```mermaid
flowchart TD
```

## 6. 申诉状态机

```mermaid
stateDiagram-v2
    [*] --> 待处理: 学生提交
    待处理 --> 受理改分: 受理(解锁→等待M6改分事件)
    待处理 --> 已驳回: 驳回
    受理改分 --> 已完成: 收到M6改分事件→重算GPA→重审重锁
    受理改分 --> 待处理: 管理员撤回受理→回退可重试
    已完成 --> [*]
    已驳回 --> [*]
```
