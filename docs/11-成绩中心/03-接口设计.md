# M11 成绩中心 — 接口设计

> RESTful。鉴权沿用 M1 JWT。聚合只读 M6;M11 写自有审核/预警/成绩单元数据。
> 最后更新:2026-05-29

---

## 1. 通用约定

- Base:`/api/v1/grade-center`
- 前端和 API client 只能使用 `/grade-center/*` 当前路径;不得保留旧 `/grade/*`、`/grade/applications/*`、`/grade/transcript/*` 或告警 `acknowledge/handle` 旧接口。
- 统一响应体同 M1。错误码段:`B0xxx`(审核/GPA/申诉/预警/成绩锁事件同步)。`B0010` 专用于成绩锁定状态事件无法写入或发布,不得复用审核状态错误码。
- 带 `page/size` 的列表接口必须返回统一分页结构 `{list,total,page,size}`,不得退回裸数组。

## 2. 等级映射与学期 `[学校管理员]`

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET/POST/PUT | `/level-configs[/{id}]` | 等级映射配置(分数段→等级→绩点) |
| GET/POST | `/semesters[/{id}]` | 学期管理 |

---

## 3. 成绩审核 `[教师/学校管理员]`

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/reviews` `[教师]` | 提交课程成绩待审(引用 M6 课程) |
| GET | `/reviews?status=&page=&size=` `[管理员]` | 待审分页列表,响应 data 为 `{list,total,page,size}` |
| POST | `/reviews/{id}/approve` `[管理员]` | 通过 + 锁定审核状态 + 发布事件驱动 M6 同步写保护投影 → 触发 GPA 聚合 |
| POST | `/reviews/{id}/reject` `[管理员]` | 驳回 |
| POST | `/reviews/{id}/unlock` `[管理员]` | 解锁回退 + 发布事件驱动 M6 解除写保护投影 |

---

## 4. GPA 与聚合 `[学生/教师/管理员]`

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/students/{id}/grades?semester=` | 学生学期成绩(聚合读 M6 明细 + GPA) |
| GET | `/students/{id}/gpa` | 学期/累计 GPA |
| POST | `/students/{id}/recompute` `[内部/管理员]` | 重算 GPA(改分/申诉后) |

> 明细课程成绩实时读 M6;M11 不复制成绩。学业概览使用 Redis 缓存,TTL 由 `GRADE_SUMMARY_CACHE_TTL_SECONDS` 配置;成绩变更、审核通过/驳回/解锁、申诉创建/受理/驳回/完成和手动重算必须主动失效学生全量与对应学期缓存。

---

## 5. 申诉 `[学生/教师]`

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/appeals` `[学生]` | 提交申诉(时效内) |
| GET | `/appeals?status=&page=&size=` `[教师/管理员]` | 申诉分页列表,响应 data 为 `{list,total,page,size}` |
| POST | `/appeals/{id}/accept` | 受理申诉并解锁审核状态,同时发布事件驱动 M6 解除写保护投影;等待 M6 改分事件触发重算 |
| POST | `/appeals/{id}/reject` | 驳回 |

同一学生同一课程存在待处理或受理改分中的申诉时,再次提交会被拒绝;受理/驳回只能从待处理状态进入,完成只能由 M6 成绩变更事件驱动从受理改分中进入。

## 6. 学业预警 `[管理员/学生]`

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET/PUT | `/warning-rules` `[管理员]` | 预警规则(挂科数/GPA阈值) |
| GET | `/warnings?student_id=&page=&size=` | 预警分页列表,响应 data 为 `{list,total,page,size}` |
| POST | `/warnings/{id}/ack` `[学生]` | 学生确认本人学业预警已知悉 |
| POST | `/warnings/scan` `[内部]` | 周期扫描生成预警(经 M10 通知) |

## 7. 成绩单 `[学生/管理员]`

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/transcripts` | 生成成绩单 PDF(学期/全部) |
| GET | `/transcripts/{id}` | 鉴权后返回统一文件服务短时下载授权 JSON,不由模块直接流式输出 PDF,不暴露对象存储直链 |

## 8. 典型时序(成绩审核 → GPA)

```
教师 → POST /reviews(提交课程成绩)
管理员 → POST /reviews/{id}/approve(通过+锁定)
M11 → 读 M6 该课程成绩 + 学分 → 按等级映射算绩点
    → 更新 student_semester_grade(GPA)
    → 触发学业预警扫描 → 经 M10 通知
```

## 9. 申诉改分时序(解锁→M6 改分事件→重审重锁完整回路)

```
学生 → POST /appeals
教师/管理员 → POST /appeals/{id}/accept
M11 → 解锁该课程审核状态(is_locked=false),发布事件驱动 M6 解除写保护投影,申诉进入"受理改分"
教师 → M6 PATCH /courses/{id}/grades/{sid} 执行单课程改分
M6 → 发布 teaching.grade.updated 事件
M11 → recompute 重算 GPA → 复核预警 → 重审重锁 → 申诉"已完成"
若 M6 未改分或事件未到达:申诉保持"受理改分"可继续处理,成绩审核状态与 M6 写保护投影保持解锁
```

M11 发布/解除锁事件必须先写 `grade_lock_outbox`,再由即时 drain 或后台任务发布到 `grade.review.lock_changed`;发布失败保留 outbox 记录重试,不得只在审批事务提交后直接调用 NATS。
受理申诉时,`grade_appeal` 状态更新、`grade_review` 解锁和 `grade_lock_outbox` 写入必须在同一租户事务内完成;任一步失败都不得留下“申诉已受理但课程未解锁”的半状态。
