# 后端路由安全暴露面审计与整改清单

> 日期:2026-06-28
> 状态:整改追踪中。2026-06-29 已开始统一修复后端、OpenAPI、K8s 与部署配置；所有结论以回归证据为准。

---

## 1. 结论

`/api/v1` 不是问题本身。`docs/总-API接口总览.md` 明确规定基础路径为 `/api/v1`,各模块在其下挂子前缀；因此不能把前端能看到 `/api/v1/...` 简单等同于路由泄露。

当前后端已经具备主要应用层保护:

- 用户接口使用 JWT access token 与角色守卫。
- 内部服务接口使用 HMAC-SHA256 服务签名,签名绑定 method、path、tenant_id、source_ref、timestamp 与 trace_id。
- sandbox、sim 等高风险模块在 service 层继续校验 owner_account_id 或 source_ref,不是只靠前端路径。
- M4 公开分享入口按分享码状态/过期时间校验,并过滤答案、flag、token、密钥和 `_` 开头内部字段。
- K8s NetworkPolicy 已有 default deny 基线,密钥文件未纳入镜像上下文和 Git 跟踪。

2026-06-29 已完成第一轮整改:纯服务内部 HTTP 路由统一迁入 `/api/v1/{module}/internal/...`,Ingress 对公网 `/api/v1/*/internal` 返回 404；`/api/readyz` 已迁到内部探针 `/-/readyz`；Gin 可信代理改为环境变量配置。仍未达到“全部完成”的标准,因为还需要重新构建镜像、部署回归,并补齐自动化路由审计、RLS 角色验收、真实 K8s 闭环验收。

## 2. 文档基线

本次判断依据如下:

- `docs/总-API接口总览.md`
  - 基础路径:`/api/v1`。
  - `[用户]`:前端调用,JWT + 角色鉴权。
  - `[内部]`:模块间调用,不对前端开放；如 HTTP 暴露,走内网鉴权并携带调用方模块、`source_ref` 与 trace 上下文。
  - 内部 HTTP 鉴权统一使用 HMAC-SHA256,并校验时间窗口。
- `docs/总-部署架构设计.md`
  - Ingress/网关通过 `/api` 转发到 backend。
  - 动态沙箱、系统命名空间按 NetworkPolicy default deny 隔离。
  - 常规后端请求必须走非属主应用角色 `chaimir_app`,以确保 RLS 生效。
- `docs/02-沙箱引擎/06-安全设计.md`
  - 链上能力等 `[内部]` 接口如果 HTTP 暴露,只能服务间鉴权调用。
  - 前端永不直接接触 kubeconfig,终端/文件/工具入口必须校验沙箱归属。
- `docs/04-仿真可视化引擎/05-接口设计.md`、`04-数据模型.md`、`07-安全设计.md`
  - M4 内部服务接口必须使用服务签名并校验 `source_ref`。
  - 公开分享剧本只返回复现所需公开参数,敏感字段必须过滤。
  - `sim_share` 只保存不可推导 code 与会话指针,不复制剧本正文。

## 3. 实现排查摘要

### 3.1 已符合预期的部分

- `backend/internal/platform/auth/middleware.go`
  - `ServiceMiddleware()` 调用服务签名校验。
  - `ServiceOrTenantAnyRoleMiddleware()` 支持“内部服务 HMAC 或指定租户角色”双入口。
  - `BrowserAccessMiddleware()` 支持浏览器工具入口 Bearer/query token/path-limited cookie。
  - `injectServiceIdentity` 校验 `X-Chaimir-Service`、`X-Chaimir-Tenant-Id`、`X-Chaimir-Source-Ref`、`X-Chaimir-Timestamp`、`X-Chaimir-Signature`、trace_id、时间窗口与 HMAC。
- `backend/internal/modules/sandbox/api.go`
  - 生命周期内部接口挂 `ServiceMiddleware()`。
  - 用户文件、终端、工具代理、链操作继续按租户角色与 owner/source_ref 校验。
- `backend/internal/modules/sim/api.go`、`service_session.go`、`rules.go`
  - `GET /api/v1/sim/shared/:code` 是文档定义的公开分享入口。
  - 分享码读取校验 status/expire_at,再回到所属 tenant RLS 上下文读取会话和动作。
  - 公开 replay 会过滤敏感字段和内部字段。
- `backend/db/migrations/*.up.sql`
  - 多数租户表已 `ENABLE ROW LEVEL SECURITY`,并配置 `USING`/`WITH CHECK` 策略。
- `deploy/base/networkpolicy/static-namespaces.yaml`
  - 系统和数据命名空间存在 default deny 与定向放行。
- `.gitignore`、`.dockerignore`
  - `.env`、`secret.env`、`deploy/config/*.env` 等敏感文件已排除。

### 3.2 暂不列为问题

| 项 | 判断 |
| --- | --- |
| `/api/v1` 前缀 | 这是总 API 文档规定的基础路径,不是暴露漏洞。 |
| 前端能看到用户接口路径 | REST API 本来需要前端调用；安全边界应是后端鉴权、授权、RLS 和服务端归属校验,不是隐藏 URL。 |
| M4 `/api/v1/sim/shared/:code` | 文档定义为公开分享入口；实现已有不可推导 code、过期/状态校验和公开字段过滤。后续只做验收覆盖,不按缺陷处理。 |
| sandbox 工具 query token | 总 API 文档允许 sandbox 交互入口在浏览器限制下使用一次性 query token,并换成路径受限 HttpOnly cookie。 |
| `/api` Ingress 总入口 | 部署文档本身要求 `/api` 到 backend；问题不在 `/api`,而在 `[内部]` 子接口缺少网关层隔离。 |
