# M8 竞赛 — 需求规格

> 模块定位:竞赛业务编排层。复用 M2 沙箱、M3 评测、M5 题库,组织解题赛与对抗赛。
> 核心理念:业务薄、引擎厚;竞赛成绩独立(不计 GPA)。
> 最后更新:2026-05-29

---

## 1. 模块目标

- 组织两类赛制:**解题赛**(含理论题型)、**对抗赛**(异步撮合天梯)。
- 报名组队、实时排行榜、行为级防作弊、战报回放、积分奖励。
- **真实漏洞驱动出题**:依赖外部漏洞源,导入即固化为自包含题目。
- 复用引擎:题目 M5、判题 M3、选手环境 M2。

## 2. 职责边界

| M8 负责 | M8 不负责(归属) |
| --- | --- |
| 赛事管理、赛程、报名组队 | 题目内容存储(→ M5) |
| 解题赛/对抗赛赛制逻辑 | 判题执行(→ M3) |
| 对抗撮合、天梯、对局回放 | 选手环境起停(→ M2) |
| 实时排行榜、积分奖励 | 成绩进 GPA(竞赛成绩独立) |
| 行为级防作弊 | 提交级查重能力(→ M3,M8 做行为级+判定) |
| 真实漏洞源接入与转化 | 题目最终存储(→ M5,转化后入库) |

## 3. 赛制(可扩展类型,非硬编码枚举)

> **可扩展原则**(呼应平台插件化):赛制与对局规则不写死为固定枚举,而是**可注册的类型**——与 M2 链运行时、M3 判题器、M4 可视化模式同一扩展哲学。首版内置"解题赛/对抗赛"两种赛制 + "攻防型/博弈型"两种对局规则,后续新增赛制(如积分闯关赛、组队接力赛)或新对局规则(如多方混战、淘汰锦标赛)= 注册一个新类型,核心代码不改。
> 实现上:`contest_type` 与 `battle_rule` 为**类型注册表**(类似 runtime 表),每种类型声明其报名方式、计分逻辑、撮合/判定规则的处理器。下文枚举值是"首版内置类型",非封闭集合。

### 3.1 解题赛(含理论题型)
- 选手独立解题,自动判定(M3),实时排行榜。
- **题型混合**:
  - 实操题:写合约、攻破漏洞、跑 flag、**真实漏洞复现**。
  - 理论题:选择/判断/填空/简答(原"理论赛"整合进来,不单列)。
- 一场解题赛可混合出题。

### 3.2 对抗赛(异步撮合天梯)
- 选手提交合约/脚本,系统后台撮合跑对局,产出结果 + 可回放记录。
- **天梯/循环赛**:提交即进对战池,持续两两撮合(ELO 天梯或循环赛),更新排名;可改进合约重新提交。
- **对局规则**(题目级配置,非赛制分类):每道对抗题配置其对局如何进行 ——
  - 攻防型规则:守方交防御合约,攻方交攻击脚本,撮合判定攻破与否。
  - 博弈型规则:双方交策略合约,链上博弈多轮按收益排名。
  - 对局规则随题目走,出题时配置。**赛制层面只有"解题赛 / 对抗赛"两种**,攻防/博弈不是独立赛制,只是对抗题的对局规则。

## 5. 非功能需求

- **业务薄**:判题/环境/题库全复用引擎。
- **答题零外部依赖**:真实漏洞题导入即固化,外部源仅出题阶段在线。
- **可复现**:对局确定可回放;forked 题锁历史区块 + 缓存状态。
- **高并发**:竞赛瞬时大量提交,判题走 M3 异步优先级队列。
- **多租户**:竞赛数据 tenant_id 隔离;跨校组队为受控例外。

## 6. 明确不做(边界)

- 不做实时在线对战(对抗赛用异步撮合)。
- 不存题目正本/不判题/不起环境(引擎)。
- 竞赛成绩不计入 GPA。

## 7. 技术取向

Go 后端 + 撮合调度器;对局复用 M2 沙箱;排行榜实时推送对接 M10;漏洞源适配器化。最终技术细节由《技术选型》统一收口。
