# M8 竞赛 — 数据模型

> PostgreSQL,租户级(tenant_id + RLS);跨校组队为受控例外。建模遵循:消灭数据冗余、表合并有度。
> 最后更新:2026-05-29

---

## 1. 表清单总览

| 表名 | 用途 |
| --- | --- |
| `contest` | 竞赛(赛制/赛程/规则) |
| `contest_problem` | 竞赛题目引用(指向 M5) |
| `team` | 参赛队伍(个人=单人队) |
| `team_member` | 队员 |
| `solve_submission` | 解题赛提交 |
| `battle_entry` | 对抗赛参战物(合约/脚本/策略) |
| `battle_match` | 对抗对局(撮合结果 + 回放引用) |
| `ladder_rank` | 天梯/排行榜积分 |
| `contest_result_snapshot` | 竞赛成绩快照(归档存档) |
| `cheat_record` | 作弊判定记录 |
| `vuln_source` | 真实漏洞外部源配置 |
| `vuln_problem` | 漏洞题转化草稿/预验证记录 |

> 消灭冗余/合并说明:
> - **题目正本在 M5**,`contest_problem` 只存引用 + 锁版本 + 竞赛内分值/配置。
> - **个人赛复用 team**(单人队),不单设个人参赛表。
> - **对局回放不存全量状态**:`battle_match` 存交易序列引用(可重放),非状态快照。
> - **解题/对抗提交分两表**:语义与字段差异大(解题判 flag/断言,对抗是参战物进对战池),分开清晰。
> - **漏洞源拉取的题目固化进 M5**,M8 只存源配置 `vuln_source`,不存漏洞题正本(在 M5)。

## 7. 多租户与索引

- 全部表 `tenant_id` + RLS(同 M1)。跨校组队:`team_member.member_tenant_id` 标识他校队员,访问竞赛数据经组织方授权(受控例外)。
- 关键索引:
  - `contest(tenant_id, status)`、`contest_problem(contest_id)`
  - `team(contest_id)`、`team_member(team_id, account_id)` UNIQUE
  - `solve_submission(contest_id, team_id, problem_id)`
  - `solve_submission(tenant_id, source_ref)` UNIQUE
  - `battle_entry(contest_id, team_id, is_active)`、`battle_match(contest_id)`
  - `ladder_rank(contest_id, team_id)` UNIQUE、`ladder_rank(contest_id, rank)`
  - `vuln_problem(source_id)`、`vuln_problem(status, prevalidate_status)`
