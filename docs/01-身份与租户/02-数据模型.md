# M1 身份与租户 — 数据模型

> 数据库:PostgreSQL。所有租户内业务表带 `tenant_id`,启用行级安全(RLS)强制隔离。
> 最后更新:2026-05-29

---

## 1. 表清单总览

| 表名 | 用途 | 是否带 tenant_id |
| --- | --- | --- |
| `platform_admin` | 平台管理员(仅 SaaS) | 否(全局) |
| `tenant` | 租户(学校),含配置 | 否(本身即租户) |
| `tenant_provision_outbox` | 新租户跨模块初始化事件可靠投递 | 是(平台流程表,不作为租户业务数据开放) |
| `tenant_application` | 学校入驻申请 | 否(审核前无租户) |
| `department` | 院系 | 是 |
| `major` | 专业 | 是 |
| `class` | 班级 | 是 |
| `account` | 账号(自然人) | 是 |
| `account_role` | 账号角色关联(一账号可多角色) | 是 |
| `account_profile` | 账号扩展信息(教师/学生统一) | 是 |
| `auth_session` | 会话(Refresh Token) | 是 |
| `sms_code` | 短信验证码 | 是 |
| `activation_code` | 账号激活码 | 是 |
| `sso_config` | SSO/LDAP 配置 | 是 |
| `import_preview` | 导入预览暂存 | 是 |
| `import_batch` | 导入批次记录 | 是 |
| `audit_log` | 审计日志 | 是 |

> 建模原则:消灭数据冗余(同一份数据只存一处);表合并有度(配置类一对一并入主表、同类实体合并 + 类型区分,多对多关系独立成表)。
> - `tenant_config` 已并入 `tenant`(配置项少,一对一)。
> - `teacher_profile`/`student_profile` 已合并为 `account_profile`(字段差异小,类型区分)。
> - `account_role` 独立(一账号可同时持教师+管理员角色,不可用单字段表达)。
> - 院系/专业/班级 保持独立三表(层级语义清晰,外键约束简单)。
> - `tenant_provision_outbox` 只保存租户 ID、部署形态、追踪编号和投递状态，不复制租户配置；`tenant_id` 唯一约束保证每个租户只有一个初始化事件正本。

## 2. 通用约定

- 主键:`id BIGINT`(雪花 ID)或 `UUID`,M1 统一用 `BIGINT` 雪花 ID。
- 时间戳:`created_at`、`updated_at`(`TIMESTAMPTZ`,UTC 存储)。
- 软删除:`deleted_at TIMESTAMPTZ NULL`,归档/注销用状态字段,不物理删除。
- 租户隔离:除全局表外,均含 `tenant_id BIGINT NOT NULL`,建索引并启用 RLS。
- 敏感字段(手机号):加密列 `phone_enc BYTEA` + 查询用 `phone_hash`(HMAC,唯一索引)。

## 5. 账号与角色

### 5.1 account — 账号(自然人)

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | BIGINT PK | |
| tenant_id | BIGINT | |
| phone_enc | BYTEA | 手机号密文(必填) |
| phone_hash | VARCHAR(64) | HMAC,唯一查询索引 |
| password_hash | VARCHAR(255) NULL | argon2id;SSO 账号可空 |
| name | VARCHAR(64) | 姓名 |
| base_identity | SMALLINT | 1学生 / 2教师(互斥,入职即定) |
| status | SMALLINT | 1待激活 / 2正常 / 3停用 / 4归档 / 5注销 |
| must_change_pwd | BOOLEAN | 首登强制改密标记 |
| pwd_failed_count | SMALLINT | 连续失败次数 |
| locked_until | TIMESTAMPTZ NULL | 锁定截止 |
| activated_at | TIMESTAMPTZ NULL | 激活时间 |
| created_at / updated_at | TIMESTAMPTZ | |

唯一约束:`(tenant_id, phone_hash)` 联合唯一。

> 一号多校:手机号在不同租户可各有一个账号,故采用 `(tenant_id, phone_hash)` 联合唯一而非全局唯一。登录时若一个手机号匹配多个租户账号,返回租户列表让用户选择后再发 Token。

### 5.2 account_role — 账号角色

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | BIGINT PK | |
| tenant_id | BIGINT | |
| account_id | BIGINT | |
| role | SMALLINT | 1平台管理员 / 2学校管理员 / 3教师 / 4学生 |

> 学校管理员 = 教师账号附加 role=2 记录。一个教师账号可同时持有 role=3(教师)与 role=2(管理员)。学生账号仅 role=4。平台管理员不在此表(独立 platform_admin)。
> **与 base_identity 的口径(避免误读为冗余)**:`account.base_identity` 是不可变的学籍身份(学生/教师),`account_role` 是 RBAC 鉴权用的角色集合。基础师生角色与 base_identity 看似重复,但**有意保留**——让鉴权层只查 account_role 一处即可拿到全部角色(含管理员),无需在两表间拼接判断。真正"附加"的是 role=2 管理员。实现上 base_identity 为权威源,account_role 的师生行随之维护一致。

### 5.3 account_profile — 账号扩展信息(教师/学生统一)

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| account_id | BIGINT PK | 与 account 一对一 |
| tenant_id | BIGINT | |
| no | VARCHAR(64) | 学号 / 工号(按 base_identity 区分语义,租户内唯一,不可变) |
| org_id | BIGINT | 归属组织:学生=班级 class.id,教师=院系 department.id |
| enrollment_year | SMALLINT NULL | 学生入学年份;教师为空 |
| title | VARCHAR(64) NULL | 教师职称;学生为空 |

> 教师与学生字段差异小,合并为一张表,用 `account.base_identity` 区分语义,避免两张高度相似的表。
> 唯一约束:`account_profile(tenant_id, no)`。
> `org_id` 含义随身份变化:学生指向 `class`,教师指向 `department`。归属关系只存这一处,院系/专业等上层组织通过外键链路推导,不冗余存储。
> 首个学校管理员例外:入驻审核/私有化初始化创建租户时,学校真实院系尚未建立,平台不得臆造默认院系。
> 因此首个学校管理员账号可暂缺 `account_profile`;登录后由其先维护真实组织架构,再补齐自己的工号、
> 院系与职称。学校管理员后续创建/导入的教师、学生仍必须写入 `account_profile`,且教师挂真实院系、
> 学生挂真实班级。

## 8. 多租户隔离(RLS)

- 所有带 `tenant_id` 的表启用 PostgreSQL 行级安全策略(RLS)。
- 应用连接时设置会话变量 `SET app.tenant_id = <当前租户>`,RLS 策略 `USING (tenant_id = current_setting('app.tenant_id')::BIGINT)` 强制过滤。
- 平台管理员操作走独立连接角色,绕过 RLS(仅限平台级表与审核流程)。
- 私有化部署:`tenant_id` 恒为固定值,RLS 仍启用但恒真,逻辑零差异。

## 9. 关键索引

- `account(tenant_id, phone_hash)` UNIQUE
- `account_profile(tenant_id, no)` UNIQUE
- `auth_session(account_id, status)`、`auth_session(refresh_token_hash)`
- `audit_log(tenant_id, created_at)`、`audit_log(actor_id, created_at)`
- `tenant(code)` UNIQUE
