# M2 沙箱引擎 — 业务流程与状态机

> 用 Mermaid 描述沙箱启动、生命周期、回收、适配器接入流程。
> 最后更新:2026-06-04

---

## 1. 分阶段启动流程

```mermaid
flowchart TD
    A[调用方请求创建沙箱] --> B[校验租户配额]
    B -->|超配额| Z[排队/拒绝+提示]
    B -->|通过| C[分配 Namespace + NetworkPolicy]
    C --> D[调度 Pod 镜像已预拉取]
    D --> E[加载固化创世链状态]
    E --> F{节点健康检查}
    F -->|失败| ERR[置 error 状态]
    F -->|通过| G[phase=2 环境就绪]
    G --> H[通知前端:可进入]
    G --> I[后台异步:个性化初始化]
    I --> J[部署初始合约/装依赖/注入数据]
    J --> K[phase=4 完全就绪,按钮解灰]
    I -->|失败| E2[记录初始化失败,保持 phase=2/3]
```

阶段一(A→G,目标 ≤5s)完成即让学生进入;阶段二(I→K)后台补齐。
阶段一任一步失败必须进入 `error` 并触发回收;阶段二失败不得伪装成完全就绪,只能保留可进入状态并禁用依赖个性化初始化的能力。

## 2. 沙箱状态机

```mermaid
stateDiagram-v2
    [*] --> creating: 创建请求
    creating --> ready: 环境就绪(phase2)
    ready --> running: 个性化完成(phase4)/学生操作
    ready --> recycling: 就绪后超时未操作
    running --> idle: 无操作计时
    idle --> running: 学生恢复操作
    idle --> recycling: 空闲超时
    running --> recycling: 主动销毁/最长生命周期到/来源级联
    creating --> error: 启动失败
    running --> error: 运行异常
    error --> recycling: 清理
    recycling --> destroyed: 资源释放(代码已持久化)
    destroyed --> [*]
```

- **ready 态超时回收**(B6 修复):就绪后长时间未操作(如 10min)直接回收,防止就绪沙箱僵尸占用。
- **回收事件通知来源**(A4 修复):沙箱进入 `recycling`(任何触发:空闲/最长生命周期/就绪超时)时,**发布 `sandbox.recycled` 事件**(经事件总线,按 `source_ref` 标识),来源方(experiment/contest)订阅后同步更新自身实例状态,避免上游仍持 `sandbox_ref` 而沙箱已销毁的悬挂。上游恢复(resume)前应探活,已销毁则触发重建(代码已持久化 + 重放部署脚本)。用事件而非反向接口调用,避免 sandbox 反向依赖业务层(见工程目录设计 §3.1.1)。

- 进入 `recycling` 前确保学生代码已持久化到 MinIO。
- `keep_alive=true` 的沙箱:`idle` 不触发空闲回收,但仍受 `keep_alive_until`、`expire_at`、租户并发与资源配额约束。

## 4. 回收流程(三重保证)

```mermaid
flowchart LR
    subgraph 回收触发
    A1[空闲超时 30min]
    A2[最长生命周期到期]
    A3[来源结束级联 source_ref]
    end
    A1 --> R[回收调度器]
    A2 --> R
    A3 --> R
    R --> S[锁定 status=recycling]
    S --> P[持久化代码到 MinIO]
    P --> V{snapshot_enabled?}
    V -->|是| VS[创建 VolumeSnapshot 并写 snapshot_ref]
    V -->|否| D[删除 Namespace 释放资源]
    VS --> P[缩容工作负载并保留快照 Namespace 至 snapshot_expire_at]
    P --> X
    D --> X
    D --> X[写 status=destroyed]
    X --> L[记录 sandbox_event + audit_log]
    L --> E[发布 sandbox.recycled]
```

回收失败处理:
- 持久化失败:保留 Namespace,写 `error`,等待重试或人工处理。
- 快照成功:由于 CSI `VolumeSnapshot` 是 namespaced 资源,不得立即删除所在 Namespace;必须先缩容/删除计算工作负载释放 CPU/内存,保留 PVC/VolumeSnapshot/Namespace 至 `snapshot_expire_at`,由过期清理任务删除。
- 快照失败:若调用方要求保留运行态,保留 Namespace 并写 `error`;若只是普通回收且未要求快照,按默认重建策略继续删除。
- K8s 删除失败:保持 `recycling`,由调度器重试,不得写 `destroyed`。

## 5. 运行时接入即测流程

```mermaid
flowchart TD
    A[平台管理员注册运行时] --> B[提交 L1 清单 + L2实现/L3插件]
    B --> C[status=接入中]
    C --> D[触发自检套件]
    D --> E[起链]
    E --> F[部署样例合约]
    F --> G[查询验证]
    G --> H[重置链]
    H --> I{全部通过?}
    I -->|是| J[selftest=通过, status=可用]
    I -->|否| K[selftest=失败, 记录明细, status保持接入中]
```

只有自检全绿的运行时才对调用方可用,杜绝"宣称支持但未验证"。

## 6. 镜像预拉取流程

```mermaid
flowchart TD
    A[平台管理员触发预拉取] --> B[校验镜像来源/签名/扫描状态]
    B -->|失败| F[prepull_status=失败]
    B -->|通过| C[创建或更新 DaemonSet]
    C --> D[轮询目标节点 Ready]
    D -->|全部节点 Ready| S[prepulled=true, prepull_status=已完成]
    D -->|拉取失败/超时| F
    F --> L[记录 prepull_detail + audit_log]
    S --> L
```

预拉取完成以 Kubernetes 节点实际 Ready 状态为准,不能以接口调用成功或 DB 更新作为成功。
