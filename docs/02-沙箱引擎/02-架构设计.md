# M2 沙箱引擎 — 架构设计

> 本文是 M2 的核心,详述三大机制:分层启动优化、运行时三层适配器、分层持久化,以及工具接入协议与生命周期。
> 这些设计针对旧版 LensChain 的已知问题(无温池冷启动慢、全量快照炸、适配器宣称多实现少)做了优化。
> 最后更新:2026-06-04

---

## 1. 总体架构

```
┌─────────────────────────────────────────────────────────┐
│  调用方:M7 实验 / M8 竞赛 / M3 评测                          │
│  调用:创建沙箱(链类型, 工具集, 初始代码, 来源标识)            │
└───────────────────────────┬─────────────────────────────┘
                            │ (统一 API)
┌───────────────────────────▼─────────────────────────────┐
│  M2 控制面(Go,client-go / operator)                      │
│  ┌──────────┬──────────┬──────────┬──────────────────┐  │
│  │ 运行时    │ 沙箱      │ 工具接入  │ 持久化/回收        │  │
│  │ 适配器    │ 编排器    │ 网关      │ 调度器            │  │
│  └────┬─────┴────┬─────┴────┬─────┴────────┬─────────┘  │
└───────┼──────────┼──────────┼──────────────┼────────────┘
        │          │          │              │
┌───────▼──────────▼──────────▼──────────────▼────────────┐
│  Kubernetes 数据面(SaaS:K8s / 私有化:k3s)               │
│  每沙箱 = 独立 Namespace + Pod(s) + NetworkPolicy        │
│  Harbor 镜像仓库(DaemonSet 预拉取) + MinIO(代码持久化)   │
└─────────────────────────────────────────────────────────┘
```

**四个控制面组件:**
- **运行时适配器**:管理"什么链",三层粒度(见 §3)。
- **沙箱编排器**:创建/管理 K8s 资源,执行分层启动(见 §2)。
- **工具接入网关**:终端代理、Web 工具反向代理(见 §4)。
- **持久化/回收调度器**:代码持久化、状态重建、空闲回收(见 §5)。

## 2. 机制一:分层启动优化

**目标**:简单实验 ≤5s 可用,复杂多节点 ≤15s。不做预热池(见 §2.4)。

### 2.1 四层优化

| 层 | 优化 | 消除的等待 |
| --- | --- | --- |
| L1 镜像层 | DaemonSet 全节点预拉取镜像 | 拉镜像的几十秒 |
| L2 链状态层 | 镜像内固化创世状态(创世块/账户/已编译依赖) | 现场初始化链的时间 |
| L3 个性化层 | "环境就绪"与"个性化初始化"分两阶段异步 | 学生无需等待个性化完成 |
| L4 池化层 | 预留扩展点,首版不实现 | (按需) |

### 2.1.1 镜像预拉取闭环

镜像预拉取由 M2 控制面使用 Kubernetes API 创建受控 DaemonSet 完成,不能只更新数据库字段。`runtime_image.prepull_status=2` 的含义是该运行时默认工作负载镜像集合已在目标沙箱节点就绪,不是只拉取运行时主镜像。

执行规则:
- 预拉取资源运行在平台受控命名空间,命名为 `chaimir-prepull-{runtime_image_id}`,带 `app=chaimir`、`module=sandbox`、`runtime_image_id` 标签。
- DaemonSet 使用目标 `runtime_image.image_url`、运行时声明的 `infra_sidecars/pods` 镜像、以及与该运行时生态兼容的可用工具镜像启动短生命周期容器,`imagePullPolicy=IfNotPresent`,资源限制使用平台配置的预拉取 requests/limits。
- 目标节点默认为全部可调度沙箱节点;如部署配置声明 sandbox node selector/toleration,预拉取必须使用同一选择器。
- M2 轮询 DaemonSet 状态,仅当 `desiredNumberScheduled == numberReady == updatedNumberScheduled` 且无拉取失败事件时,才写 `runtime_image.prepulled=true`、`prepull_status=2` 与 `prepulled_at`。
- 任一节点出现 `ImagePullBackOff`、签名校验失败、调度失败或超时,必须写 `prepull_status=3` 与 `prepull_detail`,并返回/记录可定位错误;禁止吞错或保留成功标记。
- `prepull_detail` 必须记录本轮镜像数量和镜像 URL 列表,用于验收确认工具/infra 镜像没有在沙箱启动时临时拉取。
- 预拉取成功后保留 DaemonSet,用于新节点加入时自动补拉;运行时镜像停用或删除时由 M2 删除对应 DaemonSet。

### 2.1.2 接入即测清理边界

运行时接入即测会创建独立自检沙箱,自检结束后必须发起 K8s 清理。清理阶段不得复用已取消的 HTTP 请求 context,也不得使用无界根 context;必须使用保留 trace/tenant 日志上下文且带 `SANDBOX_SELFTEST_RECYCLE_TIMEOUT_SECONDS` 超时的清理 context。清理失败只记录结构化日志并保留可排查 Namespace,不得把未完成清理伪装成成功资源释放。

### 2.2 两阶段启动(关键)

```
阶段一:环境就绪(目标 ≤5s)
  分配 Namespace → 调度 Pod(镜像已预拉取) → 加载固化链状态 → 节点健康
  → 学生此时已可进入:看代码、用编辑器、用终端

阶段二:个性化初始化(后台异步,进度提示)
  部署初始合约 / 装额外依赖 / 注入实验数据
  → 完成前,依赖这些的操作按钮置灰;完成后解灰
```

学生**感知到的"可用"在阶段一即达成**,个性化在后台补齐。这解决了"预热也预热不了个性化配置"的根本矛盾。

### 2.3 分阶段进度反馈

前端进度条分明确阶段:`分配环境 → 启动节点 → 加载链状态 → 就绪`(阶段二再显示 `初始化依赖…`)。每阶段有文案,学生不焦虑。

### 2.3.1 异步启动调度

`POST /sandboxes` 只完成参数校验、配额占用、DB 记录创建与启动任务提交,随后立即返回 `phase=1`。启动任务由 M2 内部调度器推进,并把每一步写入 `sandbox_event` 与进度 WS。

启动任务必须闭环:
- 创建 Namespace、ResourceQuota、LimitRange、NetworkPolicy、PVC 与 Pod 组任一步失败,沙箱进入 `error`,释放已占配额,写明失败阶段到日志与 `sandbox_event`。
- 阶段一超时进入 `error` 并触发回收,前端用户文案只显示"实验环境准备失败,请稍后重试"及 `trace_id`。
- 阶段二个性化初始化失败时沙箱保持可进入但 `phase` 不得标记为 4;依赖个性化的能力保持不可用,失败明细进入 `sandbox_event` 和日志。

### 2.4 为什么首版不做预热池

预热池有三个未解难题,故首版仅保留为扩展点:
1. **预热不了个性化**:池中是通用环境,学生实验的个性化配置仍需现做。
2. **预测不准**:选修课、自由实验、竞赛突发流量,课表预测覆盖不到。
3. **多租户分池难**:每校预热成本高;共享池有隔离与公平性问题。

放宽到"可接受等待区间(5/15s)"后,L1-L3 优化已足够。上线后若发现热门实验仍慢或高峰排队,再针对具体场景开池(数据模型与接口已预留)。

## 3. 机制二:运行时三层适配器

**目标**:标准链零代码接入,且接入即真实可用。针对旧版"宣称 10+ 链实际只做 4 个"的坑。

### 3.1 三层粒度(定制量从外到内递减)

```
L1 声明式清单(YAML,零代码)        ← 大多数 EVM 兼容链只需这层
   镜像 / 启动进程 / 端口 / 资源配额 / 健康检查 / 默认工具搭配
        │
L2 标准能力接口(实现 4-5 个动作)    ← 非标准链需实现
   部署合约 / 发交易 / 查账户·区块·交易 / 重置链
   (EVM 用 JSON-RPC,Fabric 用 SDK,各链自实现)
        │
L3 深度定制插件(可选)              ← 极少数特殊机制链才写
   特殊共识数据采集 / 特殊交互
```

### 3.2 L2 标准能力接口(刻意最小化)

平台只定义最小动作集,降低每条链接入成本:

| 动作 | 用途 | 消费者 |
| --- | --- | --- |
| `Deploy(code)` | 部署合约/链码 | M7 个性化初始化、M3 判分 |
| `SendTx(tx)` | 发起交易 | 工具、M3 |
| `Query(target)` | 查账户/区块/交易/合约状态 | M3 链上断言、区块链浏览器工具 |
| `Reset()` | 链恢复到创世就绪态 | 沙箱重用、重建 |

> 接口刻意收到最小,而非旧版大而全的 Collector。这是 M3 判分和工具的统一入口。

### 3.3 接入即测(堵住"宣称支持"的坑)

每个适配器必须附带**自检套件**,接入时自动跑:

```
起链 → 部署样例合约 → 查询验证 → 重置 → 通过判定
```

自检全绿才标记该运行时为"可用",否则"接入中/失败"。杜绝未验证就上架。

### 3.4 L1 声明式清单(adapter_spec)口径

> `runtime.adapter_spec` 是 **控制面直接执行的 SSOT**。字段必须稳定、显式,禁止运行时实现各自发明隐式约定。
> 运行时、工具、判题和后端计算复用统一的 WorkloadSpec 字段语义:组件(component)、服务(service)、路由(route)、卷域(volume domain)、资源(resources)、探针(probe)和网络规则(network rule)。M2 只消费沙箱类 WorkloadSpec,不得把 M3/M4 的执行器私有协议硬编码进沙箱模块。

```json
{
  "workspace_dir": "/workspace",
  "volume_domains": [
    {
      "name": "workspace",
      "mount_path": "/workspace",
      "student_access": "read_write",
      "persistence": "minio_code",
      "snapshot_scope": "always"
    },
    {
      "name": "runtime-state",
      "mount_path": "/runtime-state",
      "student_access": "none",
      "persistence": "ephemeral",
      "snapshot_scope": "snapshot_enabled"
    }
  ],
  "runtime_container": {
    "name": "runtime",
    "command": ["anvil"],
    "args": ["--host", "0.0.0.0", "--port", "8545"],
    "env": [{ "name": "CHAIN_ID", "value": "31337" }],
    "ports": [
      { "name": "rpc", "container_port": 8545, "service_port": 8545, "protocol": "TCP" }
    ],
    "resources": {
      "requests": { "cpu": "500m", "memory": "512Mi" },
      "limits": { "cpu": "2", "memory": "2Gi" }
    },
    "readiness_probe": { "type": "tcp", "port": "rpc", "period_seconds": 2, "failure_threshold": 30 },
    "liveness_probe": { "type": "tcp", "port": "rpc", "period_seconds": 10, "failure_threshold": 3 }
  },
  "infra_sidecars": [],
  "pods": [
    {
      "name": "sandbox",
      "containers": [
        { "name": "runtime" }
      ]
    }
  ],
  "services": [
    {
      "name": "runtime-sandbox",
      "component": "sandbox",
      "ports": [{ "name": "rpc", "port": 8545, "target_port": "rpc", "protocol": "TCP" }]
    }
  ],
  "routes": [],
  "network_rules": [],
  "init_assets": [
    {
      "name": "price-feed-fixture",
      "source_ref": "minio://problem-assets/evm/price-feed-fixture.tgz",
      "apply_phase": "personalization"
    }
  ],
  "default_tool_codes": ["code-server"],
  "selftest": {
    "deploy_payload": { "contract_ref": "minio://samples/evm/Counter.sol" },
    "query_target": "latest",
    "tx_payload": { "method": "increment" }
  }
}
```

字段约束:
- `workspace_dir`:沙箱代码工作目录。文件接口、初始化脚本、工具 sidecar 均挂载到该目录。
- `volume_domains`:沙箱卷安全域声明。M2 必须按安全域分别创建和挂载卷,禁止把运行态、判题私有数据或教师答案混入 `workspace`。
  - `workspace`:学生代码工作区,允许文件接口和授权工具读写;对象存储只持久化该域。
  - `public-assets`:公开题面素材或公开依赖,只读挂载给学生可进入容器。
  - `runtime-state`:链节点数据库、账本、缓存、共识状态等运行态。默认临时卷,学生文件接口不可见;仅 `snapshot_enabled=true` 且通过配额/能力校验时才纳入 CSI VolumeSnapshot。
  - `judge-private`:隐藏测试、flag、答案、评分脚本、教师私钥等私有域,永不挂载到学生可进入容器,也不允许被终端、IDE、Web 工具间接读取。
  - `student_access` 仅允许 `none`、`read_only`、`read_write`;学生可进入容器只能挂载 `read_only/read_write` 域,且文件 API 永远只以 `workspace_dir` 为根。
  - `snapshot_scope` 仅允许 `never`、`always`、`snapshot_enabled`;默认重建路径不得依赖 `runtime-state` 快照。
- `runtime_container`:运行时主容器。镜像地址来自 `runtime_image.image_url`,此处只声明**如何启动与探活**。主容器默认不是学生终端目标;只有显式声明 `labels["chaimir.io/student-access"]="true"` 且不挂载 `student_access=none` 卷域时,才允许作为 `terminal?container=` 目标。声明 `judge-private` 域的运行时必须把 `runtime_container` 保持为非学生入口,隐藏测试注入与判题执行固定走该容器;需要学生 shell 时必须使用单独的学生入口 sidecar。
- `infra_sidecars`:运行时自带的协同容器(如 graph node、Chainlink node 等真实链下组件),由运行时 manifest 显式声明。教师脚本、题目固化资产或链上依赖合约即可表达的轻量模拟能力应放入 `init_assets` 或初始化脚本,不作为平台必需模拟服务镜像。协同容器可为无端口执行类 sidecar,但必须仍声明资源、安全标签和预拉取自检命令。
- `infra_sidecars[].labels["chaimir.io/student-access"]`:只有显式为 `"true"` 的协同容器才允许作为 `terminal?container=` 目标;未声明或为 `"false"` 时只能由平台控制面/工具代理按端口访问,不得开放学生 shell。学生可进入容器只能挂载 `student_access=read_only/read_write` 的卷域,不得挂载 `runtime-state`、`judge-private`、Secret 或控制面凭据。
- `infra_sidecars[].prepull_command`:该 sidecar 镜像的预拉取自检命令,必须来自对应 `images/<category>/<name>/manifest.yaml` 的 `selftest.commands[0].command` 或同等受控声明。不得复用运行期 `command` 冒充自检;缺失时镜像不得标记 `prepulled=true`。
- `infra_sidecars[].prepull_hold`:标记该 sidecar 是否作为预拉取 DaemonSet 的常驻保持容器。一个运行时镜像集合必须至少有一个 `prepull_hold=true` 的镜像,其余镜像用 `prepull_command` 作为 initContainer 真实拉取和自检。
- `pods`:声明一个沙箱实例的运行时 Pod 组。未声明时,M2 按 `runtime_container + infra_sidecars` 生成单 Pod 拓扑;声明后,M2 逐个创建 Pod,等待所有运行时 Pod Ready 后才进入阶段二。`pods[].containers[].name` 必须引用本清单中的运行时主容器或协同容器声明;主运行时容器仍以 `runtime_image.image_url` 为唯一镜像来源,不得在 `pods` 中重复写 `image_url`;协同容器必须声明带 digest 的 `image_url` 并命中签名/扫描证明。每个 Pod 内容器名和容器端口必须唯一。
- `services`:声明组件的集群内 Service,用于工具、infra、运行时 RPC 和受控代理访问。Service 名在同一沙箱内必须唯一,`component` 必须引用已声明组件或 Pod,`target_port` 优先引用端口名。
- `routes`:声明平台代理入口。运行时通常为空;Web 工具可声明多个 path prefix 到不同组件,例如 `/api` 到 backend、`/` 到 frontend。路由只生成平台代理可达入口,不暴露直连 Service。
- `network_rules`:声明同一沙箱内不同组件或 Pod 之间的最小网络互通。默认 deny-all 下,组件之间也不可互访;只有这里显式声明 `from`、`to` 和目标端口名/端口号时,M2 才生成对应 ingress/egress NetworkPolicy。规则只能引用已存在组件/Pod 和目标已声明端口,禁止用空端口、通配或整命名空间互通表达。
- `init_assets`:题目固化资产或教师上传初始化包,由 M2 在个性化阶段按安全归档规则注入并执行。该字段只引用平台对象存储中的已审核资产,不得包含密钥、答案、flag 或未校验归档。
- `ports[].name`:同一清单内唯一,探针与能力实现按名称引用,禁止写死数字端口。
- `resources`:缺失时回退到系统默认 `LimitRange`;声明时必须同时给 `requests` 与 `limits`。
- `readiness_probe` / `liveness_probe.type`:仅允许 `tcp` / `http` / `exec` 三种,避免运行时私有探针语义。
- `readiness_probe` / `liveness_probe` 未声明 `period_seconds` 或 `failure_threshold` 时,默认值必须来自 `SANDBOX_PROBE_DEFAULT_PERIOD_SECONDS` 与 `SANDBOX_PROBE_DEFAULT_FAILURE_THRESHOLD`,不得在编排代码中硬编码。
- `default_tool_codes`:仅作调用方默认值建议,**不绕过** M2 的工具兼容校验。
- `selftest`:接入即测的最小样例参数。L2 自检统一执行“起链 → deploy → query → reset”;如该链需要额外 `tx` 校验,可补 `tx_payload`。
- `capability_commands`:声明式 L2 标准链能力命令集合。若运行时不通过服务端注册的 `capability_impl`/`plugin_ref` 提供能力,则必须在清单中声明 `deploy`、`tx`、`query`、`reset` 四个动作的受控命令。M2 通过 K8s exec 向命令 stdin 传 JSON、从 stdout 读取 JSON 对象;stderr 和底层错误只进入日志和错误链,不进入用户响应。禁止只登记 `adapter_level=2/3` 却没有可执行能力。

## 4. 工具接入协议

**目标**:工具可插拔,不绑定单一 IDE。终端复用 K8s,不自研。

### 4.1 工具类型

| 类型 | 接入方式 | 例子 |
| --- | --- | --- |
| `terminal` | K8s exec PTY + WebSocket 代理,只能进入显式学生入口容器 | 命令行终端 |
| `web-embed` | HTTP 反向代理 + iframe 嵌入 | code-server、Remix、区块链浏览器、Jupyter |
| `command-tool` | 无端口工具 Pod + 受控 argv 执行接口 | Slither、Mythril、gas-profiler |
| `platform-builtin` | 平台直供,无需额外镜像 | 文件管理、日志、状态面板 |

### 4.2 工具声明(可插拔)

每个工具声明:`类型 + WorkloadSpec + 适用链生态`。新增工具 = 加一份工具定义,平台代码不改。工具表不再单独保存镜像或端口字段;单组件工具也必须在 `resource_spec.components`、`services` 和 `routes` 中声明入口,避免单容器/多容器两套逻辑并存。

`tool.resource_spec` 统一结构:

```json
{
  "components": [
    {
      "name": "web",
      "image_url": "harbor.chaimir/tool/code-server@sha256:...",
      "mount_workspace": true,
```

约束:
- `kind=terminal`:平台直供,不额外起工具容器,`resource_spec.components` 必须为空;若运行时没有显式 `chaimir.io/student-access=true` 的安全容器,不得默认挂载或展示终端入口。
- `kind=web-embed`:必须声明至少一个组件、一个 Service 和一个入口 route。组件镜像必须携带 digest 并命中镜像准入证明。
- `kind=command-tool`:必须声明唯一 `components[].name=command` 的无端口容器、`mount_workspace=true` 和 `command_policy`;不得声明 Service、route 或 `network_rules`。容器长期执行 `keepalive_command`,用户触发时由 M2 通过统一 K8s exec 基础层执行白名单 argv,输入输出使用 base64,不暴露 Pod/Namespace。
- `kind=platform-builtin`:由平台直供(文件管理、状态面板等),不创建 sidecar。
- `components[].mount_workspace=true`:把运行时 `workspace_dir` 挂入该组件;不共享工作目录的组件必须显式写 `false`。区块链浏览器、索引器、数据库和代理类组件默认不得挂载 workspace。
- `tool.runtime_config_required=true`:表示该工具自身镜像已经登记,但必须等运行时/实验编排提供连接配置、证书或配套服务后才能调度。Seed 必须把此类工具写成停用状态,不得因为镜像已通过扫描签名就误标为可用;当对应 WorkloadSpec 组合配置真实落地并通过验收后,再由同一数据源改为可调度。
- `components[].ephemeral_mounts`:组件在只读根文件系统下需要的临时可写目录,由 M2 编排为仅挂给该组件的 `emptyDir`;不得覆盖 `/workspace`、`/runtime-state`、`/judge-private`、`/var/run`、`/etc`、`/proc`、`/sys`、`/dev` 等工作区、私有域或系统敏感路径。
- `routes`:所有学生可访问 Web 工具必须经 M2 工具代理路由进入。Blockscout/Fabric Explorer 等 frontend/backend/proxy 组合应通过多 route 表达,后端不得按工具 code 写特判。
- `network_rules`:仅 `web-embed` 工具可声明,用于放行工具组件之间、工具组件到运行时 Pod 的必要端口。未声明时工具只能被平台代理访问,不能主动连接链节点或其他沙箱组件。
- `command_policy`:仅 `command-tool` 可声明,`allowed_commands` 只允许无路径命令名,请求命令必须逐字命中白名单;`default_timeout_seconds` 与 `max_timeout_seconds` 均为正数,并受平台执行超时上限再次约束。
- `env` 仅允许字面值;密钥/凭据不得放入工具定义,需由平台 Secret 注入的场景必须通过受控 infra 组件和 Secret 引用机制表达,不得塞入普通工具环境变量。

### 4.3 动态工作台

M2 提供工具元数据与接入端点;前端按实验配置的工具清单**动态渲染**工作台(这个实验给 Remix+终端,那个给浏览器+命令行)。工具与链按生态自动过滤兼容性。

### 4.4 终端不自研

复用 K8s exec subresource(SPDY/WebSocket stream),控制面做一层 WS 代理把沙箱内 shell 接到浏览器。不重新发明终端协议。

## 5. 机制三:分层持久化(默认重建)

**目标**:代码绝不丢,链状态按需重建,大并发不炸。针对旧版"每 5 分钟全量快照"的坑。

### 5.1 区分三个真实需求

| 需求 | 解法 | 不用全量快照 |
| --- | --- | --- |
| 断点续做 | 持久化代码 + 重建环境 | ✓ |
| 防丢失 | 代码实时持久化 MinIO | ✓ |
| 评测取证 | 评测时实时读链上状态 | ✓ |

### 5.2 分层策略

```
学生代码  → 实时持久化(挂载卷 + 防抖保存 → MinIO),代码量小成本低
链状态    → 不快照,记录代码+部署脚本,恢复时重放重建
运行态    → 默认不存;教师对"重建代价高"的实验(挖矿/长共识)可显式开启
            保活或磁盘快照,默认关,且有最长保活时长上限防僵尸
```

### 5.2.1 代码持久化闭环

沙箱工作目录以 PVC 挂载到运行时与工具容器。文件写入后 M2 在配置的防抖窗口内保存到 MinIO,用户主动保存、回收前、暂停前必须立即保存。
所有学生侧 IDE/Web 工具必须经 M2 工具代理访问;代理层记录工具访问活跃度并安排防抖保存,避免绕过 M2 的直连工具写入无法进入持久化闭环。禁止在生产环境暴露工具容器直连入口。

约束:
- MinIO key 统一为 `{tenant_id}/sandbox/code/{sandbox_id}/...`,与平台对象存储路径约定一致。
- 文件路径必须在 `runtime.adapter_spec.workspace_dir` 下解析,禁止 `..`、绝对路径逃逸、符号链接逃逸。
- 文件接口、代码归档、对象存储保存和学生 IDE 只能访问 `workspace` 域;不得读取或打包 `runtime-state`、`judge-private`、平台 Secret、控制面日志、隐藏测试或答案。
- 初始代码归档恢复不得把对象存储中的 tar/zip 直接交给容器内 `tar` 解包。M2 必须在后端先逐条校验归档成员名,拒绝绝对路径、`..`、Windows 盘符、重复条目、软/硬链接、特殊文件,并按 `SANDBOX_INIT_ARCHIVE_MAX_FILES`、`SANDBOX_INIT_ARCHIVE_MAX_UNPACKED_BYTES` 限制文件数和展开大小后重打安全 tar 再写入工作区。
- 防抖保存失败必须记录结构化日志与 `sandbox_event`;回收前最终保存失败时不得删除 Namespace,沙箱进入 `error` 并等待人工/重试处理。
- 代码引用由 M2 维护为权威,M3/M6/M8 只保存引用与提交时 `code_hash`,不复制代码正文。

### 5.2.2 保活与快照闭环

保活和快照仅用于教师显式开启的高重建成本实验,默认关闭。实现采用 Kubernetes PVC 与 CSI VolumeSnapshot;部署基线必须安装可用的 VolumeSnapshot CRD 与支持快照的 StorageClass。没有快照能力的环境不能开启 `snapshot_enabled`,必须在创建时明确拒绝。

执行规则:
- `keep_alive=true` 只跳过空闲回收,仍受 `tenant_quota.max_keepalive_min`、`tenant_quota.max_lifetime_min`、租户并发与资源配额约束。
- `snapshot_enabled=true` 时,M2 在暂停、回收前或调用方要求保留运行态时创建 VolumeSnapshot,并把快照引用写入 `sandbox.snapshot_ref`、`snapshot_created_at`、`snapshot_expire_at`。
- 快照能力检测必须同时确认 CSI `VolumeSnapshot` API、部署配置中的 `SANDBOX_VOLUME_SNAPSHOT_CLASS_NAME` 对应的 `VolumeSnapshotClass` 以及 `SANDBOX_STORAGE_CLASS_NAME` 对应的 PVC 存储类均可用。未显式配置快照类或存储类时,`snapshot_enabled=true` 必须拒绝,不得只因 CRD 存在就误报快照可用。
- 快照只覆盖 `volume_domains.snapshot_scope=always` 或 `snapshot_enabled` 且通过策略校验的域。`judge-private`、Secret、答案、flag、隐藏测试和控制面凭据不得进入学生沙箱快照;默认重建路径只恢复 `workspace` 后重放部署脚本。
- 快照保留时间不得超过 `tenant_quota.max_snapshot_retention_min`;由于 CSI `VolumeSnapshot` 与源 PVC 同属 Namespace,快照回收时不得立即删除 Namespace,必须先释放计算工作负载,到期再由回收调度器删除 VolumeSnapshot、PVC 与保留的快照 Namespace。
- 从快照恢复时先新建 Namespace 与 PVC,再从 VolumeSnapshot 恢复工作目录/链运行态,随后执行健康检查;失败时必须进入 `error` 并保留原快照至过期。

### 5.3 断点续做流程

学生离开 → 空闲超时(默认 30min)回收沙箱(代码已存)→ 回来 → 新建环境 + 拉回代码 + 重放部署脚本 → 恢复到离开前的逻辑状态。

## 6. 沙箱生命周期与隔离

### 6.1 隔离

- 每沙箱 = 独立 K8s Namespace(`sbx-{sandbox_id}`)+ Pod(s)。
- NetworkPolicy **默认 deny-all**,按需放行;沙箱间、租户间网络完全隔离。
- ResourceQuota + LimitRange 限单沙箱资源;Pod Security Restricted。

### 6.2 资源配额

- 租户级:并发沙箱数 / 总 CPU / 总内存上限,防止单校占满集群。
- 超配额时排队或拒绝,并返回明确提示。

### 6.3 回收

三重回收保证无僵尸:
1. 空闲超时(默认 30min 无操作)。
2. 最长生命周期强制回收。
3. 来源结束级联(M7 实验结束/M8 竞赛结束,调用方通知 M2 批量回收)。

回收任务必须按顺序执行:锁定沙箱状态为 `recycling` → 立即保存代码 → 按需创建快照 → 普通回收删除 Namespace/PVC/Service/NetworkPolicy 等 K8s 资源;快照回收缩容/删除计算工作负载并保留 PVC/VolumeSnapshot/Namespace 至过期 → 写 `sandbox.status=destroyed` → 写 `sandbox_event` → 发布 `sandbox.recycled` 事件。任何步骤失败都必须保留可重试状态,不能把未清理完成的沙箱标记为 `destroyed`。

回收调度器是 M2 service 内部后台任务,由 `cmd/server/sandbox.go` 通过 `platform/background` 启动,不得反向调用上层业务模块。每轮只做本模块表扫描和 K8s 清理:
- `SANDBOX_RECYCLE_POLL_INTERVAL_SECONDS`:调度器轮询间隔。
- `SANDBOX_RECYCLE_BATCH_SIZE`:每轮最多锁定并处理的沙箱数量,避免全表扫描或长事务。
- `SANDBOX_READY_IDLE_TIMEOUT_SECONDS`:creating/ready 且用户未进入操作的沙箱超时回收阈值。
- `SANDBOX_SELFTEST_RECYCLE_TIMEOUT_SECONDS`:运行时接入即测结束后清理自检 Namespace 的最大等待时间。

调度器候选包括:ready 超时、idle 超时、`expire_at` 最长生命周期到期、`keep_alive_until` 到期、已进入 `recycling` 的未完成回收重试,以及 `destroyed + snapshot_expire_at` 到期后的快照 Namespace/PVC/VolumeSnapshot 清理。空闲阈值使用 `tenant_quota.idle_timeout_min`;`keep_alive=true` 只跳过 idle 超时,不跳过 ready 超时、`keep_alive_until` 或 `expire_at`。

### 6.4 内部鉴权、模块边界与审计

- 模块化单体内,M3/M7/M8 等调用 M2 必须经 `internal/contracts` 中的 `SandboxService` 等接口,禁止直接 import `internal/modules/sandbox`。
- 若 `[内部]` API 需要通过 HTTP 暴露,只能在内网路由启用,必须使用服务间鉴权,并强制携带 `tenant_id`、调用方模块、`source_ref` 与 trace 上下文。
- M2 不能反向调用 M7/M8/M3。回收、失败、状态变化等反向通知统一发布 NATS 事件,如 `sandbox.recycled`。
- 运行时、镜像、工具、沙箱创建/控制/回收、配额变更等关键操作必须写 M1 `audit_log`;`sandbox_event` 只保存 M2 内部技术事件与阶段明细。

## 7. 多租户与私有化同构

- SaaS:K8s 集群,按租户 Namespace 前缀 + 配额隔离。
- 私有化:k3s 轻量部署,单租户,逻辑与 SaaS 一致(配额恒为本校全部资源)。
- 镜像分发:私有化用离线镜像包导入本地 Harbor。
