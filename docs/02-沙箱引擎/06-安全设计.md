# M2 沙箱引擎 — 安全设计

> 沙箱直接运行用户代码,是平台攻击面最大的模块。本文覆盖容器隔离、网络、资源、镜像、执行鉴权、教师脚本安全。
> 最后更新:2026-06-04

---

## 1. 容器与运行时隔离

- **Pod Security:Restricted**。禁止特权容器、禁止 hostPath、禁止以 root 运行(non-root user)、只读根文件系统(可写层限定挂载卷)。M2 生成的沙箱 Pod 统一 `runAsUser=1000`、`runAsGroup=1000`、`fsGroup=1000`;沙箱镜像必须在构建期满足该身份契约,不得由编排层按镜像 UID 调整运行身份。
- **能力裁剪**:drop ALL Linux capabilities,按需精确添加。
- **seccomp/AppArmor**:启用默认 seccomp profile,限制系统调用,降低容器逃逸面。
- **运行时加固(可选增强)**:对高风险实验(运行不可信代码)可用 gVisor/Kata 等沙箱化运行时进一步隔离内核。
- 每沙箱独占 Namespace(`sbx-{id}`),Namespace 级 ResourceQuota/LimitRange。
- 沙箱 Pod 不自动挂载 ServiceAccount token;除平台控制面代理外,沙箱内进程不得访问 Kubernetes API。

---

## 2. 网络隔离

- **NetworkPolicy 默认 deny-all**:沙箱默认无任何出入站,按需精确放行。
- **沙箱间隔离**:不同沙箱、不同租户网络完全不可达。
- **出网管控**:默认禁止访问公网与内网其他服务;确需外网(如拉依赖)走受控代理白名单,或在镜像内预置依赖避免出网。
- **链 RPC 仅内部**:沙箱内链节点的 RPC 端口不直接暴露公网,经控制面代理 + 鉴权访问。
- **精确放行**:仅允许平台后端控制面访问本沙箱工具/终端代理所需端口;沙箱 Pod 之间默认不可互访,同一沙箱内多 Pod 互通必须由 `runtime.adapter_spec.network_rules` 显式声明 `from_pod`、`to_pod` 与目标端口,由 M2 同时生成 ingress/egress NetworkPolicy,禁止整 Namespace 或通配端口互通。

---

## 3. 资源限制与防滥用

- 单沙箱 CPU/内存/存储硬限制(LimitRange),防止挖矿、fork 炸弹、磁盘填满。
- 租户级总配额上限,防止单校占满集群影响其他租户。
- 进程数/PID 限制、文件描述符限制。
- 超限自动终止并记录事件。

---

## 4. 镜像安全

- 仅允许来自私有 Harbor 的镜像;禁止任意外部镜像。
- 镜像构建 **Trivy 漏洞扫描** + **Cosign 签名验证**,未签名/高危镜像不得上架。
- 自定义镜像(若开放)须经平台管理员审核 + 扫描后方可进入预拉取池。
- 镜像以 non-root 构建,多阶段精简体积。runtime/tool/judger 及沙箱内 infra 镜像最终 `USER` 必须为 `1000:1000`;只允许在构建期为必要运行目录做最小权限归属调整,禁止 `chmod 777`、`chown -R /` 或保留 root group。
- M2 预拉取前必须校验镜像地址在 Harbor 允许列表内,且镜像已通过扫描与签名门禁;失败时不得创建 DaemonSet。

---

## 5. 执行与接入鉴权

- **终端/文件/工具接入**:校验调用者 JWT + 该沙箱归属(owner_account_id / source_ref 对应租户与用户),禁止越权操作他人沙箱。
- **K8s exec 代理**:控制面持有受限 ServiceAccount(最小 RBAC,仅能在 `sbx-*` Namespace exec),前端永不直接接触 kubeconfig。
- **链上能力接口** `[内部]`:模块化单体内经 `internal/contracts` 调用;HTTP 暴露时仅服务间鉴权可调用,强制带 `tenant_id`、调用方模块、`source_ref` 与 trace 上下文校验沙箱归属。
- **Web 工具代理**:HttpOnly cookie + 路径绑定 sandbox_id,防止跨沙箱访问。
- **审计**:运行时/镜像/工具管理、沙箱创建/暂停/恢复/销毁/回收、配额调整必须写 M1 `audit_log`;终端命令、文件保存、K8s 阶段等技术明细写 `sandbox_event`。

---

## 6. 教师脚本安全(个性化初始化/部署脚本)

> 教师可配置初始化脚本(部署初始合约等),这是潜在注入面。

- 脚本**在目标沙箱内以受限用户执行**,与平台控制面完全隔离,无法触达平台数据库或其他租户。
- 脚本执行同样受 Namespace 资源与网络策略约束。
- 脚本内容入库前做基础校验(长度、危险命令模式告警);执行有超时与资源上限。
- 脚本引用与执行记录入 `sandbox_event` 技术事件;涉及沙箱创建、脚本配置变更等关键操作另写 M1 `audit_log`。

---

## 7. 多租户隔离

- 数据层:`sandbox` 等表 `tenant_id` + RLS(同 M1)。
- 编排层:Namespace 命名隔离 + NetworkPolicy + 配额,跨租户资源不可见不可达。
- 私有化:单租户,隔离策略仍启用(恒满足)。

---

## 8. 持久化数据安全

- 学生代码持久化到 MinIO,按 `tenant_id/sandbox_id` 路径隔离,访问经鉴权。
- 学生代码持久化到 MinIO 的路径使用 `{tenant_id}/sandbox/code/{sandbox_id}/...`;不得使用用户提交的路径片段拼接对象根目录。
- 卷按安全域隔离:`workspace` 为学生代码工作区,`public-assets` 为公开只读素材,`runtime-state` 为链节点运行态,`judge-private` 为隐藏测试/flag/答案/评分脚本等私有域。
- 文件 API、代码归档、MinIO 保存与学生 IDE 只能访问 `workspace`;不得读取、列目录、打包或下载 `runtime-state`、`judge-private`、Secret、ServiceAccount token、控制面日志、教师答案、隐藏测试和私钥。
- `runtime-state` 默认使用临时卷并通过代码和部署脚本重建;只有教师显式开启 `snapshot_enabled`、集群具备 CSI 快照能力且租户配额允许时,才可对经策略允许的运行态卷创建 VolumeSnapshot。
- `judge-private`、flag、隐藏断言、评分脚本、教师私钥、答案和平台凭据永不进入学生沙箱快照,也不得挂载到学生可进入容器。
- 声明 `judge-private` 的运行时必须让 `runtime_container` 保持非学生入口;内部隐藏资产注入和判题执行固定走该容器。学生需要终端时只能进入单独声明的学生 sidecar,不能通过 UID、文件权限或前端隐藏来隔离答案。
- 不在沙箱镜像或日志中残留敏感数据;回收时彻底删除 Namespace 与临时卷。
- 快照对象与 VolumeSnapshot 引用必须按租户和沙箱绑定,恢复时校验 `tenant_id`、`owner_account_id`、`source_ref`;禁止跨租户复用快照。

---

## 9. 安全测试要点(供实现阶段验证)

- 容器逃逸尝试(特权提升、hostPath、内核漏洞)必须被 Pod Security + seccomp 拦截。
- 跨沙箱/跨租户网络连通性测试必须全部不可达。
- 越权访问他人沙箱终端/文件/工具必须被拒。
- 资源炸弹(fork bomb、内存/磁盘填满)必须被限制且不影响宿主与邻居。
- 恶意教师脚本无法触达控制面或其他租户数据。
- 镜像签名校验、漏洞扫描门禁有效。
