# 后端层能力索引

> 本文给后续 AI 和开发者快速判断“已有能力在哪里、什么时候该复用”。它不是源码逐行说明,只记录跨层边界和通用能力入口。改动后端共享能力时,同步维护本文。

---

## 1. 使用原则

- 先看 `docs/总-工程目录设计.md`:模块间只走 `internal/contracts`,跨模块基础设施优先放 `internal/platform`,真正无业务且可外部复用的通用库才放 `backend/pkg`。
- 不要在模块里临时复制小工具。发现同类逻辑第二处出现时,先判断是否已有 `pkg` 或 `internal/platform` 能力可复用。
- 密钥、连接串、端口、超时和阈值不放 `pkg`;运行配置统一经 `internal/platform/config` 和环境变量进入。
- `pkg` 不访问数据库、Gin、NATS、K8s、对象存储和模块表;需要这些依赖的能力通常属于 `internal/platform` 或具体模块。

### 1.1 AI 接入前检查清单

后续 AI 在实现任一后端模块前,必须先按需求关键词查本文,确认是否已有统一入口:

- 跨模块调用、共享 DTO、事件主题/载荷:先查 `internal/contracts`,不要直接 import 其他模块 package,也不要在模块内复制契约结构。
- 错误码、用户向文案、内部错误链:先用 `pkg/apperr`,不要在模块内自建错误结构。
- HTTP JSON 响应、分页、`trace_id`、panic 收敛:先用 `internal/platform/response` 或 `internal/platform/httpx`,不要直接拼响应信封。
- 结构化日志、错误脱敏、关闭资源失败记录:先用 `pkg/logging`,不要裸写敏感错误字符串。
- 密码哈希、HMAC、AES-GCM、随机短码/凭证:先用 `pkg/crypto`,不要用 `math/rand`、模块私有随机函数或自定义加密格式。
- 手机号展示脱敏、凭据字段识别、用户可见结果敏感字段识别:先用 `pkg/privacy`,不要复制关键词表或切片掩码逻辑。
- BIGINT 主键生成:先注入 `pkg/snowflake.Generator`,不要用时间戳、随机数或模块私有序列。
- 链上状态断言:先用 `pkg/chainassert`,不要在 judge、contest、experiment 等模块各写一套断言解释器。

如果本文没有对应入口,再判断应放 `internal/platform`、具体模块私有实现,还是新增无业务 `pkg` 包;新增后必须同步更新本文。

## 2. internal/contracts

模块间接口契约、跨模块 DTO、事件主题和事件载荷的唯一入口。

- 用于:模块之间同步调用、聚合层只读查询、事件总线 payload、跨模块身份角色字符串与数据库/审计数字枚举。
- 需求关键词:跨模块调用、contracts、DTO、事件主题、source_ref、角色字符串、角色数字枚举、只读契约、接口注入。
- 关键能力:`IdentityService`、`SandboxService`、`JudgeService`、`SimService`、`ContentReadService`、`TeachingReadService`、`NotifyService`、`Subject*` 事件主题、`Role*`/`RoleNum*` 与 `RoleCode`/`RoleNumber`。
- 约束:本层只定义接口、DTO、事件和必要的稳定身份角色映射;不访问数据库、HTTP 框架、K8s、NATS、对象存储或具体模块实现;不得把 `internal/platform` 的基础设施实现搬进来。
- 边界:`RolePlatformAdmin`/`RoleSchoolAdmin`/`RoleTeacher`/`RoleStudent` 是身份鉴权字符串;`RoleNumPlatformAdmin`/`RoleNumSchoolAdmin`/`RoleNumTeacher`/`RoleNumStudent` 是同一身份角色在数据库与审计中的数字枚举;审计系统任务 `actor_role=5` 属于 `internal/platform/audit`,不要在 `contracts` 里新增第二套系统角色语义。
- 序列化:导出的跨模块 DTO 字段必须带稳定 `json` 标签,避免内部 HTTP、事件消息、调试导出或未来拆服务时泄漏 Go 字段名。
- 依赖方向:业务模块依赖 `internal/contracts` 注入其他模块能力;`internal/platform` 和 `pkg` 可按工程目录规范提供基础能力,但不要为了方便在 `contracts` 中复制基础层能力。

## 3. backend/pkg

### `pkg/apperr`

统一应用错误体系和全平台错误码单一入口。

- 用于:模块、平台层和 API 边界返回稳定错误码、用户向文案和内部原因链。
- 需求关键词:错误码、用户提示、`code/message/trace_id`、`WithCause`、错误码段、失败场景。
- 关键能力:`New` 定义稳定错误模板,`WithCause` 包裹内部错误,`AsAppError`/`As` 把任意错误归一。
- 约束:新增失败场景先在对应 `*_codes.go` 定义专用错误码;不要动态替换同一错误码的用户文案;技术原因只进日志。
- 不要自己实现:模块内错误码常量、`code/message` 临时结构、把 SQL/内部字段拼进前端响应。

### `pkg/logging`

统一结构化日志、上下文字段透传和敏感信息脱敏。

- 用于:服务启动设置全局 `slog`,业务/平台层记录带 `trace_id`、`tenant_id`、操作上下文的错误日志。
- 需求关键词:日志、脱敏、trace、tenant、关闭资源、`slog`、敏感字段。
- 关键能力:`Setup`、`WithAttrs`、`AttrsFromContext`、`ErrorContext`、`CloseContext`、`SanitizeError`。
- 约束:错误链落日志前必须走脱敏;带错误返回的 `Close` 用 `CloseContext` 记录失败;不要裸写密码、token、密钥、手机号明文。

### `pkg/crypto`

通用安全原语:密码哈希、HMAC、AES-256-GCM、统一随机短码/凭证生成。

- 用于:M1 密码/手机号哈希/验证码/激活码,服务间签名,对象下载授权,内容完整性摘要等非业务安全原语。
- 需求关键词:密码、验证码、短信码、邀请码、激活码、Refresh Token、HMAC、签名、AES、加密、摘要。
- 关键能力:`HashPassword`/`VerifyPassword`、`HMACSHA256Hex`、`HMACHash`、`EqualHexHMAC`、`SHA256Hex`、`NewCipher`、`RandomToken`。
- 约束:`RandomToken` 是 Refresh Token、激活码、邀请码、短信验证码等短码/凭证的统一入口,使用去歧义大写字母数字字符集和 CSPRNG 均匀抽样;不要用 `math/rand`、取模映射或模块私有随机函数生成安全凭证。
- 边界:密钥来源和轮换属于 `internal/platform/config` 与部署 Secret,本包只接收已加载的密钥字节。

### `pkg/privacy`

跨模块用户数据脱敏与敏感字段识别。

- 用于:用户向展示或日志脱敏时隐藏手机号,以及日志、配置保护、判题/链上结果摘要判断敏感字段。
- 需求关键词:手机号展示、手机号脱敏、日志手机号、敏感字段、凭据字段、flag、answer、secret、token。
- 关键能力:`MaskPhone`、`IsCredentialKey`、`IsResultSensitiveKey`、`ContainsResultSensitiveText`。
- 约束:凭据字段和用户可见结果字段语义不同,不要把 `flag/answer` 这类业务结果敏感词用于配置加密判断;非法手机号返回空字符串,避免误把原值展示给用户。

### `pkg/snowflake`

全平台 BIGINT 主键 ID 生成。

- 用于:模块 service 或装配层需要创建新主键时注入 `snowflake.Generator`。
- 需求关键词:主键、ID、雪花、`BIGINT`、生成器。
- 关键能力:`NewNode`、`Generator`、`Node.Generate`。
- 约束:节点号来自环境配置,生产多副本必须保证 `SNOWFLAKE_NODE_ID` 在同一部署内唯一;生成器会处理同毫秒序列耗尽和短时钟回拨,但节点号冲突只能由部署编排避免;模块不要自己用时间戳、随机数或数据库序列临时生成业务主键。

### `pkg/chainassert`

链上状态断言的通用判定能力。

- 用于:M3 判题和 M8 漏洞预验证对链上查询结果执行 `eq`、`ne`、`contains`、`exists` 等断言。
- 需求关键词:链上断言、判题断言、漏洞预验证、`eq/ne/contains/exists`、actual 摘要。
- 关键能力:`FromMap`、`Check`、`ShortJSON`。
- 约束:返回给前端的实际值会先按 `pkg/privacy` 的用户可见结果敏感规则脱敏再截断,避免泄露完整链上状态、判题期望结构、flag 或密钥;不要在 M3/M8 各写一套断言解释器或短 JSON 摘要函数。

## 4. 后续维护

- 新增 `pkg` 包时,必须说明它为什么是“无业务、可外部复用”,而不是 `internal/platform` 或某模块私有能力。
- 新增 `internal/platform` 通用能力时,在本文继续追加对应章节,说明它的唯一入口和模块不得重复实现的边界。
- 发现重复小功能时,先补索引再收口实现,让后续 AI 能从本文找到已有能力。

## 5. internal/platform

### `internal/platform/response`

统一 HTTP JSON 信封、分页结构、`trace_id` 贯穿和 panic 收敛。

- 用于:Gin API 层写成功/失败响应,以及请求入口注入 `X-Trace-Id`。
- 需求关键词:接口响应、分页、错误返回、panic、追踪编号、`X-Trace-Id`。
- 关键能力:`TraceMiddleware`、`RecoveryMiddleware`、`OK`、`OKPage`、`Fail`。
- 约束:API 层不要直接 `c.JSON` 拼 `{code,message}`;分页响应统一用 `Page` 形态;错误响应只输出用户向文案和 `trace_id`。
- 推荐路径:业务 API 通常优先用 `internal/platform/httpx.Write/Bind/PathID`,它最终复用本包信封。

### `internal/platform/httpx`

Gin handler 层通用绑定、ID/分页解析和附件响应入口。

- 用于:API 文件绑定 JSON、解析路径 ID/查询整数、写统一成功/失败响应、输出小型附件。
- 需求关键词:handler、BindJSON、PathID、分页、附件下载、Content-Disposition。
- 关键能力:`BindJSON`、`BindJSONWithError`、`Write`、`WritePage`、`WriteAttachment`、`PathID`、`QueryInt`、`Page`。
- 约束:handler 不直接拼响应信封;模块需要绑定失败专用错误码时传稳定 `apperr` 模板,不要动态改用户文案;路径、查询和表单数字字段必须显式解析失败,不要把非法输入静默转成 0;附件响应统一带安全文件名和 `X-Content-Type-Options: nosniff`。

### `internal/platform/auth`

JWT access token、中间服务 HMAC 签名和角色守卫中间件。

- 用于:HTTP/WebSocket 鉴权、平台/租户角色守卫、内部服务回调验签。
- 需求关键词:JWT、Bearer、WebSocket token、内部服务签名、HMAC、角色中间件。
- 关键能力:`Manager`、`IssueAccess`、`VerifyAccess`、`Middleware`、`WebSocketMiddleware`、`ServiceMiddleware`、`RequireTenantAnyRole`、`AuthorizePlatformOrAnyRole`、`ValidSourceRef`。
- 约束:刷新令牌当前由 identity 使用 `pkg/crypto.RandomToken` 随机串实现,不要在本包新增第二套 refresh JWT;内部服务请求必须带租户、source_ref、timestamp、trace_id 和签名。

### `internal/platform/db`

pgx 连接池、RLS 会话变量和事务边界。

- 用于:repo/store 装配数据库事务,按租户写入 `app.tenant_id`,受控后台维护走特权事务。
- 需求关键词:pgx、事务、RLS、tenant_id、privileged tx、no rows。
- 关键能力:`New`、`WithTenantTx`、`WithTenantTxID`、`WithAppTx`、`WithPrivilegedModuleTx`、`AppPool`、`PrivilegedPool`、`IsNoRows`。
- 约束:service 不直接 import `platform/db`;只有 repo/store 构造和事务边界使用;跨租户维护任务只能在所属模块 repo 内使用特权事务读取本模块表。

### `internal/platform/background`

后台 ticker、panic recovery、结构化错误日志和 context 停止语义。

- 用于:模块 worker/scheduler 的通用运行壳,业务状态机仍保留在各模块 service 的 `RunXOnce(ctx)`。
- 需求关键词:后台任务、ticker、worker、panic、定时清理、轮询。
- 关键能力:`Runner.Start`。
- 约束:间隔必须由配置注入且大于 0;空任务函数或非法间隔会显式记录结构化错误并停止,不要在模块里复制 ticker/recover 循环。

### `internal/platform/eventbus`

NATS 事件总线、消费重试和死信投递统一入口。

- 用于:模块间反向通知、事件发布、事件订阅、失败重试和死信。
- 需求关键词:NATS、Publish、Subscribe、事件重试、死信、trace_id、tenant_id。
- 关键能力:`Bus`、`New`、`Publish`、`Subscribe`、`Decode`。
- 约束:事件 payload 必须含真实 `tenant_id` 和 `trace_id`;空 subject、nil handler 会直接拒绝;业务模块不要自己建 NATS 客户端或绕过统一死信策略。

### `internal/platform/redis`

Redis 缓存、限频和计数原语封装。

- 用于:短信/通知限频、整数缓存、主动失效和就绪探针。
- 需求关键词:Redis、SetNX、限频、计数、缓存、TTL。
- 关键能力:`New`、`Ping`、`Close`、`SetNX`、`IncrWithTTL`、`Decr`、`GetInt64`、`SetInt64`、`Delete`。
- 约束:`IncrWithTTL` 用 Lua 原子设置首次 TTL,不要在模块里拆成 `INCR` + `EXPIRE`;不暴露底层客户端,新增 Redis 原语应先扩展本包。

### `internal/platform/storage`

MinIO 客户端、统一文件服务、对象引用和短时下载授权。

- 用于:模块上传规划、对象 key/object_ref 生成、MinIO Put/Get、下载授权 token。
- 需求关键词:MinIO、上传、附件、对象引用、object_ref、下载授权、文件服务。
- 关键能力:`Storage`、`NewServiceFromConfig`、`ObjectKey`、`ObjectRefString`、`ParseObjectRef`、`Service.PlanUpload(ctx, ...)`、`Service.IssueDownloadGrant`、`VerifyDownloadGrantToken`。
- 约束:对象 key 必须是 `{tenant_id}/{module}/{resourceType}/...`;bucket/key/object_ref 校验只有一套实现;组合根装配文件服务必须用 `NewServiceFromConfig` 收敛扫描器、扫描大小上限、下载 TTL 和签名密钥;模块不得直接暴露 MinIO 直链或自己签下载 token。

### `internal/platform/secretmap`

JSON 配置 map 中敏感字段的加密、脱敏和服务端还原。

- 用于:租户可配置外部源、适配器凭据、headers/token/password 等配置入库和返回。
- 需求关键词:配置加密、凭据字段、token/password/header、脱敏展示、Reveal。
- 关键能力:`Protect`、`Mask`、`Reveal`、`MaskedValue`。
- 约束:凭据字段识别统一走 `pkg/privacy.IsCredentialKey`;map 和数组内的嵌套配置都由本包递归处理,模块不要复制关键词表或另写加密包装格式。

### `internal/platform/upload`

跨模块上传安全边界和病毒扫描适配。

- 用于:读取大小受限的上传内容、CSV/XLSX 类型识别、ZIP/TAR 归档安全校验、ClamAV 扫描。
- 需求关键词:上传大小、CSV、XLSX、ZIP、TAR、归档遍历、病毒扫描、ClamAV。
- 关键能力:`ReadBounded`、`CheckSize`、`CSVOrXLSXKind`、`SafeArchiveEntryName`、`SafeArchiveTar`、`WalkArchiveFiles`、`VerifyScan(ctx, ...)`、`NewScannerFromConfig`。
- 约束:本包只返回通用安全原因,业务错误码由调用模块映射;扫描必须继承业务请求 `context.Context`,不要在模块里另写归档路径穿越、文件数量/展开大小、病毒扫描口径。

### `internal/platform/k8s`

client-go、dynamic client、metrics client 和容器 exec 流封装。

- 用于:M2 沙箱编排访问 Kubernetes API、读取 metrics-server、执行容器命令/终端流。
- 需求关键词:Kubernetes、client-go、metrics.k8s.io、exec、终端、Pod。
- 关键能力:`New`、`Clientset`、`Dynamic`、`Metrics`、`ImageRegistry`、`Healthz`、`Exec`、`ExecStream`。
- 约束:K8s 客户端只属于基础设施层;exec 必须带完整 namespace/pod/container/command;学生可触达命令仍由 sandbox 模块做业务权限和隔离校验。

### `internal/platform/ws`

WebSocket Hub、Origin 校验、连接生命周期和按 topic 广播。

- 用于:notify、sandbox、judge 等模块建立受控 WebSocket 连接和 topic 推送。
- 需求关键词:WebSocket、Hub、Origin、订阅、广播、单端互斥、终端流。
- 关键能力:`NewOriginPolicy`、`NewHub`、`Serve`、`ServeInteractive`、`Subscribe`、`Broadcast`、`BindSession`、`SendJSON`、`Reader`、`Writer`。
- 约束:业务模块不自建 Hub;广播是非阻塞入队并返回成功入队连接数,可靠业务状态必须仍落库或通过事件/轮询补偿。

### `internal/platform/transfer`

统一导入导出中心,管理任务状态、重试、产物引用和下载中心。

- 用于:模块创建导入/导出任务、登记完成产物、查询任务、签发下载授权。
- 需求关键词:导入、导出、任务中心、重试、下载中心、transfer_task。
- 关键能力:`Service.CreateTask`、`GetTask`、`ListTasks`、`CompleteTask`、`FailTask`、`BuildDownloadGrant`、`RegisterRoutes`。
- 约束:只放通用任务状态和文件产物边界,不要塞具体业务字段或模块私有流程;访问校验失败和角色查询错误必须显式返回。

### `internal/platform/audit`

统一审计条目构造和 actor_role 解析辅助。

- 用于:各模块成功完成敏感操作后写 identity 的唯一 `audit_log`。
- 需求关键词:审计、audit_log、actor_role、操作详情、trace_id。
- 关键能力:`Writer`、`ResolveActor`、`ActorRoleFromAccount`、`BuildEntry`、`DetailString`、`WithRequestContext`。
- 约束:模块内 `audit.go` 只包装 action/target/detail,最终写入必须经本包和 identity 实现;不要自建审计表;已通过服务签名的内部系统任务统一使用本包的 `ActorRoleSystem`,不要在 `internal/contracts` 里复制系统任务角色。

### `internal/platform/config`

环境变量读取、本地 `.env` 加载和启动期配置校验。

- 用于:进程启动读取后端实际使用的环境变量,并集中校验端口、密钥、阈值、SNOWFLAKE 节点号等。
- 需求关键词:配置、环境变量、`.env`、启动校验、SNOWFLAKE_NODE_ID。
- 关键能力:`LoadDotEnv`、`Load` 和各 `*Config` 结构。
- 约束:新增配置必须同步 `backend/.env.example` 和 `deploy/config/chaimir.env`;不要在模块里硬编码端口、密钥、超时或阈值。

### `internal/platform/jsonx` / `pgtypex` / `timex` / `pagex` / `ids` / `netx` / `tenant`

小型但全局统一的边界工具。

- 用于:JSON 严格/宽松解析、pgtype 可空值转换、UTC 时间、分页归一、字符串 ID 解析、公网/LDAPS 出站校验、租户身份上下文。
- 需求关键词:JSON、pgtype、UTC、分页、ID 字符串、SSRF、外部 URL、HTTP client、tenant context。
