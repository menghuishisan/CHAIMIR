# M4 仿真可视化引擎 — 接口设计

> RESTful + WebSocket(后端计算仿真的状态推送)。鉴权沿用 M1 JWT。
> 多数仿真前端运行,后端接口主要服务:仿真包管理、会话/操作序列持久化、后端计算仿真。
> 最后更新:2026-05-29

---

## 1. 通用约定

- Base:`/api/v1/sim`
- 统一响应体同 M1。错误码段:`4xxxx`(41xxx 仿真包/42xxx 会话/43xxx 审核)。
- `[内部]` 供 M6/M7 调用;`[用户]` 供前端。

## 2. 仿真包管理

### 2.1 查询与获取 `[用户]`
| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/packages?category=&keyword=&status=published` | 仿真包列表 |
| GET | `/packages/{code}/versions` | 某仿真的版本列表 |
| GET | `/packages/{code}/{version}/bundle` | 签发包运行授权(内置包返回 SDK 装配编码,扩展包返回短时下载授权) |

### 2.2 扩展接入 `[教师/第三方]`
| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/packages` | 提交仿真包(上传 bundle,状态=审核中) |
| GET | `/packages/{id}/preview` | 作者本人查看受控预览摘要(审核前自测) |
| POST | `/packages/{id}/validation-report` | 受控预览流程回写确定性校验报告 |
| PATCH | `/packages/{id}` | 更新草稿/新版本 |

`POST /packages` 使用 `multipart/form-data`:元数据字段只允许 `code`、`version`、`name`、`category`、`compute`、`backend_adapter`、`scale_limit`、`backend_config`,文件字段为 `bundle`;出现未知字段、重复元数据字段或重复文件字段时必须拒绝。API 层必须用 streaming multipart 逐 part 处理,元数据字段累计大小由 `UPLOAD_SIM_BUNDLE_METADATA_MAX_BYTES` 控制,bundle 文件正文大小由 `UPLOAD_SIM_BUNDLE_MAX_BYTES` 和 `internal/platform/upload.ReadBounded` 控制。bundle 根目录或唯一顶层目录内必须包含 `sim-package.json`,其中 `meta` 必须与上传表单一致,`interactions` 会提取为服务端操作白名单,`render.patterns` 只能使用封闭模式集。教师入口的 `author_type`、`author_id` 与 code 命名空间由后端按登录账号强制设为 `AuthorTeacher` + `teacher_<account_id>__`,请求结构不得包含作者类型字段,不接受客户端伪造 `builtin__`、`org_` 或其他作者类型。后端必须计算 bundle sha256,不得信任客户端提交的 hash;同时执行危险调用静态扫描,扫描不过不得进入审核。M4 在通过版本冲突、作者归属和包状态校验后才写入对象存储,避免业务校验失败留下孤儿 bundle;若对象写入后仍失败,对应包会被显式退回并留下审核轨迹,不会以待审态悬挂。对象存储删除能力统一归 `internal/platform/storage` 扩展,M4 不自建删除通道。

`POST /packages/{id}/validation-report` 只能由受控预览流程通过内部服务签名调用,只允许补充动态审核字段(`determinism_check`、`worker_preview` 及其明细)。平台管理员只执行审核决策,不得手工回写预览结论。`static_scan`、`bundle_hash`、`metadata_validation` 等字段由后端上传扫描生成,后续回写不得覆盖。`approve` 前必须满足 `metadata_validation=passed`、`static_scan=passed`、`determinism_check=passed` 且 `worker_preview=passed`,否则返回 `41005`。

`GET /packages/{id}/preview` 仅允许教师包提交者本人读取自己的最新审核报告;平台管理员通过 `/reviews` 查看待审列表和报告。任意教师或学校管理员不得按 package id 横向读取其他作者的待审报告。

`GET /packages/{code}/{version}/bundle` 不直接返回 MinIO 正文或公开直链。接口在校验包已发布和当前用户已登录后签发运行授权。平台内置且 `compute=frontend` 的包返回 `{ builtin_code, bundle_hash, expires_at }`,前端只能把 `builtin_code` 交给平台维护的 sim-sdk Worker 内部装配,不得在业务页面复制内置包清单或绕过发布目录。教师/第三方扩展包仍通过 `internal/platform/storage` 统一文件服务返回短时下载授权 `{ token, bundle_hash, expires_at }`,前端再交给平台统一对象下载入口读取 bundle;服务端不向前端暴露对象存储内部 grant 结构。

### 2.3 审核 `[平台管理员]`
| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/reviews?result=pending` | 待审核列表 |
| POST | `/reviews/{id}/approve` | 通过 → 上架 |
| POST | `/reviews/{id}/reject` | 退回 + 意见 |

### 2.4 生命周期管理 `[平台管理员]`
| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/packages/{id}/archive` | 已上架包下架,不删除历史会话与回放 |
| POST | `/packages/{id}/republish` | 已下架包重新上架 |

> 下架只影响新会话创建和普通列表可见性;历史会话、操作序列、分享剧本仍按会话权限与分享码读取。重新上架仅允许从已下架状态恢复,不得绕过审核把草稿、审核中或退回包直接发布。

## 3. 仿真会话 `[内部/用户]`

### 3.1 创建会话
`POST /sessions` `[内部]`
请求:
```json
{ "package_code": "builtin__pow-mining", "version": "1.0.0",
  "seed": 20260529, "init_params": { "minerCount": 10 },
  "owner_account_id": 1001, "source_ref": "experiment:2026:instance:55" }
```
响应:`{ "session_id": 7001, "compute": "frontend", "bundle_ref": "..." }`
> compute=frontend:前端在隔离 Worker 中运行仿真包;后端仅记录会话与操作序列。
> `bundle_ref` 是经 M4 鉴权后返回的逻辑引用,不是可绕过鉴权的公开直链。内置包使用 `builtin://sim-sdk/{code}@{version}` 表示平台内部装配来源;扩展包使用对象存储逻辑引用。前端运行时必须通过 `/packages/{code}/{version}/bundle` 取得运行授权后再启动 Worker。
> 内部服务接口必须使用服务签名头携带 `source_ref`;单会话回收与检查点写入会校验该会话 `source_ref` 与签名来源一致,禁止调用方用合法服务签名篡改其他来源的会话。

### 3.2 上报操作 `[用户]`
`POST /sessions/{id}/actions`
请求:`{ "seq": 12, "at_tick": 34, "event_type": "launch-51", "payload": { "blocks": 6 } }`
> 前端运行时把用户操作异步上报,用于回放/分享/判分;不阻塞交互。
> `seq` 在同一会话内必须连续递增;重复上报同一 `seq` 且内容一致视为幂等成功,内容不同返回 `42004`。

### 3.3 回放数据
`GET /sessions/{id}/replay`
响应:`{ "seed": 20260529, "init_params": {...}, "actions": [ ... ] }`
> 前端用 seed + actions 重放复现。

### 3.5 回收会话 `[内部]`
- `DELETE /sessions/{id}` `[内部]`:回收单个仿真会话(归档操作序列,释放后端计算资源)。
- `POST /sessions/recycle` `[内部]`:按 `source_ref` 批量回收(来源结束级联,与 M2 `/sandboxes/recycle` 对齐)。
> 前端仿真无后端常驻资源,回收主要归档会话状态;compute=backend 的仿真同时释放后端执行资源。M7 实例回收时调用本接口释放其引用的仿真会话。

### 3.4 分享剧本
- `POST /sessions/{id}/share` → 生成分享码。
- `GET /shared/{code}` → 取 `(package, version, seed, init_params, actions)` 复现。
响应:
```json
{
  "package_code": "builtin__pow-mining",
  "version": "1.0.0",
  "seed": 20260529,
  "init_params": {},
  "actions": []
}
```
> 公开分享剧本只返回复现所需的公开参数。`init_params` 和 action `payload` 中的答案、flag、token、密钥、凭据、私有字段以及 `_` 开头内部绑定字段必须过滤;完整回放仍只能通过租户内授权接口读取。

## 4. 后端计算仿真 `[用户]`(仅 compute=backend)

少数重计算仿真在后端跑,经 WS 推状态。M4 后端计算只使用 M4 自有的受控后端适配器或受控仿真镜像会话,不直接调用 M2 沙箱接口,以满足《工程目录设计》中 `sim/sandbox/content` 同层互不依赖的边界。

当前组合根未注册任何生产后端计算适配器时,`compute=backend` 包提交、审核通过和 WS 连接统一返回 `42005`,不得以空适配器或伪实现通过审核。只有装配层注册真实 `BackendAdapter` 后,对应包才能声明该 `backend_adapter` 并进入审核。

`WS /sessions/{id}/stream`
- 浏览器客户端通过 M1 `POST /api/v1/auth/ws-ticket` 为该 WS path 换取短时 `ticket`,不得在 URL 中携带 access token。
- 客户端发:`{ "event_type": "...", "payload": {...} }`(交互事件)
- 服务端推:`{ "tick": N, "state": {...} }`(新状态,前端仅渲染)
> 协议与前端 reducer 一致,仅运行位置不同。

## 5. 检查点上报 `[内部]`

`POST /sessions/{id}/checkpoints`
请求:`{ "checkpoint_id": "cp-51-success", "answer": "能", "achieved": true }`
> 叙事设问/目标达成结果上报,M3 仿真检查点判题器据此判分。
> 同一会话同一 `checkpoint_id` 重复上报按最新结果覆盖;M4 只保存结果快照,不判分、不保存正确答案。

## 6. 典型时序(实验内仿真,前端运行)

```
M7 → POST /sessions(package+seed+params+source_ref)
M4 → session_id + bundle_ref(compute=frontend)
前端 → 加载 bundle,本地运行状态机
学生 → 交互(滑块/按钮/注入攻击)→ 本地即时演化
前端 → 异步 POST /actions 上报操作序列
叙事设问 → POST /checkpoints → M3 判分
完成 → 会话归档;可 /share 分享剧本
```
