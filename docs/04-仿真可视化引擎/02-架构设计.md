# M4 仿真可视化引擎 — 架构设计

> 核心:自描述仿真插件框架、确定性状态机、前端运行时、可视化 SDK、并排对比。
> 与 M2 链插件、M3 判题器插件同一哲学:平台供框架,插件供具体。
> 最后更新:2026-05-29

---

## 1. 总体架构

```
┌──────────────────────────────────────────────────────────┐
│  调用方:M7 实验 / M6 教学(嵌入课时) / M3(取检查点判分)    │
│  加载(仿真包 + 初始参数 + 种子 + 来源标识)                  │
└─────────────────────────┬────────────────────────────────┘
                          │
┌─────────────────────────▼────────────────────────────────┐
│  M4 前端(React)                                           │
│  ┌──────────────┬───────────────┬─────────────────────┐  │
│  │ 仿真运行时     │ 可视化 SDK     │ 通用交互渲染器        │  │
│  │ (状态机驱动)   │ (画布+复合组件)│ (按声明渲染控件)      │  │
│  └──────┬───────┴───────┬───────┴──────────┬──────────┘  │
│         │ 加载/驱动       │ 渲染              │ 交互事件      │
│  ┌──────▼────────────────▼──────────────────▼──────────┐  │
│  │           仿真包(自描述插件,前端运行)                  │  │
│  │   state · interactions · view · narrative             │  │
│  └───────────────────────────────────────────────────────┘ │
│  叙事控制器 │ 操作序列记录器(种子+事件)│ 回放器              │
└─────────────────────────┬────────────────────────────────┘
                          │ (仅插件管理/重计算时)
┌─────────────────────────▼────────────────────────────────┐
│  M4 后端(Go,轻量)                                        │
│  仿真包注册/审核/版本 │ 重计算仿真执行(可选) │ 会话/序列持久化 │
└───────────────────────────────────────────────────────────┘
```

**关键**:仿真逻辑默认全在前端运行,后端只管插件管理与持久化;仅声明为"重计算"的仿真才在后端执行同一协议。

## 2. 自描述仿真插件框架(核心)

### 2.1 仿真包结构

一个仿真包 = 自描述四件事(详 §3~§6):

```
SimPackage {
  meta:         { code, name, category, version, compute: "frontend"|"backend" }
  state:        初始状态结构 + reducer(state, event, tick) → newState   // 确定性
  interactions: InteractionDef[]   // 声明式交互列表(通用,见 §4)
  view:         (state) → ViewSpec  // 用 SDK 描述怎么画
  narrative?:   NarrativeStep[]     // 可选教学叙事
}
```

### 2.2 与平台的契约

- 平台提供:运行时(加载+驱动)、SDK(渲染)、交互渲染器(控件)、生命周期(审核上架)。
- 仿真包遵守:reducer 必须是**纯函数**(同输入同输出),view 必须只用 SDK 能力,interactions 必须声明完整。
- 新仿真 = 写一个包 + 审核上架,**平台代码零改动**。

## 3. 确定性状态机

### 3.1 模型

```
状态演化:  newState = reducer(state, event, tick)
事件来源:  ① tick 推进(时间)  ② 学生/教师交互(注入事件)
随机:      种子化 PRNG,seed 固定 → 演化唯一
```

### 3.2 确定性的红利

| 能力 | 实现 |
| --- | --- |
| 单步回退 | 从初始 + 种子重算到 tick-1(不存状态快照) |
| 回放 | 重放 `(seed, event[])` |
| 分享剧本 | 导出 `(seed, event[])`,他人复现 |
| M3 判分 | 检查点读确定状态;操作序列可复现核验 |

> 用"存操作序列"替代旧版"存状态快照",轻量且天然支持回放/分享。

## 4. 通用交互协议(彻底不固定)

### 4.1 InteractionDef 声明

```
InteractionDef {
  id, kind,            // kind: button|slider|hold|select-element|drag|form
  params: FieldDef[],  // 任意字段 + 约束(范围/枚举/默认)
  target: "global"|"element",
  available_when?,     // 状态/阶段条件
  label_tag?           // normal|perturb|attack(仅视觉:配色+二次确认)
  emits: eventType     // 触发后注入的事件类型
}
```

### 4.2 通用交互渲染器

平台按 `kind` + `params` **自动渲染控件**,无需仿真写 UI:

| kind | 渲染 |
| --- | --- |
| button | 按钮,点击 emit 事件 |
| slider | 滑块调参(FieldDef range) |
| hold | 按住持续 emit(如持续 spam 交易) |
| select-element | 先点选画布元素,再施加操作 |
| drag | 拖拽(如连线、移动) |
| form | 字段表单提交 |

→ 任意仿真声明任意交互都能直接用;攻击只是 `label_tag=attack` 的普通交互。

## 5. 可视化模式引擎(核心,替代旧版"图元+自定义渲染器")

> 这是 M4 防止"重蹈旧版 SimEngine 覆辙"的关键设计。旧版在"几何图元(node/edge/grid_cell)"层抽象,太低层、作者负担重,且留了"L3 自定义渲染器"的敞开后门 → 越扩展越碎片化。M4 改为在**可视化模式**层抽象。

### 5.1 什么是可视化模式

区块链教学可视化的隐喻千变万化,但**底层骨架反复出现的就那几种**。平台提供一组**封闭的、可配置的"可视化模式"**,每种是一个完整的渲染引擎(负责布局+绘制+动画),不是固定组件:

| 模式 | 骨架 | 覆盖的仿真类型 |
| --- | --- | --- |
| 图网络 Graph | 节点 + 边,可配布局(环形/力导向/网格) | P2P 网络、共识节点、状态机、拓扑 |
| 链式 Chain | 线性/分叉的块序列 | 区块链、最长链、分叉、双花、自私挖矿 |
| 树形 Tree | 树结构 | 默克尔树、MPT、分叉树 |
| 矩阵 Matrix | 网格单元 | 投票矩阵、位运算、Bloom 过滤器 |
| 流水线 Pipeline | 分步数据流 | 哈希计算、加解密、交易生命周期 |
| 时序泳道 Lane | 多方时间轴消息 | PBFT 三阶段、多方通信、跨链 |
| 图表 Chart | 数值曲线/柱/饼 | 算力、质押权重、gas、价格曲线 |

**仿真作者不"拼图元",而是"把状态映射到一种或几种模式"。** 例:PBFT = 图网络(环形节点)+ 时序泳道(三阶段消息)+ 矩阵(投票统计),组合即成,零渲染代码。

### 5.2 两条架构红线(不重蹈覆辙的真正保证)

**红线一:封闭模式集 + 严进的扩展闸门**
- 可视化模式是**有限的、平台维护的封闭集合**(初定 7 种)。仿真作者**只能选用组合,不能自带渲染器**——彻底关闭旧版"L3 自定义渲染器"那个导致碎片化的后门。
- 确需新模式?走**平台级评审**(等同新增一个链运行时的严肃度),通过后成为**所有仿真的公共资产**。扩展是"往主干加肌肉",不是"各长各的枝杈"。

**红线二:模式是渲染引擎,作者只给语义数据**
- 作者只声明**语义数据**(有哪些节点、谁给谁发了消息、当前什么阶段、投票结果),**布局/绘制/动画全由模式引擎负责**,作者碰不到渲染细节。
- 结果:可视化质量由平台的模式引擎**统一保证**,不依赖每个作者的水平(旧版正是因为场景质量由作者水平决定而参差不齐)。
- 红利:优化某个模式的动画(如时序泳道展示"收集到 2f+1 票的瞬间高亮"),**所有用该模式的仿真自动变好**,无需逐个改。

### 5.3 模式映射示例

```
PBFT 仿真包声明:
  state: { nodes[], phase, votes[], byzantine[] }
  patterns:
    - graph:  { nodes: state.nodes, layout:'ring',
                edges: 由消息事件动态生成 }      // 节点环 + 消息飞线
    - lane:   { stages:['Pre-Prepare','Prepare','Commit'], current: state.phase }
    - matrix: { cells: state.votes }              // 投票矩阵
  // 不含任何"怎么画一个圆/一条线"的代码
```

## 6. 教学叙事层(可选)

```
NarrativeStep {
  trigger,             // 到某 tick / 某状态 / 学生操作后
  highlight?,          // 高亮哪些状态元素
  explain?,            // 解说"为什么"
  question?            // 设问:学生预测 → 揭晓(结果可供 M3)
}
```

- 叙事与仿真解耦:同一仿真可挂不同叙事(入门版/进阶版)。
- 纯沙盒探索可不写叙事。

---

## 7. 并排对比(轻量,替代旧版复杂联动)

- **多仿真并排**:同屏放几个独立仿真会话,各自独立演化,不共享状态。
- **参数对比**:同一仿真包跑两个不同参数/种子的实例并排(如"有51%攻击 vs 无攻击"),直观对比。
- 不做旧版跨画布因果弧线、owner 共享状态空间。真要联动效果 → 用参数对比表达。

## 8. 前端为主 + 后端可选

| 场景 | 运行位置 | 理由 |
| --- | --- | --- |
| 绝大多数教学仿真 | 前端 | 计算量小,零延迟交互,秒开,省成本 |
| 极少数重计算仿真 | 后端(声明 compute=backend) | M4 自有受控后端适配器按同一事件/状态协议运行,前端只渲染 |

协议一致 → 前后端可切换,不增加仿真作者心智。

后端计算边界:为遵守《工程目录设计》§3.1.1 的同层互不依赖规则,M4 不直接调用 M2 沙箱接口。`compute=backend` 的仿真包必须声明 `backend_adapter` 与资源协议配置,由 M4 内部适配器注册表承接;适配器可以对接平台运维层预置的受控仿真镜像会话,但不得 import 或调用 `internal/modules/sandbox`。需要与真实链沙箱联动的数据由 M7 在业务编排层采集后作为输入事件或初始参数传给 M4。

`backend_config` 复用镜像层统一 WorkloadSpec 的资源、命令、探针、卷和安全字段语义,但执行权属于 M4 的 `BackendAdapter`。M4 可以使用受控 `sim/*` 镜像运行重计算仿真,也可以使用等价隔离执行环境;两者都必须执行同一资源上限、网络 deny-all、non-root、超时和确定性校验。禁止把 M2 的沙箱 WorkloadSpec 当作 M4 后端计算兼容入口,也禁止在未注册真实适配器时把后端计算降级为空实现。

生产约束:`compute=backend` 不是降级为前端运行或空回显的兼容开关。组合根未注册真实 `BackendAdapter` 时,后端必须拒绝该类包提交、审核通过与 WS 连接,统一返回后端计算不可用错误;注册适配器后仍需通过同一静态扫描、确定性预览和资源隔离审核。

## 9. 与其他模块协作

- **M7 实验/M6 教学**:嵌入仿真会话,传仿真包+参数+种子。
- **M3 评测**:仿真检查点(设问/目标达成)作为判题输入,操作序列可复现核验。
- **M5 题库**:仿真相关题面、正确答案与评分配置归 M5/M3,M4 只保存仿真包元信息、会话、操作序列、检查点结果快照。
- **M1**:仿真会话/操作序列带 tenant_id 隔离。
