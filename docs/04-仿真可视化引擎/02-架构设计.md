# M4 仿真可视化引擎 — 架构设计

> 核心:自描述仿真插件框架、确定性状态机、TeachingFrame 教学画面协议、封闭可视化模式、前端运行时、并排对比。
> 与 M2 链插件、M3 判题器插件同一哲学:平台供框架,插件供具体。
> 最后更新:2026-07-06

## 1. 总体架构

```
┌──────────────────────────────────────────────────────────┐
│  调用方:M7 实验 / M6 教学(嵌入课时) / M3(取检查点判分)    │
│  加载(仿真包 + 初始参数 + 种子 + 来源标识)                  │
└─────────────────────────┬────────────────────────────────┘
                          │
┌─────────────────────────▼────────────────────────────────┐
│  M4 前端(React)                                           │
│  ┌──────────────┬───────────────┬─────────────────────┐  │
│  │ 仿真运行时     │ TeachingFrame  │ 通用交互渲染器        │  │
│  │ (状态机驱动)   │ (教学画面协议) │ (按声明渲染控件)      │  │
│  └──────┬───────┴───────┬───────┴──────────┬──────────┘  │
│         │ 加载/驱动       │ 渲染              │ 交互事件      │
│  ┌──────▼────────────────▼──────────────────▼──────────┐  │
│  │           仿真包(自描述插件,前端运行)                  │  │
│  │   state · interactions · render · narrative            │  │
│  └───────────────────────────────────────────────────────┘ │
│  叙事控制器 │ 操作序列记录器(种子+事件)│ 回放器              │
└─────────────────────────┬────────────────────────────────┘
                          │ (仅插件管理/重计算时)
┌─────────────────────────▼────────────────────────────────┐
│  M4 后端(Go,轻量)                                        │
│  仿真包注册/审核/版本 │ 重计算仿真执行(可选) │ 会话/序列持久化 │
└───────────────────────────────────────────────────────────┘
```

**关键**:仿真逻辑默认全在前端运行,后端只管插件管理与持久化;仅声明为"重计算"的仿真才在后端执行同一协议。

## 2. 自描述仿真插件框架(核心)

### 2.1 仿真包结构

一个仿真包 = 自描述四件事(详 §3~§6):

```
SimPackage {
  meta:         { code, name, category, version, compute: "frontend"|"backend" }
  state:        初始状态结构 + reducer(state, event, tick) → newState   // 确定性
  interactions: InteractionDef[]   // 声明式交互列表(通用,见 §4)
  render:       (state) → TeachingFrame  // 声明当前教学画面
  narrative?:   NarrativeStep[]     // 可选教学叙事
}
```

### 2.2 与平台的契约

- 平台提供:运行时(加载+驱动)、TeachingFrame 解释器、封闭模式渲染器、交互渲染器(控件)、生命周期(审核上架)。
- 仿真包遵守:reducer 必须是**纯函数**(同输入同输出),render 必须只输出 TeachingFrame 纯数据,interactions 必须声明完整。
- 新仿真 = 写一个包 + 审核上架,**平台代码零改动**。

## 3. 确定性状态机

### 3.1 模型

```
状态演化:  newState = reducer(state, event, tick)
事件来源:  ① tick 推进(时间)  ② 学生/教师交互(注入事件)
随机:      种子化 PRNG,seed 固定 → 演化唯一
```

### 3.2 确定性的红利

| 能力 | 实现 |
| --- | --- |
| 单步回退 | 从初始 + 种子重算到 tick-1(不存状态快照) |
| 回放 | 重放 `(seed, event[])` |
| 分享剧本 | 导出 `(seed, event[])`,他人复现 |
| M3 判分 | 检查点读确定状态;操作序列可复现核验 |

> 用"存操作序列"替代旧版"存状态快照",轻量且天然支持回放/分享。

## 4. 通用交互协议(彻底不固定)

### 4.1 InteractionDef 声明

```
InteractionDef {
  id, kind,            // kind: button|slider|hold|select-element|drag|form
  params: FieldDef[],  // 任意字段 + 约束(范围/枚举/默认)
  target: "global"|"element",
  available_when?,     // 状态/阶段条件
  label_tag?           // normal|perturb|attack(仅视觉:配色+二次确认)
  emits: eventType     // 触发后注入的事件类型
}
```

### 4.2 通用交互渲染器

平台按 `kind` + `params` **自动渲染控件**,无需仿真写 UI:

| kind | 渲染 |
| --- | --- |
| button | 按钮,点击 emit 事件 |
| slider | 滑块调参(FieldDef range) |
| hold | 按住持续 emit(如持续 spam 交易) |
| select-element | 先点选画布元素,再施加操作 |
| drag | 拖拽(如连线、移动) |
| form | 字段表单提交 |

→ 任意仿真声明任意交互都能直接用;攻击只是 `label_tag=attack` 的普通交互。

## 5. TeachingFrame 教学画面协议(核心)

> 这是 M4 的稳定底层框架。仿真包不是输出一堆可视化组件,而是输出“这一刻学生应该怎么看”的教学画面。平台负责把教学画面解释成主舞台、右侧辅助区、底部时间控制和统一动画。

### 5.1 为什么不是旧版区域布局协议

旧版区域布局协议只回答“有哪些模式、放在哪里”,无法表达“学生当前应该关注谁、历史消息何时淡出、为什么这一阶段重要”。结果很容易出现新旧步骤全部堆叠、图边越来越多、右侧信息拥挤等问题。

M4 改为 `TeachingFrame`:状态机每次 render 输出当前教学画面。画面必须同时声明:

- 当前阶段是什么,学生此刻要观察什么。
- 哪个模式是主可视化,哪些模式只是证据、时间线、指标或追踪。
- 哪些元素是焦点、上下文、历史或应淡化对象。
- 模式数据仍然只使用平台维护的七类封闭模式。

### 5.2 TeachingFrame 结构

```typescript
interface SimPackage {
  render: (state) => TeachingFrame;
}

interface TeachingFrame {
  summary: string;
  phase: FramePhase;
  focus: FrameFocus;
  layout: FrameLayout;
  patterns: VisualPattern[];
  annotations?: FrameAnnotation[];
}

interface FramePhase {
  id: string;
  title: string;
  intent: "observe" | "compare" | "verify" | "debug" | "attack" | "recover" | "replay";
  explanation: {
    what: string;   // 当前发生什么
    why: string;    // 为什么重要
    watch: string;  // 学生应该看哪里
  };
}

interface FrameFocus {
  primary: string[];    // 本帧核心对象
  secondary?: string[]; // 辅助观察对象
  muted?: string[];     // 保留但降低视觉权重的对象
}

interface FrameLayout {
  primary: string;       // 主舞台 pattern id
  evidence?: string[];   // 右侧证据 pattern id
  timeline?: string;     // 右侧或底部时间证据 pattern id
  metrics?: string[];    // 右侧指标 pattern id
  trace?: string;        // 代码追踪锚点
  checkpoints?: string[];// 检查点锚点
}
```

### 5.3 元素生命周期

每个模式内可渲染元素都可以携带统一生命周期,平台用它控制动画、淡出、历史保留和无障碍描述。生命周期不是前端兜底过滤,而是算法视图的教学语义输出。

```typescript
interface VisualElementMeta {
  id: string;
  label: string;
  role?: string;
  lifecycle: {
    state: "entering" | "active" | "settled" | "leaving" | "archived";
    fromTick: number;
    toTick?: number;
  };
  emphasis: "focus" | "context" | "history" | "ghost";
  explanation?: string;
}
```

要求:

- 正在发生的消息、投票、区块、证明路径必须是 `active/focus`。
- 已完成但仍有教学价值的对象使用 `settled/history`。
- 不再参与本阶段判断的对象使用 `archived/ghost`,由平台统一淡化或折叠。
- 不允许算法把所有历史边、历史消息无差别塞进主视图。

### 5.4 封闭可视化模式集

区块链教学可视化的具体算法不同,但可视化骨架稳定收敛为七类。`VisualPattern` 是 TeachingFrame 的材料,不再携带页面区域字段。

```typescript
interface VisualPattern {
  id: string;
  mode: "graph" | "chain" | "tree" | "matrix" | "pipeline" | "lane" | "chart";
  title: string;
  data: unknown;
}
```

| 模式 | 骨架 | 覆盖的仿真类型 |
| --- | --- | --- |
| 图网络 Graph | 节点 + 边 + 消息过程 | P2P 网络、共识节点、状态机、拓扑 |
| 链式 Chain | 线性/分叉的块序列 | 区块链、最长链、分叉、双花、自私挖矿 |
| 树形 Tree | 树结构 + 路径 | Merkle Tree、MPT、证明路径 |
| 矩阵 Matrix | 网格单元 | 投票矩阵、校验矩阵、权限矩阵 |
| 流水线 Pipeline | 分步数据流 | 哈希、签名、交易生命周期、EVM 执行 |
| 时序泳道 Lane | 多方时间轴消息 | PBFT、跨链消息、中继、调用栈 |
| 图表 Chart | 数值趋势 | 算力、质押权重、gas、价格、延迟 |

### 5.5 两条架构红线

**红线一:封闭模式集 + 严进扩展闸门**
- 仿真作者只能组合平台封闭模式,不能自带渲染器。
- 确需新模式时,走平台级评审;通过后成为所有仿真的公共资产。

**红线二:作者给教学语义,平台负责表现**
- 作者输出阶段、焦点、布局意图、元素生命周期和模式语义数据。
- 布局、绘制、动画、响应式、无障碍由平台统一处理。
- 优化某个模式或 TeachingFrame 解释器时,所有内置包、教师包和第三方包自动受益。

### 5.6 模式组合示例

PBFT 不是“固定页面模板”,而是一组面向当前阶段的教学画面:

```typescript
render(state) => ({
  summary: "Prepare 阶段正在收集 2f+1 票",
  phase: {
    id: "prepare",
    title: "Prepare 投票",
    intent: "verify",
    explanation: {
      what: "副本节点广播 Prepare 消息。",
      why: "节点需要确认同一请求已被足够多副本接受。",
      watch: "观察主舞台中活跃消息是否覆盖到法定人数。"
    }
  },
  focus: {
    primary: ["prepare-n1-n2", "prepare-n1-n3"],
    secondary: ["node-1", "node-2", "node-3"],
    muted: ["pre-prepare-old"]
  },
  layout: {
    primary: "pbft-graph",
    evidence: ["pbft-vote-matrix"],
    timeline: "pbft-lane"
  },
  patterns: [
    { id: "pbft-graph", mode: "graph", title: "副本消息传播", data: graphData },
    { id: "pbft-lane", mode: "lane", title: "三阶段消息时序", data: laneData },
    { id: "pbft-vote-matrix", mode: "matrix", title: "Prepare/Commit 投票矩阵", data: matrixData }
  ]
})
```

这里平台不会猜“第一个 pattern 是主视图”,也不会从模式自身推断布局;主舞台和辅助区完全由 `layout` 明确声明。

## 6. 教学叙事层(可选)

```
NarrativeStep {
  trigger,             // 到某 tick / 某状态 / 学生操作后
  highlight?,          // 高亮哪些状态元素
  explain?,            // 解说"为什么"
  question?            // 设问:学生预测 → 揭晓(结果可供 M3)
}
```

- 叙事与仿真解耦:同一仿真可挂不同叙事(入门版/进阶版)。
- 纯沙盒探索可不写叙事。

## 7. 并排对比(轻量,替代旧版复杂联动)

- **多仿真并排**:同屏放几个独立仿真会话,各自独立演化,不共享状态。
- **参数对比**:同一仿真包跑两个不同参数/种子的实例并排(如"有51%攻击 vs 无攻击"),直观对比。
- 不做旧版跨画布因果弧线、owner 共享状态空间。真要联动效果 → 用参数对比表达。

## 8. 前端为主 + 后端可选

| 场景 | 运行位置 | 理由 |
| --- | --- | --- |
| 绝大多数教学仿真 | 前端 | 计算量小,零延迟交互,秒开,省成本 |
| 极少数重计算仿真 | 后端(声明 compute=backend) | M4 自有受控后端适配器按同一事件/状态协议运行,前端只渲染 |

协议一致 → 前后端可切换,不增加仿真作者心智。

后端计算边界:为遵守《工程目录设计》§3.1.1 的同层互不依赖规则,M4 不直接调用 M2 沙箱接口。`compute=backend` 的仿真包必须声明 `backend_adapter`,由 M4 内部能力注册表承接;能力编号只引用部署期已登记的受控镜像配置,不得由教师提交镜像、命令、环境变量或资源参数。适配器可以对接平台运维层预置的受控仿真镜像会话,但不得 import 或调用 `internal/modules/sandbox`。需要与真实链沙箱联动的数据由 M7 在业务编排层采集后作为输入事件或初始参数传给 M4。

当前 `SIM_BACKEND_STDIO_ADAPTERS_JSON` 登记 `graph-layout-stdio`:组合根按配置构造共享 `stdio-json` K8s 适配器实例,M4 使用 `internal/platform/k8s` 为每个实时会话创建独立命名空间、deny-all NetworkPolicy 和受限计算 Pod,通过标准输入/输出调用已证明的 `sim/graph-layout` digest 镜像。初始化参数先生成 `tick=0` 状态,后续每个已通过交互 schema 的事件生成下一状态;断线、失败、归档均删除会话命名空间。该实现只复用平台 K8s 客户端,不依赖 M2 编排器。

标准 `stdio-json` 能力不接受包级 `backend_config`;镜像、无 shell 命令、受控环境变量、资源 requests/limits、超时和 I/O 上限只来自 `SIM_BACKEND_STDIO_ADAPTERS_JSON`,并在应用启动时完成严格 JSON、Kubernetes quantity、能力编号唯一性、digest 证明与命令边界校验。共享适配器同时读取已审核仿真包的 `scale_limit(nodes/max_tick/max_events)`,统一限制状态节点规模和单会话执行步数,不得由算法目录另建一套阈值。能力环境变量不得包含密码、token、私钥或其他凭据,算法容器无外部网络且不承担第三方鉴权。M4 可以使用受控 `sim/*` 镜像运行重计算仿真,也可以为不同协议实现等价隔离执行环境;两者都必须执行网络 deny-all、non-root、超时和确定性校验。禁止把 M2 的沙箱 WorkloadSpec 当作 M4 后端计算兼容入口,也禁止在未注册真实适配器时把后端计算降级为空实现。

扩展规则:适配器按执行协议复用,算法按镜像隔离。新增同为 `stdio-json` 的重计算算法时,只能新增 `sim/<package-code>` 镜像、manifest、digest 证明、能力目录项和对应 `sim_package`,不得复制 Go 适配器或 Kubernetes 编排代码;只有协议和生命周期确实不同才新增 `BackendAdapter` 实现。

生产约束:`compute=backend` 不是降级为前端运行或空回显的兼容开关。提交、审核通过、重新上架与 WS 连接都必须命中注册表并通过适配器配置校验,否则统一返回后端计算不可用或包参数无效;注册适配器后仍需通过同一静态扫描、确定性预览和资源隔离审核。

## 9. 与其他模块协作

- **M7 实验/M6 教学**:嵌入仿真会话,传仿真包+参数+种子。
- **M3 评测**:仿真检查点(设问/目标达成)作为判题输入,操作序列可复现核验。
- **M5 题库**:仿真相关题面、正确答案与评分配置归 M5/M3,M4 只保存仿真包元信息、会话、操作序列、检查点结果快照。
- **M1**:仿真会话/操作序列带 tenant_id 隔离。
