# M4 仿真可视化引擎 — 代码追踪与阶段编排补充设计

> 本文是对原 M4 文档的增强补充,新增两大特性:代码执行追踪、阶段式参数传递。
> 核心原则:不破坏确定性、不增加复杂度、提升教学价值。
> 最后更新:2026-07-06

---

## 1. 设计动机

### 1.1 为什么需要代码追踪？

**教学痛点**:
- 学生看仿真动画,但不理解"背后的代码怎么执行"
- 抽象概念(如 require 检查、哈希计算)缺乏具象锚点
- 无法建立"可视化现象 ↔ 代码逻辑"的因果联系

**方案**:仿真执行时实时高亮对应代码行,类似 IDE 断点调试,但在教学仿真环境中。

### 1.2 为什么需要阶段编排？

**教学痛点**:
- 复杂实验需要分步骤(先理解共识 → 再学攻击 → 最后综合)
- 后续步骤需要前置结果(如:攻击仿真需要知道共识的初始状态)
- 现有方案只能写巨型仿真包,不可复用

**方案**:M7 支持多阶段实验配置,阶段间通过检查点结果传递参数,保持确定性。

## 2. 代码追踪功能设计

### 2.1 仿真包扩展

```typescript
interface SimPackage {
  meta: SimMeta;
  initState: (params, seed) => State;
  reducer: (state, event, tick) => State;
  interactions: InteractionDef[];
  render: (state) => TeachingFrame;
  narrative?: NarrativeStep[];
  
  // 新增:代码追踪配置
  codeTrace?: CodeTraceDef;
}

interface CodeTraceDef {
  // 源代码(Solidity/Rust/Go/伪代码)
  sourceCode: string;
  
  // 语言类型(用于语法高亮)
  language: "solidity" | "rust" | "go" | "javascript" | "pseudocode";
  
  // 行号 → 状态条件映射
  lineMapping: LineMapping[];
  
  // 变量监视配置
  variableWatch?: VariableWatchDef[];
}

interface LineMapping {
  line: number;                    // 代码行号(1-based)
  triggerCondition: string;        // 触发条件表达式
  annotation?: string;             // 该行的教学注释
  highlightStyle?: "normal" | "success" | "error";  // 高亮样式
}

interface VariableWatchDef {
  name: string;                    // 变量名
  extract: string;                 // 从状态提取表达式
  format?: "hex" | "number" | "string" | "bool";
}
```

### 2.2 Reducer 返回追踪信息

```typescript
// Reducer 返回的状态必须包含 _trace 字段
interface State {
  // ... 仿真自定义字段
  
  // 追踪信息(由 reducer 填充)
  _trace?: {
    triggeredLines: number[];              // 当前触发的代码行
    variables: Record<string, any>;        // 当前变量值
    executionPath?: string;                // 可选:执行路径描述
  };
}
```

### 2.3 前端渲染

前端运行时自动读取 `state._trace` 并渲染:
- 代码面板高亮 `triggeredLines` 中的行
- 变量监视器显示 `variables` 的值
- 根据 `lineMapping` 显示注释

### 2.4 示例:PoW 挖矿仿真

```typescript
const powMiningTrace: SimPackage = {
  meta: { code: "pow-mining-traced", name: "PoW 挖矿(含代码追踪)" },
  
  codeTrace: {
    sourceCode: `contract Miner {
  uint256 public difficulty = 4;
  bytes32 public lastHash;

  function mine(uint256 nonce) external {
    bytes32 h = keccak256(abi.encodePacked(block.number, nonce));
    require(uint256(h) < (2**256 - 1) >> difficulty, "not solved");
    lastHash = h;
    emit BlockMined(msg.sender, nonce, h);
  }
}`,
    language: "solidity",
    lineMapping: [
      { line: 6, triggerCondition: "state.phase === 'hashing'", 
        annotation: "矿工正在计算哈希" },
      { line: 7, triggerCondition: "state.lastAttempt === 'failed'", 
        annotation: "难度检查失败,继续尝试",
        highlightStyle: "error" },
      { line: 7, triggerCondition: "state.lastAttempt === 'success'", 
        annotation: "✅ 找到有效哈希!",
        highlightStyle: "success" },
      { line: 9, triggerCondition: "state.newBlock != null", 
        annotation: "新区块已挖出并广播" }
    ],
    variableWatch: [
      { name: "nonce", extract: "state.currentNonce", format: "number" },
      { name: "hash", extract: "state.currentHash", format: "hex" },
      { name: "difficulty", extract: "state.difficulty", format: "number" }
    ]
  },
  
  initState: (params, seed) => ({
    miners: [...],
    chain: [],
    difficulty: params.difficulty || 4,
    phase: "idle",
    _trace: { triggeredLines: [], variables: {} }
  }),
  
  reducer: (state, event, tick) => {
    const newState = { ...state };
    
    if (tick % 10 === 0) {
      // 模拟挖矿
      const nonce = randomNonce(seed, tick);
      const hash = keccak256(state.miners[0].id, nonce);
      
      newState.phase = "hashing";
      newState.currentNonce = nonce;
      newState.currentHash = hash;
      
      // 填充追踪信息
      newState._trace = {
        triggeredLines: [6],  // 正在执行 keccak256
        variables: {
          nonce: nonce,
          hash: hash,
          difficulty: state.difficulty
        }
      };
      
      // 检查难度
      if (parseInt(hash, 16) < (2**256 - 1) >> state.difficulty) {
        // 成功
        newState.lastAttempt = "success";
        newState.newBlock = { height: state.chain.length, hash };
        newState.chain.push(newState.newBlock);
        newState._trace.triggeredLines = [7, 8, 9];
      } else {
        // 失败
        newState.lastAttempt = "failed";
        newState._trace.triggeredLines = [7];
      }
    }
    
    return newState;
  },
  
  render: (state) => ({
    summary: `当前难度 ${state.difficulty},链高 ${state.chain.length}`,
    phase: {
      id: state.phase,
      title: state.phase === "hashing" ? "计算候选哈希" : "等待新区块",
      intent: "observe",
      explanation: {
        what: "矿工不断尝试 nonce 并检查哈希是否满足难度。",
        why: "PoW 的安全性来自找到有效哈希所需的可验证工作量。",
        watch: "观察主舞台中的链增长和右侧代码行高亮。"
      }
    },
    focus: {
      primary: state.newBlock ? [state.newBlock.hash] : [],
      secondary: state.miners.map(miner => miner.id)
    },
    layout: {
      primary: "pow-chain",
      evidence: ["pow-network"],
      trace: "code-trace"
    },
    patterns: [
      { id: "pow-network", mode: "graph", title: "矿工网络", data: { layout: "ring", nodes: state.miners, edges: [] } },
      { id: "pow-chain", mode: "chain", title: "PoW 区块链", data: { blocks: state.chain, forks: [] } }
    ]
  })
};
```

## 3. 阶段式编排设计

### 3.1 M7 实验配置扩展

```typescript
// M7 experiment.components 新增字段
interface ExperimentComponents {
  // 原有字段
  envs?: SandboxEnvConfig[];
  sims?: SimConfig[];
  checkpoints?: CheckpointConfig[];
  
  // 新增:阶段配置
  stages?: StageConfig[];
}

interface StageConfig {
  stage: number;                    // 阶段序号(1,2,3...)
  title: string;                    // 阶段标题
  description?: string;             // 说明
  
  // 该阶段包含的组件
  components: {
    envs?: string[];                // 环境 ID 列表
    sims?: string[];                // 仿真 ID 列表
  };
  
  // 解锁条件
  unlockCondition?: {
    type: "checkpoint" | "manual";
    checkpointId?: string;          // 基于检查点
    minScore?: number;              // 最低分数
  };
  
  // 参数绑定(从前置阶段传递)
  paramBindings?: ParamBinding[];
}

interface ParamBinding {
  targetComponent: string;          // 目标组件 ID
  targetParam: string;              // 目标参数名
  sourceType: "checkpoint" | "constant";
  sourceRef?: string;               // 检查点 ID
  sourcePath?: string;              // JSON path (如 "finalState.consensusNodes")
  constantValue?: any;              // 常量值
}
```

### 3.3 确定性保证

**关键设计**:
1. 参数绑定只在 **initState 时发生**,不是运行时注入
2. 绑定的来源是 **检查点结果**(已持久化),可复现
3. 每个阶段独立记录 `seed + events`,回放时:
   ```
   阶段1: initState(params1, seed1) → replay(events1) → checkpoint1Result
   阶段2: initState({...params2, injected: checkpoint1Result}, seed2) → replay(events2)
   ```

4. 操作序列元数据包含参数来源:
   ```json
   {
     "session_id": "sess-stage2-xxx",
     "seed": 12345,
     "init_params": {
       "targetNodes": {
         "_source": "checkpoint:cp-understand-consensus:finalState.nodes",
         "_value": [...]
       }
     },
     "events": [...]
   }
   ```

## 4. 实现要点

### 4.1 前端 sim-sdk 包

```
frontend/packages/sim-sdk/
├── runtime/
│   ├── SimWorkerClient.ts        # 主线程 Worker 通信客户端
│   └── sim.worker.ts             # 隔离执行仿真包状态机与代码追踪
├── renderer/
│   ├── PatternRenderer.tsx       # 可信模式渲染器
│   ├── CodeTracePanel.tsx        # 代码追踪面板组件
│   ├── VariableWatchPanel.tsx    # 变量监视器
│   └── ... (其他7种模式渲染器)
├── types/
│   ├── SimPackage.ts             # 扩展 CodeTraceDef
│   └── StageConfig.ts            # 新增:阶段配置类型
└── index.ts
```

仿真包的 `initState`、`reducer`、`render`、`narrative.trigger`、`checkpoint.evaluate`
只允许在 `sim.worker.ts` 中执行。`SimulationWorkbench` 只能接收审核通过 bundle
的 `moduleUrl` 与初始化参数,主线程只消费 Worker 返回的 `TeachingFrame`、叙事步骤、
交互可用性和检查点结果纯数据,不得暴露或保留主线程 `SimEngine` 执行入口。

## 5. 设计边界

### 5.1 代码追踪

**允许**:
- ✅ 状态 → 代码行的单向映射
- ✅ 变量监视(从状态提取)
- ✅ 教学注释

**禁止**:
- ❌ 代码行 → 状态的反向修改(不是调试器)
- ❌ 断点、单步执行(这是仿真,不是真实执行)
- ❌ 修改源码后重新执行

### 5.2 阶段编排

**允许**:
- ✅ 检查点结果 → 后续阶段参数(initState 时)
- ✅ 阶段解锁条件(基于检查点得分)
- ✅ 常量参数注入

**禁止**:
- ❌ 运行时跨阶段状态同步
- ❌ 阶段间实时通信
- ❌ 前置阶段的动态修改影响后续阶段

## 6. 教学价值

### 6.1 代码追踪的价值

| 场景 | 传统方式 | 带代码追踪 |
|------|---------|-----------|
| 理解 PoW 挖矿 | 看动画,脑补"代码应该是..." | 看到 keccak256 那行高亮,nonce 实时变化 |
| 理解 require 检查 | 文字描述"校验失败会回退" | 看到代码行闪红,注释"难度检查失败" |
| 调试思维训练 | 无 | 类 IDE 体验,理解执行流 |

### 6.2 阶段编排的价值

| 场景 | 传统方式 | 阶段编排 |
|------|---------|---------|
| 复杂实验教学 | 一个巨型仿真,学生容易迷失 | 分3步,循序渐进 |
| 前后关联 | 手动复制参数,易错 | 自动传递,无缝衔接 |
| 个性化路径 | 所有人走同一流程 | 根据检查点得分解锁不同难度 |
